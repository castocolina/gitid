# Phase 05.7 Plan 11: Doctor SSH-Config Redundancy Advisory Check Summary

Advisory SSH-config redundancy check (FamilyRedundancy) that detects multiple Host * stanzas and duplicate global directives across the full ~/.ssh/config, wired into both cmd and TUI doctor entrypoints.

## What Was Built

### Task 1: CheckRedundancy detector + FamilyRedundancy in doctor core

**internal/doctor/checks/redundancy.go** — New `CheckRedundancy(deps doctor.Deps) []doctor.Finding` function following the `CheckOverlap` pattern exactly:
- Reads `~/.ssh/config` via `deps.ReadFile(deps.SSHConfigPath)`; returns nil on error or empty content (best-effort)
- `scanGlobalDirectives()` scans the file line-by-line counting `Host *` stanzas and tracking occurrences of `UseKeychain`, `AddKeysToAgent`, `IgnoreUnknown` at global scope (root-level lines AND lines nested under any `Host *`)
- Produces one finding for `count(Host *) > 1`, one finding per duplicated directive
- Every finding: `Family=FamilyRedundancy`, `Severity=SeverityWarning`, `Fix=nil`
- Imports neither `internal/filewriter` nor `os.Chmod` (D-01 write-free core)

**internal/doctor/doctor.go** additions:
- `FamilyRedundancy Family = "Redundancy"` constant (after `FamilyOverlap`)
- `CheckRedundancy CheckFn` field on `Deps` (documented: advisory, SeverityWarning)
- `deps.CheckRedundancy` appended last in `Run` (nil-guarded)
- `FamilyRedundancy` appended last in `Families()` — Families count is now 9

**Tests** (redundancy_test.go + doctor_test.go):
- Clean config → no findings (true negative)
- Legitimate single managed block alone → no findings (T-05.7-11-03 false-positive guard)
- Two `Host *` stanzas → at least one `FamilyRedundancy` `SeverityWarning` finding with `Fix==nil`
- User's exact UAT G-4 scenario (root `IgnoreUnknown UseKeychain` + hand-written `Host *` with `UseKeychain` + managed `_global` `Host *`) → multiple findings, all `SeverityWarning`, `Fix==nil`
- `doctor.ExitCode(redundancyFindings) == 1` (never 2 or 3)
- `Families()` last entry is `FamilyRedundancy`
- `Run` dispatches non-nil `CheckRedundancy`; nil is safely skipped

### Task 2: Wire CheckRedundancy into cmd + tui doctor

**cmd/gitid/doctor.go** — `CheckRedundancy: checks.CheckRedundancy` added in `buildDoctorDeps` alongside `CheckOverlap` (~line 481). `gitid doctor` now runs the redundancy check on the live `~/.ssh/config`.

**tui/deps.go** — Same assignment in `buildTUIDoctorDeps` alongside `CheckOverlap`. The TUI health view runs the redundancy check.

**tui/health.go** — `case doctor.FamilyRedundancy: fn = d.CheckRedundancy` added to `makeFamilyCmd`'s switch. FamilyRedundancy participates in the per-family async streaming pattern.

**tui/health_test.go** — Updated `fakeTUIDocDepsForHealth()` to include `CheckRedundancy: noop` (9th field); `TestHealthFamilies` now expects 9 families * 2 = 18 batch cmds; `TestHealthFamilyRedundancyDispatch` drives the real `makeFamilyCmd(FamilyRedundancy)` path and verifies:
- Non-nil `CheckRedundancy` dispatches and returns its finding
- Nil `CheckRedundancy` surfaces an error (not a silent pass — D-16 anti-blindspot)

## Severity Guarantee (T-05.7-11-01)

Every `CheckRedundancy` finding has:
- `Severity: doctor.SeverityWarning` — exit code 1, never 2 or 3
- `Fix: nil` — not collected by `collectFixable`, never gates Apply/write prompt
- Never blocks `doctor.Run` or any write flow

This is proven by `TestCheckRedundancy_ExitCodeWarning` and `TestCheckRedundancy_FixAlwaysNil`.
