<objective>
CORE (UI-free): add a transactional PROVISIONAL (test) SSH block lifecycle to the
config-writing core so the staged create wizard can write the `Host <alias>` block
BEFORE testing the alias, per CLAUDE.md's `test -> confirm + backup -> re-test the
resolved configuration` rule. Today the wizard tests `ssh -T git@<alias>` before any
block exists, so any alias fails "could not resolve hostname" and the wizard can never
complete (UAT G-5, BLOCKER).

A provisional block uses a DISTINCT sentinel — `# BEGIN gitid provisional: <name>` …
`# END gitid provisional: <name>` — so it is NEVER confused with a real
`# BEGIN gitid managed: <name>` identity. Its IdentityFile points at the STAGED (temp)
key so the alias resolves to exactly that key during the test, before the key is
installed to ~/.ssh. On test success the block is PROMOTED (rewritten as managed,
IdentityFile repointed to the final key); on cancel/fail it is DROPPED (restored from
backup). All writes go through the filewriter chokepoint (timestamped backup + atomic +
idempotent) — CLAUDE.md never-corrupt invariant.

This plan SUPERSEDES the simpler leg-1-reorder version of 05.7-14: instead of writing
the final managed block before the test (and leaving it behind on failure), it writes a
clearly-marked provisional block and removes it on any intentional exit, so a partially
verified identity is never silently left as a managed block.

Purpose: give the wizard (Plan 15) and doctor (Plan 16) a recipe-faithful, never-
destructive provisional-block primitive so the alias test can resolve while the key is
still staged, and a leftover provisional block is always distinguishable from a real
identity.
Output: filewriter provisional-sentinel splice helpers, sshconfig provisional lifecycle
(WriteProvisional/Promote/DropProvisional/ListProvisional), and identity EffectiveAlias +
PersistSSHProvisional/PromoteSSH/DropProvisionalSSH.
</objective>
