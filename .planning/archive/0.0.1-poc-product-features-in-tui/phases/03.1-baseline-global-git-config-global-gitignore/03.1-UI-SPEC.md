# Phase 03.1 — Terminal CLI Interaction Contract

> This phase has no graphical or web UI. This document is a **terminal-output and
> interaction contract** — the CLI equivalent of a UI design spec. It answers:
> "What must the executor render, in what order, with what copy, so that
> `gitid baseline setup` and `gitid baseline show` are legible, safe, and
> consistent with the existing `gitid identity` command surface?"
>
> Each standard UI-SPEC dimension is reinterpreted for a terminal CLI below.
> Web/GUI-specific sub-fields (hex colors, CSS, font px, component libraries) are
> marked N/A with their terminal equivalent stated.

## Spacing Scale

> Terminal equivalent: blank lines and indentation levels, not CSS spacing tokens.

| Token | Terminal Value | Usage |
|-------|---------------|-------|
| xs | 0 blank lines | Between fields within a block (key: value lines) |
| sm | 2-space indent | Field label indent inside a block or section (matches list.go pattern: `"  key: ..."`) |
| md | 1 blank line | Between sections within a single preview (e.g. between baseline block and url-rewrites block) |
| lg | 1 blank line + section header | Between the preview and the confirm prompt |
| xl | `\n===` header separator | Major section boundary (preview header, summary header) |

**Indentation rule:** 2 spaces for field lines under a section header, consistent with
`printAccounts` in `list.go` (`"  key:      %s\n"`) and `printPreview` in `add.go`
(`"--- ~/.ssh/config (Host block) ---\n"`).

**No exceptions.**

## Typography

> Terminal equivalent: text weight is conveyed by UPPERCASE labels, `===` / `---`
> delimiters, and `!` prefix markers — not font weight or size.

| Role | Terminal Convention | Usage |
|------|---------------------|-------|
| Section header | `=== Title ===\n` | Major sections: preview start, summary start |
| Sub-section header | `--- label ---\n` | Named sub-blocks within a preview (matches `printPreview` in add.go) |
| Field label | `  label:    value\n` | Column-aligned key-value pairs (2-space indent, label padded to 10 chars) |
| Warning / conflict marker | `  ! message\n` | Conflict warnings and incomplete markers (matches `"  ! incomplete: ..."` in list.go) |
| Command echo | `$ command\n` followed by output | Pre-write and resolved test results (matches `printPreWrite` in add.go) |
| Prompt | `Label [default]: ` | Interactive prompts (matches `prompt()` in add.go line 490-501) |
| Confirm prompt | `Label [y/N]: ` | Single confirmation gate (matches `confirm()` in add.go line 512-517) |

**Source:** Directly derived from existing `cmd/gitid/add.go`, `list.go`, `update.go`,
`delete.go` patterns. No divergence from established conventions is permitted.

## Command Surface Contract

### 1. Command Hierarchy

```
gitid baseline
    setup     — write the three managed surfaces with backup→preview→confirm
    show      — read back and display the current managed baseline state from disk
```

Both are direct subcommands of `gitid baseline` (analogous to `gitid identity` with
`add/list/update/delete`). `baseline` itself with no subcommand should print usage.

`setup` accepts one flag: `--dry-run` (preview without writing, consistent with
`identity add --dry-run` and `identity delete --dry-run`).

### 2. `gitid baseline setup` — Full Interaction Flow

The flow follows the established backup→preview→confirm pattern proven in `add.go`
and `delete.go`. The steps are:

```
Step 1: Conflict scan (silent, no output)
Step 2: Build default selections (Tier-1 locked, Tier-2 default-ON, url-rewrites default-ON)
Step 3: Print unified preview
Step 4: Interactive editing of Tier-2 keys and url-rewrite selection
Step 5: Single confirm prompt [y/N]
Step 6: Write (unless --dry-run)
Step 7: Print write summary with backup paths
```

## Preview Layout Contract

### Preview Header

```
=== Preview: baseline setup ===
```

### Sub-section 1: Baseline git-config block

```
--- ~/.gitconfig.d/00-baseline (baseline block) ---
[core]
	ignorecase = false
	excludesfile = ~/.gitignore_global
	autocrlf = input
	pager = less -FRX
[push]
	autoSetupRemote = true
[pull]
	rebase = true
[fetch]
	prune = true
[color]
	ui = auto
	branch = auto
	diff = auto
	status = auto
[diff]
	colorMoved = zebra
[merge]
	conflictstyle = zdiff3
[init]
	defaultBranch = main
[alias]
	st = status
	co = checkout
	br = branch
	ci = commit
	df = diff
	lg = log --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit
	unstage = reset HEAD --
	last = log -1 HEAD
```

The above is the **exact byte-stable render** (SC-1 idempotency contract). Tab character
before each key. Section order fixed: `[core]`, `[push]`, `[pull]`, `[fetch]`,
`[color]`, `[diff]`, `[merge]`, `[init]`, `[alias]`. Alias order fixed:
`st`, `co`, `br`, `ci`, `df`, `lg`, `unstage`, `last`.

**Tier-1 keys** (`core.ignorecase`, `core.excludesfile`, `push.autoSetupRemote`,
`pull.rebase`, `fetch.prune`, `color.ui`) are always shown and cannot be deselected.

**Tier-2 keys** (all others) are shown by default but the user may deselect them.
The editing step (Step 4) prompts for each Tier-2 group:

```
Include Tier-2 defaults? (autocrlf, pager, branch/diff/status color, zdiff3, main branch, aliases) [Y/n]:
```

If the user types `n`, the Tier-2 keys are removed from the preview and the written
block. The prompt uses `Y` as the uppercase default (opt-out model), consistent with
CONTEXT.md D-04.

### Sub-section 2: URL rewrites block (default-ON, editable)

```
--- ~/.gitconfig.d/00-baseline (url-rewrites block) ---
[url "git@github.com:"]
	insteadOf = https://github.com/
[url "git@gitlab.com:"]
	insteadOf = https://gitlab.com/
[url "git@bitbucket.org:"]
	insteadOf = https://bitbucket.org/
```

**Blast-radius warning** (mandatory, shown immediately after the url-rewrites sub-section):

```
  ! insteadOf rewrites affect ALL HTTPS operations for each host —
  ! including go get, npm install, cargo fetch, and CI pipelines
  ! using token-based HTTPS auth. SSH agent must be running.
```

**Rewrite selection editing step** (Step 4, after the preview):

```
Keep rewrite for github.com (https://github.com/ → git@github.com:)? [Y/n]:
Keep rewrite for gitlab.com (https://gitlab.com/ → git@gitlab.com:)? [Y/n]:
Keep rewrite for bitbucket.org (https://bitbucket.org/ → git@bitbucket.org:)? [Y/n]:
```

Each prompt uses `Y` as the uppercase default (opt-out per D-06). If the user types
`n`, that mapping is removed from the url-rewrites block. If all three are removed,
the url-rewrites block is not written (no empty block).

### Sub-section 3: Global gitignore

```
--- ~/.gitignore_global (gitignore block) ---
.DS_Store
Thumbs.db
*.log
*.bak
*.tmp
*.swp
*.swo
.idea/
.vscode/
node_modules/
__pycache__/
*.pyc
.env
```

No user editing of gitignore patterns is offered at setup time; the list is seeded and
idempotent. Users may add their own entries outside the managed block.

### Sub-section 4: include wiring

```
--- ~/.gitconfig (include block, prepended) ---
# BEGIN gitid managed: baseline-include
[include]
	path = ~/.gitconfig.d/00-baseline
# END gitid managed: baseline-include
```

Followed by a floor-semantics note:

```
  > This block will be placed at the TOP of ~/.gitconfig.
  > Keys you set elsewhere in ~/.gitconfig override the baseline (floor model).
```

### Sub-section 5: Conflict warnings (conditional)

Shown only if the conflict scan found overlapping keys. Header:

```
--- Conflicts detected in ~/.gitconfig ---
```

One line per conflict, format:

```
  ! core.ignorecase: your value=true  baseline=false  winner=yours (floor ordering)
  ! core.excludesfile: your value=~/.gitignore  baseline=~/.gitignore_global  winner=yours (floor ordering)
```

If `core.excludesfile` conflicts, append:

```
  ! Note: your excludesfile override means gitid's ~/.gitignore_global will be
  ! ignored by git. Remove your override or update it to ~/.gitignore_global.
```

If no conflicts:

```
  > No conflicts found in ~/.gitconfig.
```

## Confirm Prompt Contract

Single confirm prompt (SAFE-03), shown after all preview sub-sections and editing steps:

```
Write baseline now? [y/N]:
```

- Default is `N` (uppercase), same pattern as `confirm()` in `add.go` line 512-517.
- On `N` or Enter: `"Baseline setup cancelled; no files were written.\n"`
- Under `--dry-run`: confirm prompt is skipped; print `"--dry-run: no files were written.\n"` instead.

---

## Write Summary Contract

Shown after a confirmed write, following the pattern of `delete.go` backup reporting:

```
Baseline written.
  ~/.gitconfig backup:          ~/.gitconfig.bak.<timestamp>
  ~/.gitconfig.d/00-baseline:   written (new)
  ~/.gitignore_global:          written (new)
```

If a file already existed and was updated:

```
  ~/.gitconfig.d/00-baseline:   updated (backup: ~/.gitconfig.d/00-baseline.bak.<timestamp>)
```

If a file was unchanged (idempotent re-run):

```
  ~/.gitconfig.d/00-baseline:   unchanged (no write needed)
```

The "unchanged" line requires the executor to detect zero-diff before calling
`filewriter.Write` and skip the write + backup in that case.

## `gitid baseline show` — Read-Back Contract

`gitid baseline show` reads the current managed baseline state from disk (SC-5,
IDENT-07 pattern — no sidecar DB). No prompts.

> **Format note (intentional):** `show` deliberately uses the lighter
> label-first format below (no `=== ===` major-section wrapper) because it is a
> read-only command following the existing `list.go` print convention, not a
> multi-step interactive flow like `setup`. Executors MUST NOT add a
> `=== ===` section header to `show` output for consistency with `setup` — the
> two formats differ on purpose.

### Output when baseline is installed

```
baseline: installed
  file:     ~/.gitconfig.d/00-baseline
  include:  ~/.gitconfig (prepended)

baseline keys:
  core.ignorecase      = false
  core.excludesfile    = ~/.gitignore_global
  core.autocrlf        = input
  core.pager           = less -FRX
  push.autoSetupRemote = true
  pull.rebase          = true
  fetch.prune          = true
  color.ui             = auto
  color.branch         = auto
  color.diff           = auto
  color.status         = auto
  diff.colorMoved      = zebra
  merge.conflictstyle  = zdiff3
  init.defaultBranch   = main
  alias.st             = status
  alias.co             = checkout
  alias.br             = branch
  alias.ci             = commit
  alias.df             = diff
  alias.lg             = log --graph ...
  alias.unstage        = reset HEAD --
  alias.last           = log -1 HEAD

url rewrites: 3 active
  https://github.com/   → git@github.com:
  https://gitlab.com/   → git@gitlab.com:
  https://bitbucket.org/ → git@bitbucket.org:

gitignore: ~/.gitignore_global
  managed patterns: 13
  .DS_Store, Thumbs.db, *.log, *.bak, *.tmp, *.swp, *.swo,
  .idea/, .vscode/, node_modules/, __pycache__/, *.pyc, .env
```

### Output when baseline is not installed

```
no gitid-managed baseline found
Run 'gitid baseline setup' to initialize.
```

This matches the empty-state pattern of `runIdentityList` in `list.go`:
`"no gitid-managed identities found\n"`.

### Output when baseline is partially installed (incomplete)

```
baseline: incomplete
  ! missing: include block in ~/.gitconfig
  ! missing: ~/.gitconfig.d/00-baseline
  Run 'gitid baseline setup' to repair.
```

Uses `!` prefix marker consistent with the `acct.Incomplete` rendering in `list.go`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (setup confirm) | `Write baseline now? [y/N]:` |
| Primary CTA (show, no prompt) | `gitid baseline show` (no CTA needed — read-only) |
| Empty state (show, not installed) | `no gitid-managed baseline found\nRun 'gitid baseline setup' to initialize.` |
| Incomplete state (show) | `baseline: incomplete\n  ! missing: <artifact>\n  Run 'gitid baseline setup' to repair.` |
| Cancelled (setup) | `Baseline setup cancelled; no files were written.` |
| Dry-run completion | `--dry-run: no files were written.` |
| Write completion | `Baseline written.` |
| Idempotent re-run (no change) | `<file>: unchanged (no write needed)` |
| Conflict warning header | `--- Conflicts detected in ~/.gitconfig ---` |
| Conflict line | `  ! <key>: your value=<v>  baseline=<v>  winner=yours (floor ordering)` |
| insteadOf blast-radius warning | `  ! insteadOf rewrites affect ALL HTTPS operations for each host —` (3-line block, see above) |
| No-conflicts confirmation | `  > No conflicts found in ~/.gitconfig.` |
| Floor-model note | `  > This block will be placed at the TOP of ~/.gitconfig.` (2-line block, see above) |
| Rewrite deselection prompt | `Keep rewrite for <host> (<https-url> → <ssh-url>)? [Y/n]:` |
| Tier-2 opt-out prompt | `Include Tier-2 defaults? (autocrlf, pager, branch/diff/status color, zdiff3, main branch, aliases) [Y/n]:` |
| Error: home dir | `baseline setup: resolving home dir: <err>` |
| Error: write failure | `baseline setup: writing <file>: <err>` |
| Error: conflict scan failure | `baseline setup: scanning conflicts: <err>` |

**Copy rules (from CLAUDE.md and existing cmd patterns):**
- All copy is in English (no Spanish, no emoji).
- Error messages follow `<command>: <context>: <err>` format (matches existing
  `"identity add: resolving home dir: %w"` pattern in `add.go`).
- Prompts use `[default]` bracket notation consistent with `prompt()` helper.
- Cancellation messages are neutral, not apologetic.

## Information Architecture

### `gitid baseline` sub-tree (consistent with `gitid identity` pattern)

```
gitid
└── baseline
    ├── setup   [--dry-run]    # write/update the three managed surfaces
    └── show                   # read-back from disk, no prompts
```

No `update` or `delete` subcommands in Phase 3.1 (out of scope). The `setup` command
is idempotent and re-runnable — it IS the update path. Removal of the baseline is
not a Phase 3.1 requirement.

### Cobra wiring

- `newBaselineCmd()` returns a `*cobra.Command` with `Use: "baseline"` and
  `Short: "Manage the shared global git baseline (core/push/pull defaults, gitignore, url rewrites)"`.
- Sub-commands `newBaselineSetupCmd()` and `newBaselineShowCmd()` are registered on it.
- `baseline` is registered on the root command in `main.go`, same as `identity`.
- `--dry-run` flag on `setup` only (not on `show`).

## Accessibility Contract (Terminal)

1. **No information conveyed by color alone.** All warnings use the `!` prefix.
   All status information uses plain text labels (`baseline: installed`,
   `baseline: incomplete`).
2. **Piped output is legible.** `gitid baseline show | cat` must produce the same
   content as interactive use.
3. **All prompts show the default visually** (`[Y/n]` or `[y/N]`). Uppercase letter
   is the default. Users who press Enter get the default without confusion.
4. **All destructive or irreversible actions state consequences before confirm.**
   The blast-radius warning for `insteadOf` is shown in the preview BEFORE the
   selection prompt, not after.
5. **English only.** All output, prompts, warnings, and error messages.
