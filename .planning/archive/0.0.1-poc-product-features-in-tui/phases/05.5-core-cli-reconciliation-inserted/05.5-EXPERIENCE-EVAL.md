# Intro

This is an advisory, fresh-eyes UX evaluation of the **five CLI flows** that Phase
5.5 changed. It does NOT gate the phase; it feeds Phase 5.6 (the TUI rebuild). The
TUI was explicitly NOT evaluated.

**What was driven:** the real built binary `./bin/gitid` (built via `make build`
at commit `275cb024`, `gitid version 0.0.0-dev`).

**How it was sandboxed (no real environment touched):**
- Every invocation ran under a per-scenario temp `HOME` (`mktemp -d`), so all
  writes to `~/.ssh/config`, `~/.gitconfig`, `~/.gitconfig.d/`, and
  `~/.ssh/allowed_signers` landed inside the throwaway sandbox.
- A **fake `ssh`** (and `ssh-add`) was written to a temp dir and prepended to
  `PATH`, replicating `e2e/harness_test.go`'s `FakeSSHDir`. It handles `-Q key`
  (key-type probe), `-G <alias>` (resolution fixture), and dispatches the
  connection test on `GITID_FAKE_SSH_MODE` (`pass` ‚Üí "successfully
  authenticated" exit 1, which is GitHub's real behavior; `denied` ‚Üí "Permission
  denied (publickey)" exit 255). A mode-flipping variant proved the
  denied‚Üípass retry path.
- Interactive prompts were driven with scripted stdin (`printf`).

The only commands that touched real state were `make install` / `make uninstall`
(into `$GOPATH/bin`), used deliberately to evaluate Flow 5; both were reverted
(`make uninstall`). No production code or tests were modified.

Each finding is tagged `works` / `confusing` / `fix-suggestion` and severity
`info` / `minor` / `major`.

# Flow 1 ‚Äî Create-flow loop (`gitid identity add`, create-new)

All four branches were driven hermetically and behave per the Phase 5.5 design
(generate-to-`~/.ssh` up front ‚Üí auth-gated loop ‚Üí persist only on PASS or
typed skip-confirm).

### F1.1 ‚Äî PASS run auto-persists with NO pre-write confirm ‚Äî `works` / `info`
Input (`GITID_FAKE_SSH_MODE=pass`):
```
printf '1\ntestpass\nTest Pass\ntest@pass.dev\ngithub\n\n\n\n1\n\n\n' | gitid identity add
```
Output (abridged):
```
Pre-write connectivity test:
$ .../ssh -i .../id_ed25519_testpass ... -T git@ssh.github.com
Hi user! You've successfully authenticated, but GitHub does not provide shell access.

=== Preview: four coordinated artifacts ===
...
Resolved test:
$ .../ssh ... -T git@testpass.github
  user=git hostname=ssh.github.com port=443 identitiesonly=yes
=== EXIT 0 ===
```
The old "Write all four artifacts now?" prompt is gone; on an authenticated PASS
the artifacts are written automatically and all four surfaces are present on disk
(`id_ed25519_testpass`, `~/.ssh/config`, `~/.gitconfig` managed block,
`~/.gitconfig.d/testpass`, `allowed_signers`). This is the corrected G-05/F-2 shape.

### F1.2 ‚Äî QUIT keeps the key, writes NO config ‚Äî `works` / `info`
Input (`denied`, then `q`):
```
printf '...\nq\n' | gitid identity add
```
Output:
```
[r] retry  [s] skip-&-write (offline/upload-later)  [q] quit
Choice [r]: Quit. Key kept at .../.ssh/id_ed25519_testquit. Run 'gitid doctor' when ready to write config.
```
On-disk after: `~/.ssh/id_ed25519_testquit{,.pub}` present; `~/.gitconfig` and
`~/.ssh/config` **do not exist**. Exactly the desired "key staged, nothing
committed, told how to resume" outcome. The pointer to `gitid doctor` is a nice
recovery affordance.

### F1.3 ‚Äî SKIP requires a typed confirm + a warning, then writes ‚Äî `works` / `info`
Input (`denied`, then `s`, then `y`):
```
[r] retry  [s] skip-&-write (offline/upload-later)  [q] quit
Choice [r]: Warning: key is not yet authenticated. You must upload the key and verify authentication before using this identity.
Write config artifacts without authenticated PASS? [y/N]:
```
After `y` the four artifacts are written (2 `gitid managed` blocks in `.gitconfig`,
`~/.ssh/config` created, fragment present). The escape hatch is gated behind an
explicit second consent ‚Äî good safety ergonomics.

### F1.4 ‚Äî denied‚Üíretry‚Üípass persists ‚Äî `works` / `info`
A mode-flipping fake ssh (first attempt denied, second PASS) + `r` at the loop:
the loop re-ran the connectivity test, got the PASS banner, and auto-persisted. The
loop is genuinely re-driven each iteration (not a single-shot gate).

### F1.5 ‚Äî PersistKey re-writes the just-generated key, leaving spurious `.bak` files ‚Äî `confusing` / `minor`
After a PASS create-new, `~/.ssh` contains:
```
id_ed25519_testpass
id_ed25519_testpass.bak.20260614-135156
id_ed25519_testpass.pub
id_ed25519_testpass.pub.bak.20260614-135156
```
The key was already written to its final path by `Generate` (D-01), yet the
persist step re-wrote it through the filewriter chokepoint, which backs up the
pre-existing file ‚Äî producing a `.bak` of a key that is **seconds old and
identical**. For a brand-new identity these backups are pure noise and could
confuse a user inspecting `~/.ssh` ("why are there two copies of my new key?").
Suggestion: in the create-new path, skip the backup-on-write for the key that
`Generate` just produced (the `PrivPEM == nil` branch in `PersistKey` should be a
true no-op, which it appears to intend ‚Äî but a backup still appears, so something
is re-touching the file).

### F1.6 ‚Äî The PASS banner text says "GitHub" even for GitLab ‚Äî `confusing` / `minor`
This is a fake-ssh artifact in my harness (the script emits a GitHub banner
regardless of provider), NOT a product bug ‚Äî flagged only so the next reader
doesn't mistake it for one. The product faithfully prints whatever `ssh -T`
returns.

# Flow 2 ‚Äî Match-strategy picker (`add` and `update`) + non-interactive flags

### F2.1 ‚Äî Interactive picker offers gitdir / URL / both with good labels ‚Äî `works` / `info`
```
Match strategy:
  1) folder (gitdir)      ‚Äî activate identity for all repos under a directory (default)
  2) repo URL (hasconfig) ‚Äî activate identity for repos matching a remote URL pattern
  3) both                 ‚Äî activate by folder AND URL (OR-applied by git)
Strategy [1]:
```
The descriptions are concrete and the OR semantics for "both" are stated. This is
a clear resolution of F-6.

### F2.2 ‚Äî Suggested URL default is correct ‚Äî `works` / `info`
Choosing strategy 2 offers `URL pattern [git@ssh.github.com:urlid/**]` (i.e.
`git@<hostname>:<name>/**`). Resulting block:
```
[includeIf "hasconfig:remote.*.url:git@ssh.github.com:urlid/**"]
	path = .../.gitconfig.d/urlid
```

### F2.3 ‚Äî "both" writes two includeIf blocks ‚Äî `works` / `info`
```
[includeIf "gitdir:~/git/bothid/"]
	path = .../bothid
[includeIf "hasconfig:remote.*.url:git@ssh.github.com:bothid/**"]
	path = .../bothid
```

### F2.4 ‚Äî `add` flags skip the right prompts ‚Äî `works` / `info`
```
gitid identity add --name flagid --provider gitlab --gitdir '~/work/flag/' --url 'git@gitlab.com:flag/**'
```
The Match-strategy picker did NOT print (both `--gitdir`+`--url` ‚Üí "both"), and the
name/provider prompts were skipped. Result: two includeIf blocks + `# gitid:
provider=gitlab` marker in the Host block. Note the create **mode** prompt
(`Choose mode [1]`) is still shown even with `--name` ‚Äî expected, but worth knowing
for scripting (you must still feed a mode line).

### F2.5 ‚Äî `update` flags skip prompts and change strategy; name correctly NOT promptable ‚Äî `works` / `info`
```
gitid identity update flagid --url 'git@gitlab.com:flag-new/**'
```
No Match-strategy picker printed; preview showed
`match: hasconfig:remote.*.url:git@gitlab.com:flag-new/**` and the on-disk block
collapsed from two includeIf lines to the single hasconfig line. `update` has no
`--name` flag (name is positional/immutable) ‚Äî consistent with the design.

### F2.6 ‚Äî Scripting the picker is brittle because prompts are unlabeled on one line ‚Äî `confusing` / `minor`
Because `prompt()` writes `label [default]: ` without a trailing newline and reads
a line, the entire prompt sequence renders as one run-on line when piped
(`Choose mode [1]: Identity name: Git user.name: ...`). A human at a TTY sees them
one-at-a-time, so this is not a TTY defect ‚Äî but it makes the flow very
error-prone to script and hard to reason about offset-by-offset (I mis-aligned
stdin once and got a misleading `user.email is malformed: "github"` error that was
purely my input drift, not a product bug). For the CLI-as-engine goal, consider a
fully-flagged non-interactive create (`--name --git-name --git-email --provider
--alias --hostname --port` + a `--yes`/`--skip-test` for CI) so the engine can be
driven without prompt-offset gymnastics. Today only `--name/--gitdir/--url/--provider`
exist, so git-name/email/alias/hostname/port still require stdin.

# Flow 3 ‚Äî `identity list` provider correctness

### F3.1 ‚Äî Markerless legacy block resolves to `github` via hostname fallback ‚Äî `works` / `info`
Seeded a managed SSH block with alias `my.custom.alias` (which does NOT match
`name.provider`) and **no** `# gitid: provider=` marker, hostname
`ssh.github.com`:
```
identity: legacyid
  key:      /Users/x/.ssh/id_ed25519_legacyid
  git:      Legacy User <legacy@x.dev>
  alias:    my.custom.alias
  provider: github
  ...
```
`provider: github` (not `ssh.github.com`) ‚Äî the F-3 hostname fallback works even
when the alias gives no hint and no marker is present. A second, marked block
(`# gitid: provider=github`) also correctly showed `github`. This is the direct
fix for the original `provider: ssh.github.com` complaint.

### F3.2 ‚Äî Plain text list, no per-row health/identification beyond fields ‚Äî `info` / `info`
The list is readable (one stanza per identity with key/git/alias/provider/port/match)
but is purely textual with no incompleteness/health annotation visible in this
seeded case. Not a bug for the CLI; flagged because the Phase 5.6 TUI is meant to
turn this into a master list with per-identity health badges.

# Flow 4 ‚Äî `gitid doctor` overlap warning

### F4.1 ‚Äî Identical-gitdir overlap is detected, classified `warning`, and disambiguated ‚Äî `works` / `info`
Seeded `alpha` and `beta` both with `gitdir:~/git/personal/`:
```
=== Overlap ===
  ‚úó identities "alpha" and "beta" have overlapping match conditions (identical-gitdir) [warning]
    ...Git evaluates [includeIf] blocks in file order and the last matching block wins.
    The identity written most recently to ~/.gitconfig ("beta") will take precedence.
    Detail: both use gitdir "~/git/personal/"
    fix: Narrow the match condition of "alpha" or "beta" so they do not overlap
         (e.g. use a more specific gitdir path or URL pattern).
```
This is an excellent finding: it names both identities, explains the last-wins
mechanic, **tells the user which one wins** (`beta`), shows the offending value,
and gives an actionable remediation. Severity is `[warning]`, not error/critical ‚Äî
exactly the F-7 ask.

### F4.2 ‚Äî Nested-gitdir overlap is detected and classified `warning` ‚Äî `works` / `info`
`alpha=~/git/`, `beta=~/git/personal/` (nested):
```
  ‚úó identities "alpha" and "beta" have overlapping match conditions (nested-gitdir) [warning]
    ...Detail: gitdir "~/git/personal/" is nested inside "~/git/"
```
The classifier distinguishes `identical-gitdir` vs `nested-gitdir`, which is more
precise than a generic "overlap".

### F4.3 ‚Äî The `‚úó` glyph for a warning reads like an error ‚Äî `confusing` / `minor`
The overlap warning is rendered with the same `‚úó` mark used by hard errors
(perms/coherence). Only the trailing `[warning]` tag and the summary line
distinguish it. At a glance a user could read the overlap as a failure of equal
weight to a 0600 violation. Suggestion (carry to 5.6): use a distinct glyph/colour
for `warning` (e.g. `!`) vs `error` (`‚úó`). The summary line itself is honest:
`0 critical, 5 error, 1 warning, 1 info`.

### F4.4 ‚Äî Overlap warning does not by itself raise the exit tier above what its severity warrants ‚Äî `works` / `info`
In the seeded run the non-zero `exit code: 2` came from the perms/coherence/baseline
**errors**, not the overlap warning. (A warning-only run would land at exit 1.) The
classification is wired to severity correctly.

# Flow 5 ‚Äî Install path / PATH feedback

### F5.1 ‚Äî `gitid doctor` honestly reports binary location + PATH status ‚Äî `works` / `info`
On a sandbox where the binary dir is NOT on PATH:
```
=== Dependencies ===
  ! binary: /Users/ramon/.../bin/gitid [info]
    on PATH: no
    fix: export PATH="$PATH:/Users/ramon/.../bin"
```
When the dir IS on PATH: `on PATH: yes` (no fix line). This is `info` severity and
does NOT drive the exit tier (confirmed: a run with `on PATH: no` reported
`1 error, 1 info` and `exit code: 2`, the 2 coming from the error, not the info).
This is the doctor half of the F-1 fix and it is genuinely useful and honest.

### F5.2 ‚Äî `make install` reports the install path ‚Äî `works` / `info`
```
go install ./cmd/gitid
  installed: /Users/ramon/go/bin/gitid
  PATH: OK (gitid is on PATH)
```
The install path is now printed (the original F-1 "where did it go?" is answered).

### F5.3 ‚Äî `make install`'s "PATH: OK" is a FALSE POSITIVE ‚Äî `fix-suggestion` / `major`
This is the most important finding of the audit. `make install` printed
`PATH: OK (gitid is on PATH)` ‚Äî yet immediately afterwards, in a normal login
shell:
```
$ which gitid
gitid not found
```
Root cause (confirmed in the Makefile): line 39 does
`export PATH := $(HOME)/.local/bin:$(GOPATH_BIN):$(PATH)`, so the install recipe's
own PATH check (`echo "$$PATH" | grep -qF "$(GOPATH_BIN)"`) ALWAYS sees
`$GOPATH/bin` on PATH ‚Äî because make put it there ‚Äî regardless of the user's real
interactive PATH. The check therefore can never report the failure case it was
written to catch. This re-creates the exact original F-1 confusion ("the tool says
it's installed and on PATH, but I can't run it"). The `gitid doctor` finding is
honest because it runs in the user's real environment; the Makefile check is not.

Suggestion: have the install recipe consult the *user's* shell PATH, not make's
augmented one ‚Äî e.g. capture the inherited PATH before the `export` augmentation
(stash it in a variable like `ORIG_PATH := $(PATH)` at the very top, before line
39, and grep that), or simply defer the PATH advice to `gitid doctor` and have
`make install` print "run `gitid doctor` to confirm gitid is on your PATH" instead
of asserting it.

# Handoff to Phase 5.6 (TUI rebuild) ‚Äî UX learnings worth carrying

1. **The create-flow loop is the model to preserve.** generate‚Üíshow-pubkey+upload-instructions‚Üí
   auth-test‚Üí`[r]etry/[s]kip+confirm/[q]uit`‚Üípersist-only-on-PASS is correct and
   safe. The TUI's in-app wizard should mirror these exact states (including the
   typed skip-confirm and the "key kept, run doctor" quit affordance), just with
   live spinners instead of a re-printed `$ ssh ‚Ä¶` block each iteration.

2. **Warnings must look different from errors.** Doctor renders overlap warnings
   with the same `‚úó` glyph as hard errors (F4.3). The integrated Health view
   should give `warning` its own glyph/colour/badge so a "two identities overlap"
   advisory is not mistaken for a 0600-perm failure. The summary taxonomy
   (`critical/error/warning/info`) is already correct ‚Äî surface it visually.

3. **Install/PATH feedback should have ONE honest source of truth.** The doctor
   finding is trustworthy; the `make install` check lies (F5.3). The TUI/doctor
   should own this signal and `make install` should defer to it. Carry the F5.3
   fix regardless of the TUI.

4. **The match-strategy picker is good UX to lift verbatim.** Three labelled
   options with concrete descriptions and a pre-filled, name-derived URL default.
   In the TUI this becomes an in-pane editable per-site match strategy
   (gitdir/URL/both) ‚Äî keep the same defaulting and the "both = OR-applied" wording.

5. **Provider is now reliable for the master list.** Reconstruction resolves
   provider from marker‚Üíhostname-fallback, so the TUI's left sidebar can show
   `name ¬∑ provider ¬∑ alias` truthfully even for legacy markerless blocks.

6. **Give the CLI a fully-flagged non-interactive create.** The run-on, unlabeled
   prompt stream (F2.6) makes the engine hard to script and was the source of a
   confusing (self-inflicted) `user.email is malformed` error. The TUI rebuild
   shouldn't need it, but the "CLI stays the scriptable engine" decision does:
   add `--git-name/--git-email/--alias/--hostname/--port` (+ a `--skip-test`/`--yes`)
   so CI and the TUI's own automation can drive create without stdin offset games.

7. **Suppress spurious `.bak` files for freshly-generated keys** (F1.5). A user
   browsing `~/.ssh` right after a successful create should not see a backup of a
   key that is seconds old; it undermines confidence in the backup discipline.

## Post-Eval Remediation (orchestrator addendum, 2026-06-14)

The findings above are preserved as-observed. One was actionable enough to fix
within Phase 5.5 rather than defer:

- **major / fix-suggestion ‚Äî `make install` false "PATH: OK" (F5.3, the F-1 blocker):**
  **FIXED** in commit `166afc4`. The Makefile globally prepends `$(GOPATH_BIN)` to
  PATH for every recipe, so the install target's membership check always matched.
  It now captures `ORIGINAL_PATH` before that export and tests membership against
  the caller's real interactive PATH (with `grep -qxF` exact-segment matching).
  Verified: against a shell PATH lacking `~/go/bin` it now prints the honest
  "NOT on your PATH" hint; with it present it prints "PATH: OK".

  **F-1 status upgrade:** PARTIAL ‚Üí **RESOLVED**. Both surfaces ‚Äî `gitid doctor`
  and `make install` ‚Äî now report install path + PATH membership honestly.

The remaining `minor`/`info` findings (spurious `.bak` of fresh keys, run-on
unlabeled prompt stream / fully-flagged non-interactive create, overlap `‚úó` glyph
shared with hard errors) are carried into the Phase 5.6 handoff above as advisory
input; they do not gate this phase.
