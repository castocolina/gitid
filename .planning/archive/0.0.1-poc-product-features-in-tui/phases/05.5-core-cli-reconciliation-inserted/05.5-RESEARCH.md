## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Create-flow loop (FIX-CREATE-01 / G-05, F-2)
- **D-01:** Generate the real key pair into `~/.ssh` immediately. Only the four config artifacts (SSH Host block, includeIf, fragment, allowed_signers) wait for an authenticated PASS. Replaces current temp-stage → delete behavior (`add.go:337`/`Cleanup`).
- **D-02:** New ordering: generate key → copy `.pub` + show upload instructions → wait → loop the connectivity test on demand → on authenticated PASS, persist. The old "Write all four artifacts now?" prompt before any test (`add.go:312`) is removed.
- **D-03:** On authenticated PASS → auto-persist the four artifacts and report backup paths. No extra confirm prompt.
- **D-04:** The loop offers retry / skip-&-write / quit. Quit before PASS keeps the generated key in `~/.ssh`, prints its location, and notes that `doctor` will flag it as an orphan until config is written. No config artifacts are written on quit.
- **D-05:** The `[s]` skip-&-write escape writes all four artifacts without an authenticated PASS, but requires an explicit typed confirm with a "not yet authenticated; you must upload the key later" warning first.
- **D-06:** PASS = authenticated success (e.g. GitHub's "successfully authenticated" banner), NOT mere reachability. The current bug treats `Permission denied (publickey)` ("ReachableNotUploaded") as good-enough — that must no longer satisfy the gate.

#### Match strategy (MATCH-URL-01 / F-5, F-6)
- **D-07:** `identity add` prompts a picker first — `1) folder (gitdir)`, `2) repo URL (hasconfig)`, `3) both` — then prompts only for the value(s) the choice needs. Folder (`gitdir`) stays the default. "both" writes two `[includeIf]` conditions.
- **D-08:** The URL pattern prompt suggests a default derived from the host alias's hostname + identity name, e.g. `git@ssh.github.com:<name>/**` (user edits). Value is written as `hasconfig:remote.*.url:<pattern>`.
- **D-09:** Wire match-strategy editing into `identity update` too (change/add gitdir/URL/both on an existing identity), and add non-interactive flags (`--name/--gitdir/--url/--provider`) to `add`/`update`. *(Deliberate scope addition confirmed by the user.)*
- **D-10:** Relabel the `alias` prompt → "Host alias" (F-5).

#### Provider reconstruction (FIX-RECON-01 / F-3)
- **D-11:** Persist the user-entered provider as a marker comment inside the SSH managed Host block (e.g. `# gitid: provider=github`), invisible to `ssh` itself. The `sshconfig` parser (`ParseManagedHosts`) reads it back.
- **D-12:** On read, prefer the marker; fall back to a hostname→provider map (`ssh.github.com`/`github.com`→`github`, `altssh.gitlab.com`/`gitlab.com`→`gitlab`) for legacy/markerless blocks and custom hosts. Replaces the brittle `TrimPrefix(alias, name+".")` derivation at `loader.go:46-51`.
- **D-13:** Legacy identities (no marker) are tolerated via the fallback — no forced rewrite/migration. Markers get added naturally on the next `update`/`rotate`.

#### Doctor overlap check (DOC-08 / F-7)
- **D-14:** Overlap detection flags all three kinds: identical `gitdir`, nested/prefix `gitdir` (parent vs child dir), and overlapping `hasconfig` URL patterns (conservative heuristic acceptable for URL globs).
- **D-15:** Severity = warning (config still resolves last-wins; report which identity currently wins + how to disambiguate). Exit code stays in the warning tier.
- **D-16:** Detection is a shared function consumed by both `doctor` and the `add`/`update` write paths. At write time, an overlap triggers warn + explicit y/N to proceed.

#### Install feedback (FIX-INSTALL-01 / F-1)
- **D-17:** `gitid` self-reports the binary path + whether it is on `$PATH` (surfaced via doctor and/or an install/where hint), so the hint also works for a direct `go install` that bypasses `make`. `make install` also prints the resolved install path + a PATH export hint when not on `$PATH`.

## Summary

Phase 5.5 fixes five concrete defects found during guided UAT of Phase 5. All five map to existing seams in the codebase — this is a reconciliation phase, not a greenfield build. No new external packages are required. The central fix (FIX-CREATE-01) rewires the create-flow so key generation goes to `~/.ssh` immediately but the four config artifacts are held until an authenticated PASS is confirmed; this requires rearchitecting the `Create`/`runPipeline` orchestration in `internal/identity/identity.go` and `cmd/gitid/add.go`. Provider reconstruction (FIX-RECON-01) adds a `# gitid: provider=<p>` comment inside the managed SSH Host block body and upgrades `ParseManagedHosts` + `Reconstruct` to read it. Match strategy (MATCH-URL-01) adds a picker + flags to the existing prompt flow in `add.go`/`update.go`; the rendering layer (`RenderIncludeIf`) already accepts `[]Match` and already serializes `hasconfig:` conditions. Doctor overlap (DOC-08) adds a new shared check function in `internal/doctor/checks/` mirroring the existing carve-out pattern. Install feedback (FIX-INSTALL-01) uses `os.Executable()` + `exec.LookPath` + `go env GOBIN` to surface path and PATH membership.

The agent-driven E2E test harness is the most novel engineering element: the existing unit test pattern (`strings.NewReader` scripted stdin + `fakeDeps`) covers unit paths, but the connectivity-test loop (D-01..D-05) requires a fake `ssh` binary on PATH and a hermetic sandbox HOME to test deterministically without a real network or uploaded key.

**Primary recommendation:** Execute the fixes in dependency order: (1) provider marker write/read (all downstream readers must handle the new format first), (2) create-flow re-ordering and the PASS gate, (3) match strategy picker + flags, (4) overlap detector, (5) install feedback. E2E tests for each feature area run in `e2e/` with `//go:build e2e` tag, driven by `make test-e2e`.

### Pattern 1: Create-flow re-ordering (D-01..D-06)

**What:** The current `runPipeline` in `identity.go` uses a `StagedKey.TempPrivatePath` for the pre-write gate and only persists (`PersistKey`) after `Confirmed=true`. The new flow must:
1. Write the real key to `~/.ssh` immediately (D-01) — not temp-stage it.
2. Enter a loop that runs `PreWrite` against the final path and classifies the result.
3. On `PASS` → auto-persist the four config artifacts without an extra confirm (D-03).
4. On `[s]` skip → require a typed confirm, then persist (D-05).
5. On `[q]` quit → return without persisting config (D-04), key stays in `~/.ssh`.

**Key insight on the seam:** The current `Generate` dep in `buildDeps` writes to a temp dir. To implement D-01, `Generate` must write directly to the final `~/.ssh` path. `TempPrivatePath` and `FinalPrivatePath` become identical. The `Cleanup` dep becomes a true no-op (there is no temp dir to clean up). The `StagedKey.PrivPEM` sentinel (`nil` = existing key, non-nil = newly generated) is preserved for the `PersistKey` no-op guard — but since the key is now already on disk, `PersistKey` is only needed for the (now identical) final write, which is itself a no-op. **The simplest implementation:** in `buildDeps` (add.go), `Generate` writes directly to `~/.ssh` and returns `StagedKey{TempPrivatePath: finalPriv, FinalPrivatePath: finalPriv, ...}`. `Cleanup` becomes unconditional no-op.

**Loop control:** The loop prompt and `[r]/[s]/[q]` parsing belongs in the CLI layer (`runCreateNew` in `add.go`), not in `identity.Create`. The domain function receives a `LoopFunc` dep or returns an intermediate result the CLI layer interrogates. The cleanest seam: extract a `runCreateLoop(reader, out io.Writer, deps Deps, in CreateInput, staged StagedKey) (CreateResult, error)` function in `add.go` that drives the loop and calls `identity.PersistAll(in, staged, deps)` (a new exported function factoring out the four-writer sequence from `runPipeline`).

**Source:** [VERIFIED: codebase] — `internal/identity/identity.go:224-299`, `cmd/gitid/add.go:319-425`

### Pattern 2: Provider marker comment (D-11..D-13)

**What:** The `# gitid: provider=github` comment is placed inside the managed Host block body, not outside it. This is the only format that:
- Survives `cfg.String()` round-trip (kevinburke/ssh_config preserves comments inside Host blocks)
- Is invisible to `ssh` (comments are skipped by the OpenSSH parser)
- Lives inside the `# BEGIN gitid managed: <name>` / `# END gitid managed: <name>` sentinels (D-11, no-sidecar)

**Placement:** First line of the block body, before the `Host <alias>` line, or as the last line before the end sentinel. Recommended: **last line before the end sentinel** to keep the `Host` stanza clean and parseable without the comment interfering with any future `cfg.Get(alias, ...)` calls.

**RenderHostBlock change (renderer.go):**
```go
// Source: codebase — internal/sshconfig/renderer.go RenderHostBlock
// Add provider parameter; append marker comment at end of block body:
func RenderHostBlock(alias, hostname string, port int, identityFile, provider string) string {
    var b strings.Builder
    fmt.Fprintf(&b, "Host %s\n", alias)
    fmt.Fprintf(&b, "%sHostname %s\n", hostIndent, hostname)
    fmt.Fprintf(&b, "%sPort %d\n", hostIndent, port)
    fmt.Fprintf(&b, "%sUser git\n", hostIndent)
    fmt.Fprintf(&b, "%sIdentityFile %s\n", hostIndent, identityFile)
    fmt.Fprintf(&b, "%sIdentitiesOnly yes\n", hostIndent)
    if provider != "" {
        fmt.Fprintf(&b, "# gitid: provider=%s\n", provider)
    }
    return b.String()
}
```

**ParseManagedHosts change (reader.go):** `SSHHostInfo` gains a `Provider string` field. `parseHostBlockBody` scans block body lines for `# gitid: provider=<p>` before calling `ssh_config.Decode` (the comment is preserved in the raw `b.Body` string from `filewriter.ListBlocks`).

**Round-trip safety:** `filewriter.ListBlocks` extracts the raw body bytes between sentinels (not the parsed ssh_config representation). The `# gitid: provider=` line is a comment inside those bytes, so it survives `ReplaceBlock` → re-read cycles byte-identically. A `Parse → String → Parse` stability check (as recommended by CLAUDE.md) confirms this.

**Backward compat (D-13):** When no marker is found, `SSHHostInfo.Provider` is empty. `Reconstruct` in `loader.go` applies the hostname→provider map fallback (D-12):
```go
// Source: ASSUMED (derived from codebase pattern)
var hostnameToProvider = map[string]string{
    "ssh.github.com":    "github",
    "github.com":        "github",
    "altssh.gitlab.com": "gitlab",
    "gitlab.com":        "gitlab",
}
```
The `TrimPrefix(alias, name+".")` derivation at `loader.go:46-51` is removed.

### Pattern 3: Match strategy picker (D-07..D-10)

**What:** The existing `prompt(r, out, "Match gitdir", ...)` in `gatherCreateInput` (`add.go:282`) is replaced by:
1. A picker: `"Match strategy: 1) folder (gitdir)  2) repo URL (hasconfig)  3) both [1]: "`
2. Conditional value prompt(s) based on the choice.

The result is passed as `[]gitconfig.Match` to `CreateInput.Matches` — the same field that already holds the single-gitdir match today. `RenderIncludeIf` already serializes both kinds correctly [VERIFIED: codebase — `internal/gitconfig/renderer.go:59-81`].

**Cobra flags (D-09):** Add to `newAddCmd` and `newUpdateCmd`:
```
--name <string>      identity name (non-interactive; skips name prompt)
--gitdir <string>    gitdir match value (non-interactive; skips gitdir prompt)
--url <string>       hasconfig URL pattern (non-interactive; skips URL prompt)
--provider <string>  provider name (non-interactive; skips provider prompt)
```
When a flag is present, skip the corresponding prompt and use the flag value. Flags and prompts are not exclusive — a user can provide `--name` and still get prompted for the rest. This is the standard Cobra/CLI pattern [ASSUMED].

**Default URL pattern (D-08):** Derived from the alias hostname + identity name:
```
// alias = "user_z3r0_gh.github" → hostname = "ssh.github.com", name = "user_z3r0_gh"
// default URL = git@ssh.github.com:user_z3r0_gh/**
```
Format: `git@<hostname>:<name>/**` → written as `hasconfig:remote.*.url:git@<hostname>:<name>/**`.

**Update flow wiring (D-09):** `runIdentityUpdate` in `update.go` currently has a single `prompt(reader, out, "Match gitdir", ...)` at line 140 that collapses all `existing.Matches` to a gitdir string via `renderMatches`. Replace with the same picker + value prompt(s) as `add.go`, pre-filled with the existing match(es).

### Pattern 4: Doctor overlap detection (D-14..D-16)

**What:** A new `internal/doctor/checks/overlap.go` file exposes a shared function:
```go
// Source: ASSUMED (derived from existing doctor/checks pattern in codebase)
// CheckOverlap detects ambiguous/overlapping includeIf match conditions across identities.
func CheckOverlap(deps doctor.Deps) []doctor.Finding
// DetectOverlaps returns the raw overlap pairs — exported for use by add/update write paths.
func DetectOverlaps(accounts []identity.Account) []OverlapPair
```

`CheckOverlap` calls `DetectOverlaps` and wraps results in `doctor.Finding{Severity: SeverityWarning}`.

The `add`/`update` write paths in cmd/ call `DetectOverlaps` before persisting and, if overlaps are found, print the warning and prompt `y/N to proceed` (D-16).

**Three overlap kinds (D-14):**
1. **Identical gitdir** — two identities have the same `gitdir:` value after trailing-slash normalization.
2. **Nested gitdir** — one gitdir value is a prefix of another (e.g. `~/git/` and `~/git/personal/`). The parent captures all commits the child captures, creating ambiguity about which identity "wins" for the child path.
3. **Overlapping hasconfig URL globs** — a conservative heuristic: two `hasconfig:remote.*.url:` patterns overlap when one is a prefix of the other or they share the same `git@<hostname>:` prefix with one using a wildcard that subsumes the other. Since git glob matching is implementation-defined, a conservative over-approximation is explicitly acceptable (D-14).

**Reserved block exclusion:** Mirror the existing `gitconfig.IsReservedBlockName(name)` and `_global` skip patterns from `orphans.go:82-88` and `reader.go:26-41`. The overlap check must only evaluate actual identity accounts, not reserved wiring blocks. [VERIFIED: codebase]

**Severity and exit code (D-15):** Warning severity. Doctor exit code stays in tier 1 (warnings/info only). The text surfaces the current "last-wins" winner identity name to guide disambiguation.

**Last-wins semantics:** git evaluates `[includeIf]` blocks in file order; the last matching `[path = ...]` for a given key wins. Since `WriteIncludeIf` uses `filewriter.ReplaceBlock`, the order of managed blocks in `~/.gitconfig` is the write order (most recently written identity wins for a given key conflict). The overlap warning should note: "git resolves last-match-wins; the identity written most recently to `~/.gitconfig` will take precedence." [ASSUMED — derived from git documentation]

### Pattern 5: Install feedback (D-17)

**What:** Two surfaces:

**Surface A — `gitid` self-report:**
```go
// Source: ASSUMED (Go stdlib os.Executable)
func binaryInfo() (path string, onPATH bool) {
    exe, err := os.Executable()     // returns resolved binary path
    if err != nil { return "", false }
    exe, _ = filepath.EvalSymlinks(exe)
    _, lookErr := exec.LookPath(filepath.Base(exe))
    return exe, lookErr == nil
}
```
Surface this via `gitid doctor` (a new "Install" or "Binary" finding family or an info-level finding under Deps), or as a standalone `gitid where` subcommand. The CONTEXT.md locks D-17 to "doctor and/or an install/where hint."

**Surface B — `make install` echo:**
```makefile
# Source: ASSUMED (derived from existing Makefile pattern)
install:
	go install ./cmd/gitid
	@INSTALL_PATH="$(GOPATH_BIN)/gitid"; \
	echo "  installed: $$INSTALL_PATH"; \
	echo "$$PATH" | tr ':' '\n' | grep -qF "$(GOPATH_BIN)" \
	  && echo "  PATH: OK (gitid is findable)" \
	  || echo "  PATH: $(GOPATH_BIN) is NOT on your PATH — add: export PATH=$$PATH:$(GOPATH_BIN)"
```

**GOBIN vs GOPATH/bin:** `go env GOBIN` returns empty string when GOBIN is not explicitly set; in that case the effective install dir is `$(go env GOPATH)/bin`. The Makefile already uses `GOPATH_BIN := $(shell go env GOPATH)/bin` and `go install ./cmd/gitid` uses the effective GOBIN. The self-report logic in `binaryInfo()` must replicate the same fallback. ### Anti-Patterns to Avoid
- **Business logic in cmd/:** The loop exit condition (PASS / skip-confirm / quit) is domain logic and must live in a function callable by both the CLI and the future TUI. cmd/ provides the `io.Writer` for prompts and reads the `reader *bufio.Reader` for input — the loop function itself belongs in `identity` package or a new `internal/createflow` package.
- **Provider in a sidecar file:** Provider must live in the managed SSH Host block comment (D-11). No `~/.config/gitid/` state files.
- **Calling `ssh` with a hardcoded path:** `exec.Command("ssh", ...)` uses PATH resolution (verified: `tester.go:106`). In E2E tests, prepend a dir containing the fake `ssh` script to `PATH` — do NOT hardcode `/usr/bin/ssh` in production code.
- **Treating `Permission denied (publickey)` as PASS:** This is the central reconciliation bug. `tester.ReachableNotUploaded` is explicitly NOT a passing outcome for the persist gate (D-06).
- **Applying overlap check to reserved/global blocks:** Use `gitconfig.IsReservedBlockName(name)` and check for `_global` SSH block name before feeding accounts to the overlap detector.

## E2E Test Harness (Agent-Driven)

This is the most critical new engineering work in Phase 5.5 and is user-mandated.

### What "agent-driven E2E" means operationally

An agent (or a human) can run `make test-e2e` and get a deterministic, hermetic result asserting the full observable behavior of the binary — not just internal functions — including disk artifacts, exit codes, and stdout. No network access, no real SSH keys uploaded.

### Fake SSH strategy

**The seam:** `exec.Command("ssh", args...)` in `tester.go:106` resolves `ssh` via `os.Getenv("PATH")`. This is the injection point.

**The approach:** Place a fake `ssh` script at a temp dir that precedes `/usr/bin` on PATH. The fake script reads from `$GITID_FAKE_SSH_MODE` and emits the corresponding output:

```bash
#!/bin/sh
# e2e/testdata/fake-ssh/ssh
case "$GITID_FAKE_SSH_MODE" in
  pass)     echo "Hi user! You've successfully authenticated, but GitHub does not provide shell access." ; exit 1 ;;
  denied)   echo "git@ssh.github.com: Permission denied (publickey)." ; exit 255 ;;
  timeout)  echo "ssh: connect to host ssh.github.com port 443: Operation timed out" ; exit 255 ;;
  *)        echo "git@ssh.github.com: Permission denied (publickey)." ; exit 255 ;;
esac
```

Note: exit code is discarded by `tester.go` (D-01/Pitfall 2). The fake can exit with any code; only output matters.

**Also fake `ssh -G <alias>`:** The `Resolved` dep calls `exec.Command("ssh", "-G", alias)`. The fake `ssh` script must detect `-G` and emit a fixture `ssh -G` output:
```bash
#!/bin/sh
case "$1" in
  -G) echo "user git"; echo "hostname ssh.github.com"; echo "port 443"; echo "identitiesonly yes"; echo "identityfile /home/test/.ssh/id_ed25519_test" ; exit 0 ;;
  ...
esac
```

**gosec concern (G-204):** The fake `ssh` is only used in test code. `exec.Command("ssh", ...)` in production is gosec-clean (arg-slice form). The test harness does NOT call `exec.Command` with user-controlled strings — the fake script path is controlled by the test. The `//nolint:gosec` annotation in `tester.go` already covers production. [ASSUMED — gosec G-204 applies to shell-form commands, not arg-slice]

**FakeSSHDir helper — no shell invocation:** The Go test helper that installs the fake script must use `os.WriteFile` + `os.Chmod`, NOT `exec.Command("sh", "-c", ...)`. The shell script content is a static string literal in the test helper — it is not constructed from user input and is never passed to a shell interpreter from Go code:

```go
// Source: ASSUMED (standard Go test helper pattern)
func FakeSSHDir(t *testing.T, mode string) (dirPath string) {
    t.Helper()
    dir := t.TempDir()
    script := "#!/bin/sh\ncase \"$GITID_FAKE_SSH_MODE\" in\n  pass) echo \"Hi user! You've successfully authenticated...\" ; exit 1 ;;\n  denied) echo \"git@ssh.github.com: Permission denied (publickey).\" ; exit 255 ;;\n  *) echo \"git@ssh.github.com: Permission denied (publickey).\" ; exit 255 ;;\nesac\n"
    scriptPath := filepath.Join(dir, "ssh")
    if err := os.WriteFile(scriptPath, []byte(script), 0o700); err != nil { //nolint:gosec // test-only static script, not user input
```

The E2E test invokes the real gitid binary via `exec.Command(bin, "identity", "add")` (arg-slice form). The fake `ssh` executes as a standalone process when gitid's internal `tester.execRunner` calls `exec.Command("ssh", args...)` — PATH resolution finds the stub first.

## Common Pitfalls

### Pitfall 1: Loop prompt ordering — stdin exhaustion
**What goes wrong:** The interactive loop (retry/skip/quit) reads additional lines from stdin. A scripted test that does not include the loop prompt response will see the bufio.Reader hit EOF and return empty string, which may default to an unexpected action.
**Why it happens:** `bufio.Reader.ReadString('\n')` on EOF returns the buffered bytes + `io.EOF`; `strings.TrimSpace("")` yields `""` which maps to the default branch in the loop selector.
**How to avoid:** Always append an explicit trailing `\n` to scripted stdin, and include the expected loop response (e.g. `"q\n"` for quit). Document the full prompt sequence in `harness_test.go`.
**Warning signs:** Test hangs (waiting for loop input that never arrives) or exits with unexpected behavior.

### Pitfall 2: SSH round-trip: `# gitid: provider=` comment position
**What goes wrong:** If the provider marker comment is placed before the `Host <alias>` line, kevinburke/ssh_config may parse it as a top-level comment (not a Host stanza comment) and `cfg.Get(alias, ...)` may return empty values for that stanza.
**Why it happens:** The kevinburke parser associates comments with the *following* stanza node, not the preceding one. A comment placed before `Host` belongs to the Host node; a comment placed inside the stanza (after `Host`) belongs to the containing Host node as a sub-comment.
**How to avoid:** Place `# gitid: provider=<p>` as the **last line of the Host stanza body**, after `IdentitiesOnly yes`, before the end sentinel. Verify with a parse→render→parse stability test.
**Warning signs:** `ParseResolved` returns empty fields for the alias after writing the marker.

### Pitfall 3: gosec G-204 / shell injection in E2E test helpers
**What goes wrong:** A test helper that writes the fake `ssh` binary using `exec.Command("sh", "-c", "chmod +x " + path)` or constructs any shell command string with a variable argument triggers gosec G-204 and is a command injection risk.
**Why it happens:** `exec.Command("sh", "-c", someVar)` passes the argument through a shell interpreter; special characters in `someVar` are treated as shell metacharacters.
**How to avoid:** Never use `exec.Command("sh", "-c", ...)` for test setup. Use stdlib instead:
- Write the script body with `os.WriteFile(path, []byte(staticContent), 0o700)` — no `chmod` call needed.
- The fake script content is a static string literal, never constructed from user input.
- The E2E test itself invokes `exec.Command(bin, "identity", "add")` (arg-slice) — no shell involved.
- PATH injection is done via `cmd.Env = append(os.Environ(), "PATH="+fakeDir+":"+os.Getenv("PATH"))` — no shell string.
**Warning signs:** golangci-lint reports G-204 on a test file; any `"sh", "-c"` call in `e2e/`.

### Pitfall 4: Overlap detection including `_global` SSH block
**What goes wrong:** The `_global` block (macOS `Host *` stanza) appears in `filewriter.ListBlocks` output. If the overlap detector iterates `SSHManagedBlockNames` without skipping `_global`, it may try to match it against identity names and emit spurious overlap findings.
**Why it happens:** `ParseManagedHosts` skips `_global` (reader.go:29-31) but a new overlap check that reads raw block names must apply the same skip.
**How to avoid:** In `DetectOverlaps`, filter the input `[]identity.Account` — accounts are already reconstructed via `Reconstruct` which skips `_global` and reserved gitconfig blocks. Source the overlap check from the account list, not raw block names. [VERIFIED: codebase — sshconfig/reader.go:29-31]

### Pitfall 5: Confirmed=false vs new PASS-gate flow
**What goes wrong:** `CreateInput.Confirmed` was used as the persist gate. After the refactor, `Confirmed` must be removed from `CreateInput` or repurposed — the PASS gate is now dynamic (loop result), not a pre-gathered static field. If the field remains but is always `false`, nothing will be written.
**Why it happens:** The old flow set `in.Confirmed = confirm(r, out, "Write all four artifacts now?")` at gather-time (`add.go:312`). The new flow must NOT ask this question upfront.
**How to avoid:** Remove `Confirmed` from `CreateInput` (or keep it but always pass `true` from the loop after a PASS/skip-confirm). The persist decision belongs to the loop, not to input gathering. Audit all callers of `Create`/`runPipeline` — `Rotate` and `Reuse` also use `runPipeline` with `Confirmed: true` (already correct) and must not be broken by the refactor. [VERIFIED: codebase — identity.go:262, modes.go:93, modes.go:131]

### Pitfall 6: `identity update` match field collapse
**What goes wrong:** The current `update.go:140-141` collapses `existing.Matches` (which may contain both gitdir and hasconfig conditions) to a single gitdir string via `renderMatches`. After this phase, matches may be a multi-element `[]Match`. Writing only a single gitdir drops the hasconfig condition silently.
**Why it happens:** `renderMatches` was written for the single-gitdir case.
**How to avoid:** The match strategy picker in `update.go` must read `existing.Matches`, pre-fill the picker with the current strategy type (gitdir/hasconfig/both), and let the user edit each value. The updated `[]Match` replaces `existing.Matches` wholesale. [VERIFIED: codebase — update.go:134-141]

### Pitfall 7: `update.go` provider prompt omission note
**What goes wrong:** The update.go comment at line 125-130 explicitly explains that Provider is NOT prompted because it cannot round-trip (it was derived at read time, not persisted). After D-11, provider IS now persisted (marker comment). The comment must be updated and the Provider prompt restored (or the `--provider` flag used).
**How to avoid:** Add `--provider` flag to `update.go` (D-09). On `update`, if `--provider` is given or the user changes the alias, re-derive and re-write the marker in the new SSH Host block. [VERIFIED: codebase — update.go:125-130]

### Overlap detection — gitdir prefix check (heuristic)
```go
// Source: ASSUMED (derived from domain description D-14)
func gitdirOverlaps(a, b string) bool {
    // Normalize trailing slash
    a = strings.TrimSuffix(a, "/") + "/"
    b = strings.TrimSuffix(b, "/") + "/"
    if a == b {
        return true // identical
    }
    // Nested: one is a prefix of the other
    return strings.HasPrefix(a, b) || strings.HasPrefix(b, a)
}
```

### os.Executable + LookPath for install feedback
```go
// Source: ASSUMED (Go stdlib)
func binaryInstallInfo() (path string, onPATH bool, err error) {
    exe, err := os.Executable()
    if err != nil {
        return "", false, err
    }
    exe, err = filepath.EvalSymlinks(exe)
    if err != nil {
        return exe, false, nil
    }
    _, lookErr := exec.LookPath(filepath.Base(exe))
    return exe, lookErr == nil, nil
}
```

## Open Questions (RESOLVED)

All three resolved in CONTEXT.md `### Claude's Discretion` (2026-06-14) and implemented in the plans.

1. **Loop prompt during `--dry-run`**
   - What we know: `--dry-run` currently skips `Confirmed` and returns `PreWriteOnly=true`
   - What's unclear: Should `--dry-run` also run the key generation to `~/.ssh` (so the `.pub` is available for the user to see), or skip all writes including keygen?
   - RESOLVED: Dry-run skips all writes including keygen (consistent with current semantics — SAFE-03). No key is generated on dry-run; user sees a preview of all four artifacts.

2. **`--name` flag vs `cobra.ExactArgs(1)` on `update`**
   - What we know: `update.go` uses `cobra.ExactArgs(1)` — name is a positional argument.
   - What's unclear: D-09 says add `--name` flag to `update`. This conflicts with the existing positional arg design.
   - RESOLVED: Keep positional arg on `update` (consistent with existing `delete`, `rotate`); `--name` flag is only on `add` where name is currently prompted.

3. **Provider prompt re-introduction on `update`**
   - What we know: `update.go:125-130` explicitly comments that provider is not prompted because it cannot round-trip. After D-11, it CAN round-trip via the marker.
   - What's unclear: Should the update flow re-introduce a `--provider` flag/prompt to overwrite the marker?
   - RESOLVED: Add `--provider` flag to `update` (flag only, no interactive prompt); only write a new marker if the flag is explicitly set. If no flag, keep the existing marker unchanged.
