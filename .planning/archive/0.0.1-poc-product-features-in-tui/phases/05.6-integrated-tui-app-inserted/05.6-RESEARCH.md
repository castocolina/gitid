## Summary

Phase 5.6 replaces the existing `tui/` screen-stack architecture (rootModel + screen stack) with a persistent two-pane layout app shell. The Bubble Tea v2 + Lipgloss v2 + Bubbles v2 stack is already pinned and in active use in the repository — no dependency changes are needed. The existing patterns are well-established and proven: async `tea.Cmd` goroutines for doctor streaming, runID stale-guard, spinner.Tick seeding, the initializer interface for push-time init, and the `tea.View` (struct, not string) v2 return type from `View()`.

The architectural shift is from screen-stack navigation (push/pop screenModel) to a persistent layout with a single `activeView` enum and a single `activeModal` field. The sidebar remains permanently rendered; views swap the right pane content; modals layer over the entire rendered layout via a string-overlay technique (render full layout, apply `StyleDimmed.Render()` to it, then `lipgloss.Place()` a modal box on top).

The proven logic blocks from prove.go, dashboard.go/deps.go, createform.go, updateform.go, addaccountform.go, and copy.go must be ported — they are the implementation of the create wizard, health streaming, inline editing, and copy workflows. These modules are being kept; only the navigation architecture around them changes.

**Primary recommendation:** Rewrite `tui/model.go` to a two-pane persistent shell; migrate sub-models from `screenModel` to inline sub-model fields on the root; keep the `tea.Cmd`/`tea.Msg` async patterns and the `tuiDeps` injection architecture unchanged. The new Delete and Rotate workflows require wiring `identity.DeleteDeps` and re-using the Rotate flow from `identity/modes.go`.

### Pattern 1: Two-Pane Persistent Layout Rendering (Lipgloss v2)

**What:** Compose header + sidebar + main pane + footer using `lipgloss.JoinVertical` and `lipgloss.JoinHorizontal`. Sidebar is 18 columns fixed; main pane fills the remainder.

**When to use:** Every `View()` call when no modal is active.

```go
// Source: lipgloss v2 docs / tui/styles.go existing pattern
func (m rootModel) renderPersistentLayout() string {
    sidebarWidth := 18
    mainWidth := m.width - sidebarWidth - 3  // 3 for border cells

    sidebarStr := lipgloss.NewStyle().
        Width(sidebarWidth).
        Height(m.height - 3). // subtract header + footer rows
        Render(m.sidebar.view())

    mainStr := lipgloss.NewStyle().
        Width(mainWidth).
        Height(m.height - 3).
        Border(lipgloss.NormalBorder(), false, false, false, true).
        BorderForeground(lipgloss.Color("8")).
        Render(m.activeViewContent())

    body := lipgloss.JoinHorizontal(lipgloss.Top, sidebarStr, mainStr)
    return lipgloss.JoinVertical(lipgloss.Left, m.renderHeader(), body, m.renderFooter())
}
```

**Key constraint:** `lipgloss.NewStyle()` — NOT `renderer.NewStyle()` (does not exist in v2). [VERIFIED: existing tui/styles.go confirms this pattern is already in use]

### Pattern 2: Modal Overlay Over Dimmed Layout (Lipgloss v2 string-overlay technique)

**What:** Lipgloss v2 has no native z-layer or compositing API. The standard technique is:
1. Render the full persistent layout to a string
2. Apply `StyleDimmed.Render()` (Faint) to it
3. Center the modal box string over the terminal using `lipgloss.Place()`
4. The dimmed background + modal together form the visual; they cannot be truly composited (the dimmed string is "behind" the modal in the rendered output but there is no pixel compositing — each terminal line is either background or modal)

**Implementation approach (line-by-line overlay):**

```go
// Source: lipgloss v2 pattern documented in tui/styles.go (existing); Place API [ASSUMED]
func (m rootModel) renderWithModal(base, modal string) string {
    dimmed := StyleDimmed.Render(base)
    centered := lipgloss.Place(
        m.width, m.height,
        lipgloss.Center, lipgloss.Center,
        modal,
        lipgloss.WithWhitespaceChars(" "),
    )
    // Note: lipgloss.Place does NOT composite over dimmed — it produces a new string.
    // For a true overlay: use the PlaceOverlay helper from lipgloss v2 if available,
    // OR render the modal as a foreground-only string and manually merge lines.
    // Current best practice (verified from lipgloss v2 README / issues): render dimmed
    // base as a background string, then overlay the modal using PlaceOverlay.
    // See Pitfall 3 for the exact API.
    return centered
}
```

**IMPORTANT - Pitfall 3 clarification:** See the Pitfall 3 section below for the confirmed `lipgloss.PlaceOverlay` API.

### Pattern 3: Modal State Machine (single activeModal field)

**What:** Root model holds a single `activeModal` field (not a stack). Only one modal open at a time (D-15 modal stack rule: single `activeModal`, max depth 1).

```go
type modalKind int
const (
    noModal modalKind = iota
    createWizardModal
    deleteConfirmModal
    rotateConfirmModal
    fixConfirmModal
    copyPubkeyModal
    paletteModal
    helpModal
)

type rootModel struct {
    // layout
    width, height int
    // view
    activeView   viewKind
    // modal
    activeModal  modalKind
    wizardModel  createWizardModel   // populated when activeModal == createWizardModal
    confirmModel confirmModel        // populated when activeModal == delete/fix/rotateConfirmModal
    paletteModel paletteModel        // populated when activeModal == paletteModal
    // panes
    sidebar      sidebarModel
    detailPane   identityDetailModel
    healthPane   healthViewModel
    globalPane   globalOptionsModel
    // toast
    toast        string
    deps         tuiDeps
}
```

### Pattern 4: Async tea.Cmd with runID Stale-Guard (port from dashboardModel)

**What:** Each async operation gets a `runID int` field on its enclosing sub-model. The `tea.Cmd` captures `runID` at dispatch time; the result message carries it back. In `Update`, drop messages whose `runID` doesn't match the current one.

```go
// Source: tui/dashboard.go (existing, proven pattern)
type familyResultMsg struct {
    runID    int
    family   doctor.Family
    findings []doctor.Finding
    err      error
}

// On refresh: m.runID++; dispatch new Cmds with new runID; drop msgs with old runID.
```

**This pattern must be preserved** for the Health view async streaming (D-11). Extend it to any new async operations (delete, fix, rotate).

### Pattern 5: Spinner Initialization — Must Seed Tick on Init

**What:** `spinner.Model` requires an initial `m.sp.Tick` command to start animating. Without it, the spinner renders but never advances.

```go
// Source: tui/dashboard.go dashboardModel.init() + tui/prove.go proveModel.init()
func (m healthModel) init() (healthModel, tea.Cmd) {
    cmds := make([]tea.Cmd, 0)
    for _, fam := range doctor.Families() {
        cmds = append(cmds, makeFamilyCmd(m.runID, fam, m.deps.doctor))
    }
    for i := range m.spinners {
        cmds = append(cmds, m.spinners[i].Tick)  // REQUIRED: seed initial tick
    }
    return m, tea.Batch(cmds...)
}
```

### Pattern 6: tea.View (v2 breaking change — View() returns tea.View, not string)

**What:** In Bubble Tea v2, the root model's `View()` method returns `tea.View` (a struct), NOT a string. Alt-screen is set via `v.AltScreen = true` on the returned `tea.View` — NOT via `tea.WithAltScreen()` which is a v1 option.

```go
// Source: tui/model.go (existing, verified)
func (m rootModel) View() tea.View {
    v := tea.NewView(m.renderContent())
    v.AltScreen = true
    return v
}
```

Sub-models (sidebar, detail, health, etc.) return `string` from their `view()` method (lowercase); only the root model returns `tea.View`.

### Pattern 7: Port prove.go Into a Wizard Modal Sub-Model

**What:** The existing `proveModel` is a full-screen screen-stack screen. For Phase 5.6, the same logic (phase1Running → phase1Done → phase2Running → phase2Done → written) must run inside a modal overlay. The state machine is identical; the change is that `view()` renders into a `StyleModal`-bordered box at bounded width (`min(terminalWidth-8, 72)`), and key handling returns `clearModalCmd` instead of `popCmd`.

**Port strategy:**
- Copy `proveModel` struct and its `update`/`view` methods as `wizardProveModel`
- Change `view()` to return modal-bounded content (no full-width title, no footers — modal footer is rendered by rootModel)
- Replace `popCmd()` with `clearModalCmd()` (sets `m.activeModal = noModal`)
- The `initScreen()` / `initializer` pattern becomes `init()` called when `activeModal` is set to `createWizardModal`

### Anti-Patterns to Avoid

- **Screen-stack for view switching:** The old `pushCmd(newScreen)` / `popCmd()` pattern for switching between Identities/Health/Global Options is WRONG in the new architecture. Views switch via `m.activeView = healthView`. Screen-stack remains valid ONLY for the pre-existing wizard/prove/form use case, which is now ported to modal overlays.
- **Blocking in Update():** Never call doctor check functions or file I/O synchronously in `Update()`. Always wrap in `tea.Cmd` goroutines. Pitfall preserved from Phase 5.
- **renderer.NewStyle():** Does not exist in lipgloss v2. Use `lipgloss.NewStyle()` only.
- **tea.WithAltScreen():** v1 option; does not exist in v2. Use `v.AltScreen = true` on the returned `tea.View`.
- **Multiple modals:** The spec (D-15 modal stack rule) mandates at most one modal at a time. Implement as a single `activeModal` enum field, not a modal stack slice.
- **Width/height assumptions:** Always use `m.width` and `m.height` from `tea.WindowSizeMsg`. Never hardcode terminal dimensions.
- **nil CheckFn panic:** As learned in Phase 5 (dashboard.go makeFamilyCmd), a nil `CheckFn` must error explicitly rather than silently passing. Keep the nil-guard pattern.

## Common Pitfalls

### Pitfall 1: AltScreen and View() Return Type (v2 Breaking Change)

**What goes wrong:** Calling `View() string` as in v1 causes a compile error. Using `tea.WithAltScreen()` as a program option (v1 pattern) fails silently or doesn't compile in v2.
**Why it happens:** Bubble Tea v2 changed `View()` to return `tea.View` (a struct). AltScreen is now a field on `tea.View`.
**How to avoid:** Confirmed in `tui/model.go`: `func (m rootModel) View() tea.View { v := tea.NewView(content); v.AltScreen = true; return v }`. Keep this pattern exactly.
**Warning signs:** Compile error "cannot use string as tea.View" or "tea.WithAltScreen undefined".

### Pitfall 2: lipgloss.NewStyle() vs renderer.NewStyle()

**What goes wrong:** Calling `renderer.NewStyle()` results in "undefined: renderer" — this method does not exist in lipgloss v2.
**Why it happens:** v1 had a `Renderer` type with `NewStyle()`; v2 removed it.
**How to avoid:** Confirmed in `tui/styles.go`: always `lipgloss.NewStyle()`. UI-SPEC explicitly calls this out.
**Warning signs:** `renderer undefined` or `lipgloss.Renderer has no method NewStyle`.

### Pitfall 3: Modal Overlay — PlaceOverlay API in Lipgloss v2

**What goes wrong:** `lipgloss.Place()` does NOT composite one string over another — it places content in a terminal-sized area of whitespace. To render a modal ON TOP of a dimmed background, you need `lipgloss.PlaceOverlay()`.
**Why it happens:** Confusion between "place in terminal" (`Place`) and "overlay on background string" (`PlaceOverlay`).
**How to avoid:** [ASSUMED — verify before implementing] The `lipgloss.PlaceOverlay(x, y int, fg, bg string, opts ...WhitespaceOption) string` function takes foreground and background string renders and composites them line-by-line. If `PlaceOverlay` is not available in v2.0.3, the fallback is: split both strings by `\n`, replace lines in the background string within the modal's bounding box.
**Warning signs:** The modal renders but covers only whitespace, not the dimmed layout. Or compile error if `PlaceOverlay` doesn't exist at this version.

**Recommended implementation for the planner:** Include a Wave 0 task to verify `lipgloss.PlaceOverlay` existence at v2.0.3 by checking its exported symbols, and implement accordingly.

### Pitfall 4: Spinner Never Animates (Missing Initial Tick)

**What goes wrong:** Spinner renders but stays on frame 0 forever.
**Why it happens:** `spinner.TickMsg` only re-arms if a tick has already been received. Without seeding `m.sp.Tick` in `init()`, no tick ever arrives.
**How to avoid:** Confirmed in `tui/prove.go` and `tui/dashboard.go`: always `tea.Batch(mainCmd, m.sp.Tick)` in any `init()` or startup cmd that starts a spinner.
**Warning signs:** Loading spinner shows but doesn't move.

### Pitfall 5: Stale familyResultMsg After Refresh

**What goes wrong:** Refreshing the Health view while old family results are still in-flight causes the new run's findings to be overwritten by the old run's results.
**Why it happens:** `tea.Cmd` goroutines are not cancellable; the old goroutines complete and deliver results after `m.runID` has been incremented.
**How to avoid:** Confirmed in `tui/dashboard.go`: check `msg.runID != m.runID` and return early on mismatch. Port this pattern to the new `healthViewModel`.
**Warning signs:** Finding counts change unexpectedly after refresh; some families show stale data.

### Pitfall 6: buildTUIDeps Replication vs. package main Imports

**What goes wrong:** The TUI cannot import `package main` (cmd/gitid). All dependency wiring that `cmd/gitid/doctor.go` and `cmd/gitid/add.go` do must be replicated in `tui/deps.go`.
**Why it happens:** Go's import cycle rules; `package main` is not importable.
**How to avoid:** Confirmed in `tui/deps.go`: `buildTUIDeps()` replicates `buildDoctorDeps()` and `buildDeps()` from `cmd/gitid`. For Phase 5.6, add `buildTUIDeleteDeps()` following the same pattern.
**Warning signs:** Missing wiring leads to nil function fields → the nil-guard in `makeFamilyCmd` will surface them as errors at runtime, but delete/rotate/fix nils will panic without a guard.

### Pitfall 7: D-16 Blindspot — Tests Green on Injected Seams, Real Path Unwired

**What goes wrong:** All unit tests pass (using `fakeDocDeps()` / `fakeIdentityDeps()`), but the real program crashes or silently fails because the seam wiring in `buildTUIDeps()` has a nil function field or wrong path.
**Why it happens:** Tests drive the model with fake deps; `Run()` builds real deps. The two paths diverge when wiring is added to fakes but forgotten in real builders.
**How to avoid:** D-16 mandate: every verification step must launch `gitid` with no arguments on a real TTY and observe real behavior (create, navigate, delete, run doctor). The `wiring_test.go` in Phase 5 was introduced for exactly this reason — extend it for the new capabilities.
**Warning signs:** All unit tests green but `gitid` crashes on startup or on first real action.

### Pitfall 8: Width Accounting in Lipgloss — Borders Consume Columns

**What goes wrong:** Setting `Width(18)` on a bordered panel produces a panel that is 18 columns in content but 20 columns total (1 border each side).
**Why it happens:** Lipgloss `Width()` sets CONTENT width; borders add on top.
**How to avoid:** Account for border width when computing `mainWidth = m.width - sidebarWidth - borderCells`. The UI-SPEC already specifies `mainWidth = terminalWidth - 18 - 3` (sidebar 18 + 3 for the vertical separator border). Follow this formula exactly.
**Warning signs:** Sidebar + main pane overflow the terminal width; content wraps.

### Pitfall 9: Tab Key in Bubble Tea vs. Textinput — Key Routing Priority

**What goes wrong:** When a `textinput.Model` is focused, `Tab` is consumed by the textinput before the TUI can use it to cycle focus between panes.
**Why it happens:** Bubbles `textinput.Model` intercepts Tab for cursor movement if not explicitly told to pass it through.
**How to avoid:** When NOT in inline-edit mode, ensure textinput models are blurred (`m.field.Blur()`). When IN inline-edit mode, Tab should advance fields (not switch panes). The root model routes Tab differently depending on `m.inlineEditMode` state.
**Warning signs:** Tab stops working for pane focus cycling when any form field exists on screen.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | Not applicable — TUI only calls the core's existing SSH test pipeline |

### Known Threat Patterns for TUI Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Private key display / copy | Information Disclosure | Copy is public key only (D-13, locked); private key path shown as `StyleFaint` text only, never clipboard |
| File writes without backup | Tampering | `identity.DeleteDeps` / `identity.UpdateDeps` / `identity.Deps` all route through `filewriter.Write` with timestamped backup |
| Delete without confirm | Tampering | Delete always requires modal confirm (D-09); Esc is the safe default (Accessibility Contract item 4) |
| Structural edit without re-test | Tampering | D-07: structural edits (alias, hostname, port, match strategy) MUST open the prove-before-write loop |
| Clipboard injection via crafted hostname | Tampering | `atotto/clipboard` dispatches to `pbcopy`/`wl-copy`/`xclip` with the raw public key string; no shell interpolation |
| Goroutine panic crashing TUI | Denial of Service | `recover()` wrap in all `tea.Cmd` goroutines (pattern from `tui/dashboard.go` `makeFamilyCmd`) |
