# Phase 5.6: Integrated TUI App (INSERTED) - Context

**Gathered:** 2026-06-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the thin doctor-dashboard TUI with **one integrated Bubble Tea v2 terminal
application** built on the proven Phase 5.5 core. The app presents a persistent
layout — header + left identity sidebar + master-detail main pane + bold footer —
with a view switcher across **Identities · Health (doctor) · Global Options
(baseline)**. Every capability is performed in-app with live feedback: the create/add
wizard (the 5.5 prove-before-write loop), in-pane editing of per-site and global
options, delete/rotate, and copy (public key only). Nothing hands off to the CLI.
Runs over SSH; ergonomics modeled on `../tools-installer` (a Textual app).

**This phase supersedes** the Phase 5 TUI surface (TUI-01/TUI-02). The CLI command
surface from Phase 5 stays the scriptable engine and is NOT changed here.

**In scope:** the integrated app shell + layout, the three views, the in-app create
wizard, inline editing, in-app delete/rotate/copy, in-app doctor + auto-fix, a
read-only "unmanaged" section (Host blocks + unlinked keys), command palette, help
overlay, empty-state, and the footer/badge polish that closes G-01/G-02/G-03/G-04/F-4.

**Out of scope (deferred):** adopting/importing unmanaged Host blocks or keys into
gitid management; a fully-flagged non-interactive CLI create; mouse parity beyond
optional support; Linux validation (Phase 6).
</domain>

<decisions>
## Implementation Decisions

### Layout & Overlay Architecture
- **D-01:** **Two-pane, Tab-cycle focus.** Sidebar + detail are always rendered
  side-by-side; the sidebar never disappears (SC-1). `Tab`/`Shift+Tab` (and `←/→`)
  move focus between panes; `↑/↓` navigate within the focused pane.
- **D-02:** **Modal overlays over a dimmed layout.** The create/add wizard and the
  delete/rotate confirms render as centered modal boxes on top of the (dimmed)
  persistent layout, which stays visible behind. Not a full-screen screen-stack.
- **D-03:** **Responsive: collapse to a single pane** below a width threshold (hide
  the sidebar, show only the focused pane, with a key to toggle the sidebar back).
  Define and document a minimum width. SSH terminals vary in size.
- **D-04:** **Keyboard-first; mouse optional.** Every action has a documented key
  binding (keyboard is the complete path). Mouse (click sidebar row, click view tabs)
  is a bonus where the terminal supports it, never required.

### Editing (per-site + global options)
- **D-05:** **Inline editing in the detail pane.** `e` makes detail-pane fields
  directly editable in place (field highlights, type to change, Enter commits, Esc
  cancels) — literal "in-pane" editing (SC-4). Applies to per-site options (signing,
  port, hostname) and the global baseline options.
- **D-06:** **Match-strategy control redesigned for the TUI with a live preview.**
  Keep the 5.5 semantics (gitdir / repo-URL `hasconfig` / both = OR-applied, with the
  name-derived URL default), but render it as a richer control that shows a **live
  preview of the resulting `includeIf` block(s)** as the user edits.
- **D-07:** **Structural edits trigger the prove-before-write loop.** Committing a
  structural change (alias, hostname, port, match strategy) opens the same
  prove-before-write test loop (as a modal): test → PASS → confirm → backup+write.
  Non-structural edits (e.g. git email, signing toggle) write after a confirm.
  Backup + preview + confirm applies to **every** write (SAFE-01/02/03, IDENT-04).

### Health Integration
- **D-08:** **Per-identity badges + a global Health view, fed by one doctor run.**
  Sidebar rows + the detail pane show badges for findings scoped to that identity
  (its perms, coherence, signing, overlap). The dedicated **Health** view shows the
  full doctor run — all 7 families, including global ones (deps, baseline, agent,
  binary/PATH).
- **D-09:** **In-app auto-fix with modal confirm.** Each fixable finding in the Health
  view offers a fix action; selecting it shows a modal confirm (per-finding, or
  batch-confirm for permissions, mirroring the CLI's `--yes`/per-finding semantics),
  runs the fix through the core, re-evaluates, and updates badges live. No CLI handoff.
- **D-10:** **Distinct glyph AND color per severity** (critical/error/warning/info/
  pass). Fixes the Eval #2 complaint that overlap warnings render with the same `✗`
  as hard errors. Glyph-and-color (not color alone) so it survives monochrome/limited
  SSH terminals; rely on Lipgloss v2 color downsampling for portability. Exact symbols
  finalized during UI design.
- **D-11:** **Preserve async per-family streaming.** The Health view streams each of
  the 8 doctor families independently with spinners (runID stale-guard), as today's
  dashboard does — responsive on slow checks (ssh-agent, network). (The families are
  Deps, Perms, Coherence, Orphans, Signing, Agent, Baseline, and Overlap — `FamilyOverlap`
  was added in Phase 5.5; earlier drafts said "7" before that addition.)

### Visualization Scope
- **D-12:** **Sidebar = managed identities + a read-only "unmanaged" section.** Managed
  identities render as the master list (name · provider · alias/site count — closes
  F-4). A separate, greyed, **read-only** section lists hand-written `Host` blocks AND
  unlinked `~/.ssh` keys (best-effort), reusing doctor's existing orphan/unused-key
  detection from Phase 4.
- **D-13:** **Unmanaged entries get light affordances only:** reveal full path, copy a
  public key, open file location. **No** edit/health/management for them. "Adopt into
  gitid" is explicitly **deferred** to a future phase (see Deferred Ideas).

### Command Palette & Build Strategy
- **D-14:** **Command palette = views + actions.** A `Ctrl+P` fuzzy-filterable command
  launcher lists view switches (Identities/Health/Global Options) AND actions (create
  identity, add account, delete, rotate, copy pubkey, run doctor, fix) — the single
  discoverable entry point. View switching is also bound to `1..N` keys (locked design).
- **D-15:** **Clean rewrite of the `tui/` package.** Build the new two-pane + overlay
  shell fresh for a coherent v2 layout rather than evolving the screen-stack
  `model.go`. **Port** the proven, tested logic rather than rewriting it from scratch:
  the prove-before-write test loop (`tui/prove.go`), the async per-family doctor
  streaming (`tui/dashboard.go`/`tui/deps.go`), form field/validation logic
  (`tui/createform.go`/`updateform.go`/`addaccountform.go`), and copy
  (`tui/copy.go`). The screen-stack architecture itself is what gets replaced.

### Claude's Discretion
- Exact severity glyphs/symbols and color tokens (D-10) — finalize during UI design.
- The width breakpoint and sidebar-toggle key for collapse-to-single-pane (D-03).
- Per-file judgment of which existing `tui/` logic survives the clean rewrite (D-15).
- Field-by-field classification of "structural" (force re-test) vs. cosmetic edits
  (D-07) — follow the existing core's classification.
- The precise unmanaged-detection breadth (which Host blocks / keys are surfaced) —
  reuse doctor's orphan/unused-key logic; best-effort.

### Verification (anti-blindspot — RECURRING)
- **D-16:** Verification MUST drive the **real program entry** (`gitid` no-args → TUI),
  not injected seams or model internals. Phase 4 (doctor) and Phase 5 (TUI) both had
  green tests over injected seams that hid unwired live paths. Exercise the actual
  app: launch, navigate, run the wizard, edit, fix — observe real behavior.
</decisions>

<specifics>
## Specific Ideas

- "Ergonomics modeled on `../tools-installer`" — one app shell, a base layout + shallow
  modal stack, a `Ctrl+P` command palette with `1..N` view keys, rich widgets. The user
  considers the current TUI "pobre" because of *scope*, not the framework.
- The 5.5 match-strategy picker is "good UX" — keep its three labelled options and the
  "both = OR-applied by git" wording; the TUI adds a live `includeIf` preview (D-06).
- Copy is always **public key only** — the private key is never copied; at most reveal
  its path (locked 2026-06-13).
- For the unmanaged section the user explicitly wants "unlinked ssh keys also if
  possible" surfaced alongside hand-written Host blocks (D-12).
</specifics>

<deferred>
## Deferred Ideas

- **Adopt/import unmanaged entries into gitid management** — surfacing hand-written
  Host blocks and unlinked keys is in scope (read-only, D-12/D-13), but an "adopt"
  action that brings them under management (detection + import flow + safety) is a new
  capability for its own future phase.
- **Fully-flagged non-interactive CLI create** (`--git-name/--git-email/--alias/
  --hostname/--port` + `--skip-test`/`--yes`) — Eval learning #6. A CLI-engine
  improvement for scripting/CI, not part of the TUI rebuild; note for a CLI phase.
- **Full mouse parity** (drag scrollbars, click-everything) — this phase is
  keyboard-first with optional mouse (D-04); full parity is a later enhancement.
- **Completion auto-install** (`make install-completions`) — polish logged in the
  playbook; not TUI scope.
- **Suppress spurious `.bak` of freshly-generated keys** (Eval F1.5) — a core/create
  refinement, not TUI; carry to a core polish pass.

</deferred>

## Amendments — Phase 5.6 UX delta (2026-06-16)

Following the failed D-16 manual smoke and the ui-ux-designer review
(05.6-UX-REVIEW.md), two locked decisions were reopened with user sign-off:

- **D-01 (amended): the sidebar is contextual per view.** It remains always
  present (18 cols, Tab-cycle focus — D-01's real intent), but its content now
  follows the active view: identities in Identities, the 8 doctor families in
  Health, config sections in Global Options. The original "identity list in all
  views" reading produced a false affordance and was dropped. (WP-5)
- **D-04 (extended): ←/→ also switch views.** Additive third path alongside the
  locked 1/2/3 + Ctrl+P; Tab still owns pane focus. (WP-6)

Deferred (honest scope): the full D-06 three-option match-strategy selector
(hasconfig/both) needs backend match-building support; the wizard currently
presents the wired gitdir strategy readably with a live includeIf preview, and
the Health/Global rails are read-only status/anchor indexes (jump/filter into
the main pane is a follow-up).
