# Phase 05.6 Plan 04: Detail Pane + Inline Edit + Prove Modal + Global Options Summary

Vertical slice 3 — the master-detail + in-pane editing capability. Fills the detail
pane for the selected identity, adds inline editing ('e') with match-strategy live
includeIf preview, routes structural edits through the prove-before-write loop and
non-structural edits through a simple confirm, and brings the Global Options view
to life reading the baseline from disk with the same inline-edit contract.

## What Was Built

### Tasks 1+2: All sub-models + model wiring (one logical commit, a02b85e)

**tui/detail.go (new):**

- `identityDetailModel`: renders the selected `identity.Account` per UI-SPEC View 1:
  Name/Git Name/Git Email/Provider/Port (StyleLabel 16-col + StyleBody values),
  Aliases/Sites block (bullet + alias + match condition in StyleFaint), Signing state,
  Key Path (StyleFaint). Empty state when `account == nil`.
- `inlineEditMode` ('e' key): builds `[]detailField{textinput.Model}` for editable
  fields. Tab/Shift+Tab cycle focused field; active field gets StyleInputActive border;
  `[editing]` indicator in view.
- Enter routing (D-07):
  - Structural field (alias/hostname/port/match-strategy): sets `proveModalPending = true`
  - Non-structural field (git email/provider): sets `editConfirmPending = true`
  - Root model reads these flags in `handleKey` (inline-edit intercept block)
- Esc: safe default — clears `inlineEditMode`, discards all edits, dispatches nothing.
- `renderMatchPreview()`: when match-strategy field is focused, renders live
  `[includeIf "gitdir:..."]` preview via `gitconfig.RenderIncludeIf` with PaddingLeft(4).
- `buildDetailEditFields`, `isStructuralField`, `editedAccountFromFields`: field
  construction, classification, and extraction helpers.

**tui/wizard.go (new):**

- `wizardProveModel`: shared prove-before-write modal (SHARED SEAM for Plans 05/06).
- Phase 1→2 state machine: `wizardProvePhase1Running` → `wizardProvePhase1Done` →
  `wizardProvePhase2Running` → `wizardProvePhase2Done` → `confirmActive = true`.
- Write gate (D-07, T-05.6-10, T-write-gate): `confirmActive` is the gate; fires
  only after phase1+phase2 PASS and Enter. Verified by `TestProveModalWriteGate`.
- Retry/skip/quit loop: 'r' re-runs phase1 with incremented runID; 's' sets
  `skipConfirmPending` (second Enter required, T-05.6-12 explicit double-confirm);
  'q' emits clearModalCmd keeping key without writing.
- Spinner Tick seeded in `init()` (Pattern C — mandatory).
- `runPreWriteCmd`, `runResolvedCmd`, `runProveWriteCmd`: async goroutine-wrapped cmds
  calling `deps.identity.PreWrite`, `deps.identity.Resolved`, `identity.Update`.

**tui/globalopts.go (new):**

- `globalOptionsModel`: refresh() → `gitconfig.ReadBaselineState` → `baselineLoadedMsg`.
- view(): renders Core (ignorecase/excludesfile), Push/Pull/Fetch, Color, Global
  Gitignore (pattern preview), URL Rewrites. Empty state when `Installed = false`.
- Inline edit ('e'): `editing` flag toggle + `[editing]` indicator; Esc exits.
- Structural edit (excludesfile path): deferred to Plan 05 (no write path in this plan).

**tui/model.go (modified):**

- `modalKind` enum: added `proveModal`, `editConfirmModal`.
- `rootModel` struct: added `detail identityDetailModel`, `globalopts globalOptionsModel`,
  `globaloptsReady bool`, `proveWizard wizardProveModel`.
- `newRootModel()`: initializes `detail` and `globalopts` sub-models.
- `Update()`: routes `preWriteResultMsg`/`resolvedResultMsg`/`writeResultMsg` to
  `proveWizard`; `writeResultMsg.err == nil` → dismiss `proveModal`, clear
  `detail.inlineEditMode`, refresh sidebar. `baselineLoadedMsg` → `globalopts`.
- `handleKey()`:
  - Inline-edit intercept block: when `detail.inlineEditMode && identitiesView`, Enter/Esc
    route to `detail.handleKey`; proveModalPending/editConfirmPending checked post-call.
  - Tab routing (Pitfall 9): when `detail.inlineEditMode`, Tab cycles fields (not pane).
  - '3': initializes `globalopts.refresh()` on first visit (parallels healthReady pattern).
  - 'r': refreshes globalopts in globalOptionsView.
  - 'e': routes to `detail.handleKey("e")` in identitiesView, to `globalopts.handleKey("e")` in globalOptionsView.
  - `proveModal`/`editConfirmModal` modal dispatch.
  - `renderEditConfirmModal()`: simple "Write Changes? [Enter/Esc]".
- `renderMainPane`: `identitiesView` → `m.renderIdentitiesMainPane(w)` (method that syncs `detail.account`), `globalOptionsView` → `m.globalopts.view(w, h)`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Tasks 1+2 collapsed into one logical commit per CLAUDE.md**
- **Found during:** Implementation
- **Issue:** `model.go` references `identityDetailModel` (detail.go), `wizardProveModel` (wizard.go), and `globalOptionsModel` (globalopts.go) before they exist. Without all three, model.go cannot compile and no commit can pass the pre-commit hooks.
- **Fix:** Implemented all 6 files together as one logical commit (a02b85e). CLAUDE.md: "One coherent change — its implementation, its tests, and its docs — is a single commit. Do not split the test/feat/docs of the same logical change into separate commits."
- **Commit:** a02b85e

**2. [Rule 1 - Bug] identityDetailModel.update() removed — unused linter rejection**
- **Found during:** `make lint` (golangci-lint unused linter)
- **Issue:** `update(msg tea.Msg) (identityDetailModel, tea.Cmd)` wrapper was declared but never called. `model.go` calls `handleKey()` directly. The unused method would fail the hard-reject lint hook.
- **Fix:** Removed the update() wrapper. model.go calls `detail.handleKey(key)` directly in the inline-edit intercept block.
- **Commit:** a02b85e

**3. [Rule 2 - Missing Critical] Tab routing intercept block in handleKey**
- **Found during:** Design analysis (Pitfall 9 from RESEARCH.md)
- **Issue:** When `detail.inlineEditMode` is true, the global Tab case (pane cycle) would override the field-cycling intent. Plan requires Tab routes to field cycling in edit mode.
- **Fix:** Added inline-edit intercept block before the global key switch: when `detail.inlineEditMode && identitiesView`, Tab/Enter/Esc route to `detail.handleKey` first. proveModalPending/editConfirmPending checked after.
- **Commit:** a02b85e

**4. [Design - Match Strategy field] buildMatches/matchKinds are in cmd/gitid (package main)**
- **Found during:** Implementation of renderMatchPreview
- **Issue:** `buildMatches`, `matchKinds`, `strategyNumFromKind` live in `package main` (cmd/gitid/match.go) and cannot be imported by the tui package. The plan says "reuse the core" match helpers.
- **Resolution:** `renderMatchPreview` uses `gitconfig.RenderIncludeIf` directly with the existing matches from the account, not the cmd-layer helpers. The Match Strategy field stores the gitdir value (the most common case). Full 3-option strategy picker is planned for Plan 05's create wizard where it would be exposed as a separate form step. This is acceptable for Plan 04's scope (detail pane edit).
- **Commit:** a02b85e

## Threat Surface Scan

No new network endpoints or auth paths introduced. All mutation paths:

- **Structural inline edit → proveWizard**: calls `identity.Update` via `runProveWriteCmd` which calls `deps.update.WriteSSH`/`WriteGitconfig`/`WriteFragment`/`WriteAllowedSigners` — all wired through `filewriter` (backup+atomic+chmod) in `buildTUIUpdateDeps`. Write gate enforced: `confirmActive` only set after phase1+phase2 PASS (`TestProveModalWriteGate` verifies gate holds). Skip requires explicit double-confirm (`TestProveModalRetryPath` verifies skipConfirmPending).
- **Non-structural inline edit → editConfirmModal**: calls `runProveWriteCmd` on Enter after modal; Esc is safe default; `TestEditEscCancels` verifies no write on Esc.

Threats T-05.6-10 (structural write without proving) and T-05.6-11 (any edit without backup/confirm) are mitigated: `TestProveModalWriteGate` confirms the gate; `TestProveModalRetryPath` confirms quit/retry/skip behavior. T-05.6-12 (skip writes silently) is mitigated: skip sets `skipConfirmPending`, second Enter required.

No new threat flags.
