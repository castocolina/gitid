<objective>
Fix the create-new pre-write connectivity gate so it dials the real provider
endpoint instead of the not-yet-written SSH alias. A live end-to-end run found
three bugs that abort the create path against ANY real provider:

- BUG-1 (Critical): `runPipeline` passes `in.Alias` (e.g. "ramon.github") to
  `PreWrite`. The gate runs BEFORE the SSH config is written, so the alias is
  unresolvable → `ssh: Could not resolve hostname` → classified Failure → abort.
  It must pass `in.Hostname` + `in.Port`.
- BUG-2 (Critical): `preWriteArgs` never passes `-p <port>`. gitid's own defaults
  are ssh.github.com:443 / altssh.gitlab.com:443, so `-p 443` is required.
- BUG-3 (Important): `preWriteArgs` omits `-o StrictHostKeyChecking=accept-new`;
  first contact with a never-seen host → `Host key verification failed` →
  misclassified Failure → abort.

Verified-good target invocation (returns "Permission denied (publickey)" →
ReachableNotUploaded → proceeds):
  ssh -i <key> -o IdentitiesOnly=yes -o BatchMode=yes -o ConnectTimeout=10 \
      -o StrictHostKeyChecking=accept-new -p <port> -T git@<hostname>

Purpose: make `gitid identity add` (create-new, reuse, rotate, add-account) able
to complete its pre-write gate against real GitHub/GitLab endpoints.
Output: new 3-arg PreWrite signature threading hostname+port, correct arg slice,
all callers and fakes updated, full suite green under `make test -race`.
</objective>
