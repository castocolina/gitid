<objective>
Close security gap T-02-23 (Tampering/Elevation — command/path injection via a
user-entered identity name). The mitigation already exists for `gitid identity
rotate` (cmd/gitid/rotate.go: `sanitizeName` + `identityNameRe.MatchString`), but
the identical `name` field is UNGUARDED on the two interactive-input paths in
cmd/gitid/add.go:
- `gatherCreateInput` (add.go:265): only `if name == ""`, then `name` flows into
  `filepath.Join(home, ".gitconfig.d", name)` and into key paths — a name like
  `../../tmp/evil` filepath.Join-cleans OUTSIDE `~/.gitconfig.d/`, escaping
  gitid's managed-block + backup guarantees.
- `gatherAddAccount` (add.go:216): same exposure, same `filepath.Join`.

Apply the EXISTING validator (no new regex literal) to both entry points.
rotate.go and add.go are both `package main`, so `identityNameRe` and
`sanitizeName` are already in scope from add.go — call them; do not move or
duplicate them, and do not touch rotate.go.

Purpose: a user-entered identity name can never path-escape `~/.gitconfig.d/` or
inject shell/newline metacharacters into downstream exec/filepath operations.
Output: charset validation guarding both add.go input gatherers, proven by
table-driven tests.
</objective>
