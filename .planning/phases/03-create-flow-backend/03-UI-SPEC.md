# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3 (Create Flow Backend). Generated by
> gsd-ui-researcher, verified by gsd-ui-checker.

**This is a derivation, not a design exploration.** The design was approved
2026-07-06 by Pepe (DLV-08, `.planning/design/APPROVAL.md`) and is FROZEN. This
document consolidates that binding contract for Phase 3's scope (create-flow
backend wiring, real `cmd/gitid` binary) and specifies, in full, the **only**
three new visual surfaces Phase 3 is allowed to introduce — the scoped
divergences D-02, D-16, D-19 from `.planning/phases/03-create-flow-backend/03-CONTEXT.md`.
Every other value below is cited from an approved upstream artifact, not
re-decided. Where this project's stack is a Go Bubble Tea v2 terminal app
(not a web app), template sections written for CSS/web design systems are
translated to their TUI equivalent (theme roles instead of hex/CSS tokens,
100×30 fixed capture geometry instead of responsive breakpoints).

## Design System

This project has **no shadcn / npm component registry** — it is a Go Bubble
Tea v2 TUI. The shadcn initialization gate does not apply; the equivalent
"design system" is the central Go `Theme` (`internal/dummytui/theme.go`),
mirrored 1:1 by role name with the web reference skin's `theme.ts`
(`.planning/design/mockup-src/src/theme.ts`) per `02-STYLE-SPEC.md` §1. Both
already exist and are BINDING — Phase 3 consumes them, it does not create a
new one.

| Property | Value |
|----------|-------|
| Tool | none (no shadcn — Go TUI, not React/Next/Vite) |
| Preset | not applicable |
| Component library | central Go `Theme` struct (`internal/dummytui/theme.go`) — 12 semantic roles, ANSI-16 palette; mirrored role-by-role in the web reference `theme.ts` (design-reference only, never shipped) |
| Icon library | none (no icon font/SVG set) — frozen glyph constants only: `glyphCheckOn/Off` (`☑`/`☐`), `glyphRadioOn/Off` (`●`/`○`), plus status glyphs `✓` healthy, `!` warning, `✗` error, `~` info (`internal/dummytui/theme.go`, `data.go`) |
| Font | terminal's own monospace (TUI has no font choice); the web reference skin pins JetBrains Mono / IBM Plex Mono (`02-UX-DIRECTION.md` §0, Risk 1 mitigation) |
| Capture geometry | fixed **100 cols × 30 rows** (`frameBodyRows(30) = 25` body rows) — this is the TUI's substitute for responsive breakpoints; kept identical for the D-24 visual-regression gate to stay apples-to-apples (`02-STYLE-SPEC.md` §7, `02-DESIGN-DECISIONS-CHECKPOINT-2.md` geometry anchors) |

## Spacing / Layout Scale (TUI-adapted: rows and columns, not px)

There is no CSS px spacing scale in a terminal grid. The equivalent contract
is the frozen row/column budget below — every value is cited from the
checkpoint-2 contract, not invented.

| Token | Value | Usage | Source |
|-------|-------|-------|--------|
| Frame | 100 cols × 30 rows, fixed | capture + PTY e2e geometry | `02-STYLE-SPEC.md` §7 |
| Body rows | 25 | rows available below header/breadcrumb chrome | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` geometry anchor |
| Tightest pane (measured) | ~24 of 25 body rows (wizard step 2 / Git identity step) | 1 row headroom — `make test-e2e`'s 100×30 PTY walk is the standing proof | `02-STYLE-SPEC.md` §7 |
| Detail/form pane width | 62 cols @ 100-col minimum | form-left / preview-right guided-form archetype | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` geometry anchor |
| Field row height | exactly 1 row, EVERY state (focused or blurred) | D1 — `renderFocusedFieldBox` deleted; fields never reflow | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` D1 |
| Marker gutter | 2 cells | `▸ ` (accent) when focused, `  ` (blank) otherwise | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` D1 |
| Label column | `padRight(label, 16)` | bold in all states | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` D1 |
| Radio group | 1 header row + 3 option rows, ALWAYS rendered (no collapse) | algorithm catalog, match-strategy | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` D2 |
| Git-step action row | 1 row (`Back (Esc)  [ Skip Git ]  [ Continue ]`), hints below | ≈43 cols, fits the 62-col pane | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` D6 |
| Chord-hint line | 1 row, directly under the stepper, ALWAYS visible | step-conditional Shift+←/→ hint | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` D5/D7 |
| Contextual footer | 0 extra body rows (chrome line) | e.g. `Tab/↑↓ fields` + `Enter rewrite Host block` | `02-DESIGN-DECISIONS-CHECKPOINT-2.md` affordance audit |

**Exceptions for this phase:** none beyond the checkpoint-2 contract already
in force, PLUS the three scoped divergences below (D-02/D-16/D-19), each of
which is budgeted to reuse an EXISTING row slot (test-outcome line,
top-of-body banner line, disabled-reason line) rather than adding new rows —
see the "Scoped Divergences" section for the exact placement of each.

## Typography (TUI-adapted: semantic roles, not px sizes / weights)

A terminal has one font at one size — "3-4 sizes / 2 weights" translates to
the role table's bold/faint/color distinctions, all already defined and
BINDING (`02-STYLE-SPEC.md` §1). Phase 3 introduces no new role.

| Role | TUI treatment | Web mirror (reference only) | Usage |
|------|----------------|------------------------------|-------|
| Label | `Bold(true)` | `roles.label`, `fontWeight: 700` | field labels, always bold in every state |
| Field (value) | plain, no styling | `roles.field`, visible border | the value itself |
| Field — focused | `Foreground(ANSI 4) + Bold`, NO border (D1) | `roles.focusedField` — accent tint, no layout change | value + `▸` marker + bracket delimiters |
| Field — blurred | `Faint(true)` | `roles.blurredField`, opacity 0.85 | dim brackets/value, unchanged single row |
| Hint | `Faint(true)` | `roles.hint` | helper/explanation copy, never collapses |
| Warning | `Foreground(ANSI 3)` — yellow | `roles.warning` | advisory / needs-action states, incl. D-02/D-16 below |
| Error | `Foreground(ANSI 1)` — red | `roles.error` | test-fail, hard failures |
| Healthy | `Foreground(ANSI 2)` — green | `roles.healthy` | `result-success` |
| Info | `Foreground(ANSI 6)` — cyan | `roles.info` | least used in Phase 3's own screens |
| Preview | `Faint(true)` + dashed border | `roles.preview` | bounded config-text preview panes |

**Non-negotiable, carried from `02-UX-DIRECTION.md` §2:** every colored state
pairs with a glyph AND a word — never color alone. This applies to D-02's new
warning state exactly as it applies to every pre-existing one.

## Color

The TUI stays ANSI-16 (no truecolor/adaptive light-dark) by binding project
decision (`02-STYLE-SPEC.md` §1) — a 60/30/10 hex breakdown does not apply.
The equivalent restricted-and-semantic contract, mapped onto the actual role
table:

| Role | Value | Usage |
|------|-------|-------|
| Dominant surface | terminal default fg/bg (unthemed) | body text, plain field values — the terminal's own background/foreground, deliberately NOT a `Theme` role (`02-STYLE-SPEC.md` §1 scope-note: "pure layout grays... carry no semantic meaning") |
| Secondary | `Faint`/`Hint`/`DisabledNav` (dim) | hint copy, blurred fields, inactive-while-capturing nav tabs |
| Accent (ANSI 4, blue) | `FieldFocused`, `ActiveArea`, `ActiveNav` (background), `ActiveNavDimmed` (foreground-only), wizard stepper active segment/dots | reserved for these SPECIFIC elements — never decorative (anti-slop stance, `02-UX-DIRECTION.md` §1: "Rainbow color... every field a different hue" is explicitly what to avoid) |
| Warning (ANSI 3, yellow) | `Theme.Warning` | needs-action/advisory states (existing taxonomy states `incomplete`/`git-only`/`key-unused`, GSSH/GGIT advisories) — **and the NEW D-02 `ReachableNotUploaded` state** |
| Error/Destructive (ANSI 1, red) | `Theme.Error` | `test-fail`, hard `Failure`; Phase 3 has no delete/destructive-confirm screen (that is Phase 5's Identity Manager) |
| Healthy (ANSI 2, green) | `Theme.Healthy` | `result-success` |
| Info (ANSI 6, cyan) | `Theme.Info` | rarely used directly in Phase 3's own screens |

Accent reserved for (explicit list, never "all interactive elements"):
focused-field marker + value color, the active header nav tab's background
(no pane capturing keys) or foreground-only (D4 `ActiveNavDimmed`, a pane
IS capturing keys), the breadcrumb/divider line directly above an active
pane (`ActiveArea`), and the wizard stepper's active segment + dots-so-far.

## Scoped Divergences (D-02 / D-16 / D-19)

These are the **only** new visual surfaces Phase 3 may introduce. Each reuses
an EXISTING theme role and an EXISTING row slot from the approved contract —
none adds a new color, a new glyph, or a new screen. Each is a documented,
allowlisted departure from the approved dummy goldens under D-24's
visual-regression gate, exactly like D9's precedent in Phase 2.

### D-02 — `ReachableNotUploaded` warning state (TEST-01/02/03 gate semantics)

- **What it is:** a THIRD test-outcome state, alongside the approved demo's
  binary PASS/Failure — the connectivity test succeeded (the host is
  reachable and, for stage 2, `ssh -G` proves the correct `IdentityFile`
  resolves) but the key's `.pub` has not yet been registered with the
  provider (upload is Phase 9's job), so authentication cannot fully succeed
  yet.
- **Theme role (EXISTING, no new role):** `Theme.Warning` / `roles.warning`
  — the SAME yellow already used for the `incomplete`/`git-only`/
  `key-unused` taxonomy states and the GSSH/GGIT advisories.
- **Glyph (EXISTING, no new glyph):** `!` — the SAME glyph already used
  everywhere else a warning-tier state is shown (`data.go`
  `SeverityWarning = "!"`, taxonomy glyph map). Paired with the word, never
  color alone, per the standing glyph contract.
- **Placement:** the existing test-outcome line in `test-stage1-direct` /
  `test-stage2-by-alias` (`FIELDS.md` `test_output` row) — this state
  REPLACES that row's PASS/Failure rendering when applicable; it is not a
  new row or a new screen.
- **Draft copy** (freezes via the §6 grep mechanism once approved):
  `! Reachable — key not uploaded yet`
- **Follow-on affordance (D-03, same divergence):** directly below the
  outcome line, ONE hint line (existing `Theme.Hint`/`hint` role) naming the
  provider's key-settings page, plus a copy-`.pub`-to-clipboard action
  (`internal/clipboard`). This reuses the `hint-persistence` pattern (a
  reserved line that never collapses) — no new visual mechanism.
- **Store-gate consequence:** the confirm-write ceremony unlocks on PASS OR
  `ReachableNotUploaded` (D-01) — only a hard `Failure` blocks. An identity
  stored at `ReachableNotUploaded` carries the `key-unused` taxonomy state
  (MGR-02), not `complete`; the confirm-write / result-success copy must
  say so plainly (no silent "success" framing over an unauthenticated key).
- **D-24 allowlist entry:** `test-stage1-direct` and `test-stage2-by-alias`
  golden text gain a third possible outcome line vs. the approved dummy
  goldens (PASS/Failure only) — register both screen IDs in the D-24.1
  automated-gate allowlist, and flag both for D-24.2 reviewer critique
  (`agent-ui-ux-designer` + Codex).

### D-16 — "Preview — demo data" banner (not-yet-wired views)

- **What it is:** Phase 3 makes the real `cmd/gitid` binary the live app
  shell (D-15), with create-flow fully wired. Every OTHER view the shell
  can reach (git-screen, identity-manager, global-ssh, global-git, health,
  fixer) still renders its approved DEMO content verbatim (its own backend
  arrives in a later phase) — those views need a persistent, honest
  "this is not real yet" signal so a user navigating the real binary is
  never misled into thinking a demo screen reflects their actual system.
- **Theme role (EXISTING, no new role):** `Theme.Warning` / `roles.warning`
  — deliberately the SAME role as D-02 (not the passive `Hint` role),
  because this is an advisory the user must notice, not ambient help copy.
- **Glyph (EXISTING):** `!` + the word, same contract as every other
  warning-tier state.
- **Placement:** persistent, ONE row, top of the view's body region,
  directly under the header/breadcrumb (`ActiveArea`) line. Chrome-cost,
  not body-content cost — measured against the row budget the same way the
  existing breadcrumb/status lines already are (`02-STYLE-SPEC.md` §7).
- **Draft copy** (freezes via the §6 grep mechanism once approved):
  `! Preview — demo data, not wired to your system yet`
- **Removal contract:** each later phase deletes this banner from ONLY the
  screens it wires (Phase 4 removes it from git-screen, Phase 5 from
  identity-manager, etc.) — Phase 3 adds it to every view it does NOT wire,
  and removes nothing (create-flow itself never carries this banner, since
  Phase 3 wires it).
- **D-24 allowlist entry:** every non-create-flow view/screen ID gains this
  new top-of-body line vs. the approved dummy goldens (which carried no such
  banner — the entire dummy WAS demo data, with nothing else to contrast
  against). Register as ONE blanket allowlist entry scoped "any non-wired
  view, top-of-body row" rather than enumerating every affected screen ID
  individually, given the number of views involved; still subject to
  reviewer critique (D-24.2) to confirm it reads as intended across all of
  them.

### D-19 — Git-form `[ Continue ]` disabled reason (Phase-3-scoped, REAL binary only)

- **What it is:** the Git-identity wizard step (step 2 of 4) stays demo'd
  in Phase 3 (D-18 — Git backend wiring is Phase 4's job). In the REAL
  binary, `[ Continue ]` must stay disabled regardless of what the demo
  form's own client-side validity check would otherwise compute, because
  there is no real Git backend behind it yet — the existing validation-
  reason string (`— needs user.name + a valid email`) would be a LIE about
  capability if shown in the real binary (it implies "fill the form
  correctly and it will work").
- **Visual treatment (EXISTING, no new role):** the disabled-`[ Continue ]`
  button styling already established by the checkpoint-2 contract (D7) is
  reused unchanged — only the REASON STRING differs between the two
  binaries.
- **Placement (EXISTING slot):** directly under the disabled `[ Continue ]`
  button — the identical row the frozen D7 reason string occupies
  (`FIELDS.md` `wizard_continue_disabled_reason` row). No new row.
- **Draft copy** (freezes via the §6 grep mechanism once approved), REAL
  binary (`cmd/gitid`) only:
  `— Git configuration arrives with the next build`
- **Copy-freeze handling — two frozen strings, two binaries, same slot:**
  this is a NEW frozen string joining the §6 grep mechanism ALONGSIDE the
  existing one, not a replacement of it. The gate must assert BOTH:
  (a) `— Git configuration arrives with the next build` is present in
  `cmd/gitid`'s real wizard render path, AND (b) `internal/dummytui` (the
  demo, unaffected by Phase 3) still carries the UNCHANGED
  `— needs user.name + a valid email` string. The executor must not
  collapse these into one string or delete the demo's original — they are
  deliberately different, in deliberately different binaries, for the same
  visual slot.
- **D-24 allowlist entry:** the Git-identity step-2 screen's golden text,
  captured from the REAL binary, differs from the approved dummy's golden
  at exactly this one line — register as an allowlisted diff scoped to
  "create-flow / git-form step, REAL-binary capture only" (the dummy's own
  capture is unaffected since it is not diffed against itself).

## Inherited Non-Negotiables (quick reference for the executor)

These are restated here for convenience only — they are NOT decided by this
document; each is cited from its binding source and must not be re-litigated
during Phase 3 planning or execution.

- **Arrow-key precedence rule** (expanded-select > text-input-cursor >
  wizard-step-nav validity-gated forward/always-allowed back > Shift+←/→
  focus-override), implemented identically in Go and (reference) TypeScript
  — `02-STYLE-SPEC.md` §2.
- **Glyph contract:** every colored state pairs with a glyph AND a word,
  never color alone — `02-UX-DIRECTION.md` §2.
- **Four-beat mutation ceremony** (preview → confirm → backup notice →
  result) for every write, no exceptions — `02-UX-DIRECTION.md` §5. Phase
  3's create-flow confirm-write is this ceremony; Cancel is never the
  default focus.
- **Copy-freeze grep gate**, extended per D-02/D-16/D-19's new strings —
  `02-STYLE-SPEC.md` §6.
- **Recipe fidelity:** the live `Host` block preview and the macOS `Host *`
  globals block must match `recipes/ssh-config.recipe`'s structure exactly
  (alias per identity, `Port 443`, `IdentitiesOnly yes`, ed25519 not RSA)
  — `recipes/README.md`, `recipes/ssh-config.recipe`.
- **Recipe-superseding decision (D-06):** the confirm-write ceremony's
  `Targets`/`Backups` in the real binary must reflect the AUTO-DETECTED
  storage layout (in-file managed blocks OR the gitid-owned Include'd
  file, defaulting to Include'd on a fresh machine) — the dummy's fixture
  data (`internal/dummytui/identities.go` `reviewCeremony()`, currently a
  single hardcoded `~/.ssh/config` target) is DEMO-ONLY and must not be
  copied verbatim into the real backend wiring; the on-screen "named
  target file" affordance (`FIELDS.md` `target_file` row) stays, its
  VALUE becomes dynamic.
- **100×30 capture geometry** is kept identical for the D-24 gate — do not
  change frame dimensions to "fit" new copy; fit new copy into the existing
  row budget instead (see the divergences above, each of which reuses an
  existing row).
