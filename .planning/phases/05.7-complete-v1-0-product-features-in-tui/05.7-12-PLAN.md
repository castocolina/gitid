<objective>
Cross-cutting TUI render fix. The expanded match-strategy panel overflows the modal:
modal height is computed POST-render (tui/model.go:1002) and placeOverlay silently
clamps overflowing rows (tui/overlay.go:62-64), so the panel renders incomplete. The
editable gitdir sub-field lives INSIDE that overflowing panel, so the overflow also
hides the folder field. The user reported "La pantalla a veces se muestra incompleta
cuando hago Tab... además no veo un campo para editar el folder."

Two fixes, both required:
1. Make modal rendering viewport-aware so content taller than the terminal renders
   completely (scroll or measured layout) instead of being silently clamped.
2. HARD USER REQUIREMENT: expose clearly editable fields for BOTH the target-host
   alias AND the folder (gitdir) that are ALWAYS visible/reachable — never buried
   inside the expandable panel.

Depends on Plan 10 (shares tui/wizard.go; must run after the wizard legibility + no-
write plans land so this builds on the final form layout). Closes: G-1.

Purpose: The match panel must never render half-drawn, and the two fields the user
must edit to wire LEG 1 (alias) and LEG 2 (gitdir folder) must always be on screen.
Output: A viewport-aware modal and a wizard form with always-visible alias + gitdir
fields.
</objective>

  <name>Task 1: Viewport-aware modal — render tall content completely, never silently clamp (G-1)</name>
  <files>tui/overlay.go, tui/overlay_test.go, tui/model.go, tui/model_test.go</files>
  <behavior>
    - When modal content height > available terminal height, the modal renders completely
      via scrolling (a viewport that shows the focused region and is scrollable) OR a
      measured layout that fits — it must NOT silently drop overflowing rows.
    - placeOverlay (tui/overlay.go:56-69) currently `continue`s on bgRow >= len(bgLines)
      (lines 62-64), silently discarding rows below the background. The fix ensures modal
      content that would overflow is made reachable (scrolled into view) rather than lost.
    - modalOrigin / renderContent (tui/model.go:1001-1004) must compute the modal height
      against the terminal height BEFORE compositing, so y is not negative and the modal is
      not anchored such that its tail falls off-screen unrecoverably.
    - For a terminal smaller than the modal content, the focused field (and the field the
      user is editing) is always within the visible region.
    - No regression to the existing centered-overlay behavior when content fits (the
      existing TestPlaceOverlayCompositesModal contract still holds).
  </behavior>
  <action>
    Make modal rendering viewport-aware so tall content (the expanded match panel ~15-18
    lines plus the rest of the form) renders completely instead of being silently clamped
    (UAT G-1 root cause: tui/model.go:1002 post-render height + tui/overlay.go:62-64 silent
    clamp).

    Choose the smallest correct mechanism (prefer scroll over shrinking content):
    - Option A (preferred) — viewport/scroll: when the rendered modal height exceeds the
      available rows (m.height minus margins), wrap the modal body in a scrollable region
      (charm.land/bubbles/v2 viewport is already a project dependency per CLAUDE.md stack)
      or implement a focused-window slice of the modal lines so the focused field is always
      shown. Track scroll offset in the wizard/modal state; ensure the focused field stays in
      view as focus moves. Add a visible affordance (e.g. "↑↓ scroll" or a "more ↓" marker)
      so the user knows content continues.
    - Option B (fallback) — measured layout: compute the modal height first, clamp the modal
      box to the available rows, and render only the bounded region with an explicit scroll
      indicator; never `continue`-drop rows without an indicator.

    In tui/overlay.go: change the silent clamp at lines 62-64 so overflow is either
    accommodated by the chosen viewport or surfaced (the modal must not anchor content below
    the background with no way to see it). Keep the ANSI-aware overlayLine compositing
    intact. In tui/model.go renderContent (line 1001-1004): measure modalContent height,
    bound it to m.height, and feed the viewport-bounded content to placeOverlay so the modal
    never extends past the screen unrecoverably.

    Tests:
    - overlay_test.go: a modal taller than the background no longer silently loses its
      tail — assert that with a viewport offset (or bounded render) every modal line is
      reachable across scroll positions, and the existing fits-on-screen contract
      (TestPlaceOverlayCompositesModal) still passes.
    - model_test.go: with a small terminal (e.g. 80x24) and the wizard match panel expanded,
      renderContent shows the focused field and a scroll affordance; no panic; resizing taller
      shows the full panel.

    ANTI-BLINDSPOT: test through the real rootModel.renderContent / wizard view at realistic
    small terminal sizes, not the overlay primitive in isolation only.

    English-only for all code, comments, and test names.
  </action>
