<objective>
STAGED WIZARD -- SCREEN 2 (SSH Connectivity Test, LEG 1 write). This is the second
SSH screen: the user uploads the public key to the provider, tests connectivity
against the STAGED key, and on success the wizard PROGRESSIVELY writes LEG 1 (the
keypair to ~/.ssh/ + the Host block to ~/.ssh/config) via the Plan-09
identity.PersistSSH step -- BEFORE the git screen. A mid-exit after Screen 2 leaves a
valid SSH-only identity (doctor flags it incomplete; that is the accepted design).

Four deliverables:
1. Make Screen 2 show the FULL ssh command AND the tested key path CONSISTENTLY -- on
   pre-run, on SUCCESS, and on failure -- not only on failure (G-3). The command comes
   from tester.Result.Command (never hand-built); the pre-run command is built via a
   pure tester helper so it is byte-identical to what PreWrite will run.
2. NO gh/glab auto-upload in the wizard. Screen 2 instructs "upload this public key to
   your provider, then test" and points at the existing standalone copy/upload assist
   (untouched). Auto-upload is a deferred future multi-identity feature.
3. On SUCCESS, call identity.PersistSSH (Plan 09): persist the keypair to ~/.ssh/
   (final) AND write the Host block (timestamped backup, idempotent), then advance.
4. Keep the D-05 offline skip-&-write as a SECONDARY escape with clear no-write
   feedback (nothing written yet + why + how): [s] + confirming Enter writes LEG 1 via
   PersistSSH with the double-confirm + unauthenticated-write warning preserved.

Depends on Plan 10 (the staged scaffolding + Screen 1 + the alt-SSH editable fields)
and on Plan 09's PersistSSH (wave 5). Wave 7, after 10 (wave 6); shares tui/wizard.go
so it must run after 10. LEG 1 (this) lands before LEG 2 (Plan 13, Screen 3).

Closes (command+path half): G-3. Closes (write + no-write-feedback half): G-2.

Purpose: The user lost sight of the command/path once the test passed, and a brand-new
key that fails the test never wrote the alias, leaving a dangling unusable host. The
primary path is now upload-manually -> test -> on-pass persist LEG 1, with the command
and key path always visible and the offline escape clearly explained.
Output: A Screen 2 that always shows the full command + key path, writes LEG 1 on
success via PersistSSH, and offers a clear, discoverable offline skip-&-write escape.
</objective>
