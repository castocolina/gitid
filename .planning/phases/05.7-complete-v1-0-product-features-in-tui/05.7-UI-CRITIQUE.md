## Dimension 1 — Copywriting

**Scope:** All new Phase 5.7 copy elements per UI-SPEC §Copywriting Contract.

**Findings:**

### D1-COPY-01: Match-strategy collapsed field copy is correct [PASS]
- Frame evidence: `Match Strategy: gitdir (folder)` with `includeIf preview:` label
- Implementation: `renderMatchSelector` with `StyleFaint` for description
- Assessment: Matches UI-SPEC §1a collapsed spec exactly

### D1-COPY-02: Adopt affordance guard copy is accurate [PASS]
- Frame evidence: Toast "select a fragment in the Unmanaged section" (truncated in header) — fired when 'A' pressed on managed identity
- UI-SPEC requirement: Adopt opens on `kindFragment` row only
- Assessment: Guard copy is clear, user-actionable

### D1-COPY-03: Add Repo modal title and footer copy [PASS]
- Frame evidence: `Add Repo` title, `tab next  esc cancel` footer
- UI-SPEC spec: title = `Add Repo`; footer hints = `tab next  esc cancel`
- Assessment: Matches spec. Clean single-line footer.

### D1-COPY-04: Match strategy expanded selector — recipe-alignment note not visible in collapsed frames [DEFERRED]
- Frame evidence: The match-strategy selector did not render in expanded/focused state in the captured frames (the PTY test navigated past the field). The collapsed view shows correctly, but expanded copy (radio labels, "Recipe: hasconfig is the primary match..." footnote) is not confirmed in a PTY frame.
- Source: tui/wizard.go `renderMatchSelector` contains the expanded radio labels and recipe note
- Risk level: LOW — the code exists and is unit-tested (TestWizardMatchSelectorNavigation); the PTY test timing issue caused the frame capture to miss the expanded view
- Deferral rationale: The unit tests cover the expanded selector text; the PTY harness can capture the expanded frame in a future test run with corrected Tab count (6 not 7). No code change needed.

### D1-COPY-05: Wizard form placeholder text uses example values [MEDIUM concern]
- Frame evidence: `Git Name: | e.g. Pedro Perez`, `Git Email: | e.g. pedro.perez@example.com`
- UI-SPEC: Not explicitly specified but placeholders with personal names (Pedro Perez) are culturally biased; generic "e.g. Jane Smith" or `Your Name` would be more neutral
- Risk level: LOW — functional, non-blocking; purely cosmetic
- Deferral rationale: Name-related placeholders are a long-standing cosmetic issue predating Phase 5.7; deferred to a dedicated UX pass

## Dimension 2 — Visuals

**Scope:** Modal layouts, sidebar affordance, state machine transitions.

### D2-VIS-01: Create wizard modal renders correctly at 80x24 [PASS]
- Frame evidence: `wizard-name-input.txt` — modal uses `╭──╮` rounded border (styled), 72-col modal width, correctly overlays the two-pane layout
- UI-SPEC: `min(terminalWidth-8, 72)` cols; rounded border via `StyleModal`
- Assessment: PASS — active field has rounded border `╭/╰`, inactive fields use square `┌/└` (Focus indicator is correct per the 05.6 design system)

### D2-VIS-02: Add Repo modal renders correctly [PASS]
- Frame evidence: `addrepo-modal-open.txt` — `╭──╮` border, `Add Repo` title, `Clone URL:` label with `>` prompt
- UI-SPEC: modal box with `StyleModal` + `StyleModalTitle`
- Assessment: PASS

### D2-VIS-03: Sidebar Unmanaged section renders with fragment row [PASS]
- Frame evidence: `sidebar-unmanaged.txt` — `Unmanaged` section header + `○ .gitconfig_d ~` row with `~` glyph
- UI-SPEC §2a: `○` glyph, `~` affordance marker for orphan-key/fragment rows
- Finding: The fragment row shows `○ .gitconfig_d ~` with `~` glyph even for `kindFragment` entries (should show `[Adopt]` when FOCUSED — the test cursor was not on this row)
- Assessment: Fragment row IS shown (PASS for section rendering); the `[Adopt]` affordance when focused was not captured in this frame because the cursor was on the managed identity

### D2-VIS-04: Toast appears in header bar for guard messages [PASS]
- Frame evidence: `adopt-modal.txt` header shows toast: "select a fragment in the Unman..." (truncated at 80-col header width)
- UI-SPEC: Toasts auto-dismiss after 3s; `StylePass` for success, other styles for info
- Finding: Toast IS shown in the header. The truncation is expected at 80 cols with the full header text. The text ends at ~80 chars.
- Assessment: PASS — correct placement and behavior

### D2-VIS-05: Match strategy expanded selector not captured in PTY frames [DEFERRED]
- See D1-COPY-04 above. The expanded selector (radio buttons, gitdir sub-field, hasconfig sub-field, Preview block) was not visible in the captured PTY frames due to test timing (Tab count off by 1).
- Code review of `renderMatchSelector` confirms: the expanded state IS implemented (radio buttons with `[x]`/`[ ]`, sub-fields, Preview label, recipe footnote).
- Deferral rationale: Unit-tested; PTY frame capture requires corrected Tab count.

## Dimension 3 — Color

**Scope:** Correct application of the 10-token palette on new surfaces.

### D3-COLOR-01: Active vs inactive field border differentiation [PASS]
- Frame evidence: `wizard-name-input.txt` — active field has rounded `╭/╰` border (StyleInputActive: blue rounded), inactive fields have square `┌/└` (StyleInputInactive)
- UI-SPEC: `StyleInputActive` (blue rounded border) vs `StyleInputInactive`
- Assessment: PASS — visible differentiation without color (safe for monochrome terminals too)

### D3-COLOR-02: `[Adopt]` accent color on fragment row [NOT CAPTURABLE IN TEXT-ONLY FRAME]
- Frame evidence: PTY frames are text-only (vt.Emulator.String() strips ANSI codes); `[Adopt]` presence can be confirmed by text content but color cannot be verified from text frames
- UI-SPEC: `ColorAccent` (blue) on `[Adopt]` label when fragment row focused
- Assessment: Code review confirms `StyleAccent` (bold blue) is applied in `renderUnmanagedRow` (tui/sidebar.go). Text-only frames cannot verify color; color verification requires real terminal inspection.
- Deferral: Color verification is the thin slice reserved for the optional human SSH-terminal pass (Task 4, NON-BLOCKING per plan).

### D3-COLOR-03: Color never sole carrier of meaning [PASS]
- All modal states use glyphs (`✓`, `✗`, `!`, `~`, `○`, `[x]`, `[ ]`) in addition to color
- Footer text identifiers confirm action type independently of color
