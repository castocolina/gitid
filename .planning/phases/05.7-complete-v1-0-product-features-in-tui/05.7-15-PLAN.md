<objective>
WIRE the provisional-block lifecycle (Plan 14) into the staged create wizard Screen 2 so
the SSH write becomes transactional per CLAUDE.md (`test -> confirm + backup -> re-test
the resolved configuration`), closing UAT G-5 (BLOCKER).

Today the wizard runs Phase 2 `tester.Resolved` (ssh -T git@<alias> + ssh -G <alias>)
BEFORE any Host block is written, so the alias is unresolvable ("could not resolve
hostname") and the wizard can never complete. This plan reorders the flow around a clearly-
marked PROVISIONAL block:

1. Screen 1: generate keypair to STAGING (unchanged).
2. Screen 2, BEFORE the resolved test: write the PROVISIONAL Host block (timestamped
   backup, idempotent) with IdentityFile pointing at the STAGED key, so the alias resolves
   to exactly that key during the test. Effective Host = the typed alias, else the provider
   host (e.g. github.com) — via identity.EffectiveAlias, NOT an invented <name>.<provider>.
3. Test THROUGH the alias (now resolvable): Phase 1 connectivity (explicit -i, real
   hostname — unchanged) then Phase 2 resolution (ssh -T git@<alias> + ssh -G <alias>).
   Show the full command + key path on EVERY phase (pre-run, Phase 1 success, Phase 2).
4. On Phase 2 SUCCESS: install the key to ~/.ssh, PROMOTE the provisional block to managed
   (repoint IdentityFile to the final key), then advance to Screen 3 (Plan 13).
5. On CANCEL / quit / give-up after the provisional write: DROP the provisional block
   (restore from backup) and discard the staged key — never leave a provisional block.
6. BOOTSTRAP: on wizard entry, if a leftover `gitid provisional:` block exists, WARN and
   offer [d] drop & start fresh OR [r] resume that identity.

This plan captures the write-seam ORDER — provisional write BEFORE the resolved test — as
the core G-5 regression guard, driving the REAL model Update path with faithful key events
(MEMORY: do not assert on stubbed helpers; drive the real program path).

Purpose: make the alias actually resolvable at test time without leaving a partially
verified managed identity behind, and make the alias honest (blank -> provider host).
Output: a Screen 2 that writes-provisional -> tests -> promotes/drops, a WYSIWYG alias, a
command+path on every phase, and stale-provisional bootstrap recovery.
</objective>
