<objective>
STAGED WIZARD -- SCREEN 1 (SSH Identity, LEG 1 inputs) + the cross-cutting viewport
fix. This plan restructures the create wizard from a single overflowing form into a
staged multi-screen flow and lands its first screen. By giving Screen 1 only a few
fields and moving the match panel off to a later screen (Plan 13), it STRUCTURALLY
dissolves the overflow gap (G-1) rather than papering over it.

Three deliverables:
1. Introduce the staged-flow scaffolding: a wizardScreen enum (Screen 1 SSH Identity,
   Screen 2 Connectivity Test, Screen 3 Git Config, Screen 4 Review) and route the
   existing form/keygen/upload/prove steps under Screen 1 / Screen 2 so later plans
   slot Screens 3-4 in cleanly. This plan implements Screen 1 fully; Screen 2 stays
   wired to the existing upload+prove machinery (Plan 12 reworks it).
2. SCREEN 1 fields, ALL editable and ALWAYS visible (HARD G-1 requirement): identity
   name, key algorithm, provider, SSH host alias (default <name>.<provider>), Hostname
   (default alt-SSH ssh.<provider> via identity.DefaultHostname from Plan 09), Port
   (default 443 via identity.DefaultPort), folder/gitdir (default ~/git/<name>/). The
   folder and a NEW editable Hostname field are promoted to top-level rows, never
   buried inside an expandable panel.
3. A LIVE preview of the exact `Host <alias>` block that WILL be written, via
   sshconfig.RenderHostBlock, updating as the user edits alias/hostname/port (G-2
   preview half) -- plus the viewport-aware modal fix so tall content always renders.

Depends on Plan 09 (the UI-free identity.DefaultHostname/DefaultPort helper this
screen's fields pre-fill from). Wave 6, after 09 (wave 5). Touches tui/wizard.go,
tui/overlay.go, tui/model.go -- no overlap with the wave-5 doctor plan.

Closes: G-1 (dissolved by staging + always-visible editable fields + viewport).
Closes (preview half): G-2.

Purpose: The user reported the form renders incomplete when tabbing, with no editable
folder field, and that the alias block never appears. Staging keeps each screen small;
promoting alias/hostname/port/folder to always-visible editable rows guarantees they
are reachable; the live Host-block preview makes LEG 1 legible before any write.
Output: A staged wizard whose Screen 1 shows all LEG-1 fields editable and a live,
byte-faithful Host-block preview, inside a viewport-aware modal that never clamps.
</objective>

  <behavior>
    - When modal content height > available terminal rows, the modal renders
      completely: either a scrollable region that shows the focused field, or a
      measured layout bounded to the available rows WITH an explicit scroll
      indicator. It must NOT silently drop overflowing rows.
    - placeOverlay (tui/overlay.go:56-69) currently `continue`s on bgRow >=
      len(bgLines) (lines 62-64), silently discarding rows below the background. The
      fix makes overflowing modal content reachable (scrolled into view / bounded with
      an indicator) rather than lost.
    - modalOrigin / renderContent (tui/model.go:1001-1004) measures the modal height
      against m.height BEFORE compositing, so y is not negative and the modal tail does
      not fall off-screen unrecoverably.
    - For a terminal smaller than the modal content, the focused/edited field is always
      within the visible region.
    - No regression to centered-overlay behavior when content fits
      (TestPlaceOverlayCompositesModal contract still holds).
  </behavior>
  <action>
    Make modal rendering viewport-aware so tall content renders completely instead of
    being silently clamped (UAT G-1 root cause: model.go:1002 post-render height +
    overlay.go:62-64 silent clamp). Even with staging shrinking each screen, a tiny
    terminal can still overflow, so this guard is required.

    Choose the smallest correct mechanism (prefer scroll over shrinking content):
    - Option A (preferred) -- viewport/scroll: when the rendered modal height exceeds
      the available rows (m.height minus margins), wrap the modal body in a scrollable
      region (charm.land/bubbles/v2 viewport is already a project dependency) or slice
      a focused window of the modal lines so the focused field is always shown. Track
      scroll offset; keep the focused field in view as focus moves; add a visible
      affordance (e.g. "more ↓" / "↑↓ scroll").
    - Option B (fallback) -- measured layout: compute the modal height first, clamp the
      box to available rows, render only the bounded region with an explicit scroll
      indicator; never `continue`-drop rows without an indicator.

    In tui/overlay.go: change the silent clamp at lines 62-64 so overflow is
    accommodated/surfaced, keeping the ANSI-aware overlayLine compositing intact. In
    tui/model.go renderContent (1001-1004): measure modalContent height, bound it to
    m.height, feed the bounded content to placeOverlay so the modal never extends past
    the screen unrecoverably.

    Tests:
    - overlay_test.go: a modal taller than the background no longer silently loses its
      tail -- every modal line is reachable across scroll positions (or bounded with an
      indicator); the existing TestPlaceOverlayCompositesModal still passes.
    - model_test.go: with a small terminal (e.g. 80x24) and a tall wizard screen,
      renderContent shows the focused field + a scroll affordance; no panic; resizing
