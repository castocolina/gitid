<decisions>
## Implementation Decisions

### Scope reconciliation
- **D-01:** 5.7 delivers ONLY the four open items above. Do NOT re-implement
  `GLOBAL-01`/`URLRW-01`/`CLI-02`/alt-SSH-443 — they are complete. Any work on them
  is limited to incidental wiring/verification if a gap surfaces.

### Match-strategy selector (completes Phase 5.6 D-06, deferred)
- **D-02:** **Keep `gitdir` as the DEFAULT match for new identities**; ADD
  `hasconfig` and `both` as selectable options in the create/add wizard, using the
  existing live `includeIf` preview (5.6 D-06). This resolves the long-open
  `gitid gitdir` vs `recipe hasconfig` divergence in favor of *gitdir default,
  hasconfig exposed* (was an open decision in project memory).
- **D-03:** The `hasconfig` value auto-derives from the identity's alias as
  `hasconfig:remote.*.url:git@<alias>:*/**` (recipe form), shown in the live preview
  and editable before write. `both` = the two `includeIf` blocks OR-applied by git
  (carry the 5.5 "both = OR-applied" wording).

### Fragment adoption (ADOPT-01)
- **D-04:** Default action = **MIGRATE** — copy the detected plain-style
  `~/.gitconfig_<name>` into gitid's `~/.gitconfig.d/<name>` convention and rewrite
  the managed `includeIf` to point at the new path. Offer **"reference in place"**
  (point `includeIf` at the existing flat file) as the alternative.
- **D-05:** Migrate is **copy-then-repoint, never destructive.** The original
  `~/.gitconfig_<name>` is preserved; every touched file gets a timestamped backup.
  After a successful migrate, OFFER (confirm) to remove the now-unreferenced
  original — never delete it automatically (CLAUDE.md safe-write rule).
- **D-06:** Detection reuses doctor's existing orphan / non-included-fragment
  detection (Phase 4). Adoption is surfaced by turning the Phase 5.6 read-only
  "unmanaged" section (D-12/D-13) into an actionable **Adopt** affordance, AND as a
  CLI command (full parity — D-10).

### add repo (REPO-01)
- **D-07:** `gitid add repo <url>`: detect provider from URL → ask **personal /
  client** (candidates from `~/git/<client>` folders AND existing accounts; new
  client allowed on the fly) → rewrite the clone URL to the resolved alias → clone
  into the per-client base dir → verify with `git -C <dest> pull`, showing clone +
  pull output.
- **D-08:** When NO identity/alias matches the URL's provider → **offer inline
  create**: launch the proven create-identity prove-before-write flow, then continue
  to alias-rewrite + clone in the same continuous flow (no abort, no silent
  default-identity fallback).
- **D-09:** Base dir default = `~/git`, per-client subfolder
  `~/git/<client>/<repo>` (PRD open-decision #2 — assumed `~/git`; confirm at plan
  time if it changes). A configurable base dir is a nice-to-have, not required.

### Surface split — full TUI + CLI parity
- **D-10:** ALL four capabilities are reachable from BOTH the integrated TUI and the
  CLI (user choice: full parity). `add repo` = a CLI command + an in-app interactive
  clone flow; adoption = the unmanaged-pane Adopt action + a CLI command; the match
  selector = wizard/detail control (+ CLI create where flags exist); gh/glab upload =
  wired into the create/copy flow in both surfaces.

### gh/glab assisted upload (AUTOUP-01)
- **D-11:** **Detect + prompt, never auto.** If `gh`/`glab` is present, offer to run
  the upload command but require **explicit per-key confirmation each time**; the
  default path remains today's manual upload instructions. Non-blocking / best-effort
  — it never gates create or the test loop.
- **D-12:** When run, upload BOTH the **authentication** key and the **signing** key
  per the user's per-key confirmation (GitHub accepts both types); show the exact
  command + its result. Auth model = reuse existing `gh auth` / `glab auth` status;
  if unauthenticated, fall back to manual instructions (gitid never drives an
  interactive login).

### Cross-cutting (locked — carried from 5.6/5.5, not re-decided)
- Every mutation of `~/.ssh/config`, `~/.gitconfig`, and fragments: timestamped
  backup → preview/diff → explicit confirm → idempotent managed-block rewrite;
  validate with a parse→render→parse pass (SAFE-01/02/03).
- Structural writes go through the prove-before-write test loop; **copy = public key
  only**. Core stays UI-free + TDD.

### Verification (anti-blindspot — RECURRING)
- **D-13:** Verification MUST drive the **real program entry** — actual `gitid`
  subcommands and the live TUI over a real TTY — not injected seams or model
  internals (repeats the Phase 4 / 5 / 5.6 lesson, D-16). `add repo`'s clone + pull
  must be exercised against a real/realistic remote, not a fake.

### Claude's Discretion
- Whether existing identities can be retro-fitted `gitdir → hasconfig` via the
  detail-pane match control (it already renders both kinds) — follow the existing
  `tui/detail.go` match editor; new identities get the choice at create time.
- How a detected fragment is matched to an identity name (filename suffix `_<name>`,
  falling back to the email inside the fragment) — best-effort, confirm mapping
  before writing.
- New package boundary for gh/glab invocation (e.g. `internal/uploader`) vs
  extending `internal/platform` — planner's call; `os/exec` without shell.
</decisions>

<specifics>
## Specific Ideas

- The recipe's PRIMARY match is `hasconfig` (`git@<alias>:*/**`); the user keeps
  `gitdir` as gitid's default but wants `hasconfig`/`both` selectable — resolving the
  project-memory open decision `gitid-default-match-divergence` as "gitdir default,
  expose hasconfig/both."
- `add repo` no-match → **inline create** (one continuous flow), never abort.
- gh/glab = **detect + prompt, never auto**; manual instructions stay the default.
- **Full TUI + CLI parity** for all four capabilities.
- Migrate must never destroy the user's original fragment (copy + offer-to-remove).
</specifics>
