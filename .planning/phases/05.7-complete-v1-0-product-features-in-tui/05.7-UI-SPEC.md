# Phase 05.7 — Terminal TUI Interaction Contract

> This phase has no graphical or web UI. This document is a **terminal TUI
> interaction contract** for the Bubble Tea v2 / Lipgloss v2 / Bubbles v2
> application built in Phase 5.6. All six UI-SPEC dimensions map to terminal
> idioms — character-cell layout, ANSI color, keymap, and copywriting.
>
> **Baseline inheritance:** Phase 05.6-UI-SPEC.md is the LOCKED design-system
> baseline. Every token, glyph, layout rule, style declaration, and keymap
> defined there carries into this phase WITHOUT CHANGE. This document contracts
> ONLY the four NEW surfaces Phase 5.7 adds:
>
> 1. Match-strategy selector (wizard + detail pane) — completes 5.6 D-06
> 2. Fragment adoption flow (unmanaged pane → Adopt modal)
> 3. `add repo <url>` interactive clone flow (new in-app modal + CLI mirror)
> 4. `gh`/`glab` assisted key upload (detect+prompt, wired into copy/create flow)
>
> **North Star alignment:** every editor surface that produces `includeIf` or
> `insteadOf` blocks must output structure matching `recipes/gitconfig.recipe`
> (hasconfig primary, gitdir as alternative, `git@<alias>:*/**` pattern) using
> ed25519 keys. The recipes define the canonical end state; gitid keeps `gitdir`
> as its DEFAULT match per 05.7-CONTEXT.md D-02 but exposes `hasconfig`/`both`.
>
> **Sources for pre-populated decisions:**
> - `05.7-CONTEXT.md` D-01..D-13 (locked decisions — all binding)
> - `05.6-UI-SPEC.md` (locked baseline — inherit verbatim; not repeated here)
> - `05.6-CONTEXT.md` D-12/D-13 (unmanaged section that adoption builds on)
> - `recipes/gitconfig.recipe` (canonical hasconfig pattern)
> - `CLAUDE.md` (safe-write, English-only, hypothesis→test→implement)

## New Surface 1: Match-Strategy Selector (Wizard + Detail Pane)

> Source: 05.7-CONTEXT.md D-02, D-03. Completes the deferred 05.6-CONTEXT.md D-06.
> The match-strategy control already exists in `tui/wizard.go` and `tui/detail.go`
> but only supports `gitdir`. This surface adds `hasconfig` and `both`.

### 1a. Wizard Step 1 — Match Strategy Field (expanded)

**Trigger:** Tab to focus the `Match Strategy:` field in the Create Identity wizard.

**Before focus (collapsed — same as 5.6 baseline):**
```
  Match Strategy:   gitdir (folder)
```

**After focus (expanded inline within the wizard modal):**
```
  Match Strategy:

    [x] gitdir (folder)
        gitdir path:  [~/git/personal/                  ]

    [ ] hasconfig (repo URL)
        URL pattern:  [git@personal.github.com:*/**      ]

    [ ] both (gitdir + hasconfig, OR-applied by git)

  Preview:
      [includeIf "gitdir:~/git/personal/"]
          path = ~/.gitconfig.d/personal
```

**Layout rules:**
- The field expansion pushes the wizard modal taller; if the modal would exceed
  `terminalHeight - 6` rows, enable `bubbles/v2 viewport.Model` scrolling within
  the modal.
- Option rows: `PaddingLeft(2)` relative to modal inner left edge.
- Sub-field rows (gitdir path, URL pattern): `PaddingLeft(4)`.
- Preview block: `PaddingLeft(4)`, headed by `StyleHeader` `Preview:` label.
- The live preview updates on every keystroke in the sub-fields.

**Default state:**
- `gitdir` pre-selected; path defaults to `~/git/<identity-name>/` (derived from
  the Identity Name field value, or blank if not yet filled).

**hasconfig pattern derivation (D-03):**
- Auto-derives on focus to `git@<alias>:*/**` where `<alias>` is built from the
  SSH Alias field (or `<identity-name>.<provider>` when SSH Alias is blank).
- Pattern is editable before write.
- Rendered in preview as:
  ```
  [includeIf "hasconfig:remote.*.url:git@personal.github.com:*/**"]
      path = ~/.gitconfig.d/personal
  ```
  This matches the `recipes/gitconfig.recipe` canonical structure.

**`both` option behavior:**
- Selecting `both` reveals BOTH sub-fields (gitdir path AND URL pattern).
- The live preview shows two `includeIf` blocks stacked:
  ```
  [includeIf "gitdir:~/git/personal/"]
      path = ~/.gitconfig.d/personal

  [includeIf "hasconfig:remote.*.url:git@personal.github.com:*/**"]
      path = ~/.gitconfig.d/personal
  ```
- Footer hint appends: `git applies these as OR (either match activates the fragment)`
  rendered as `StyleFaint` below the preview.

**Navigation within the selector:**
- `↑`/`↓` or `j`/`k` move between `gitdir`, `hasconfig`, `both` options.
- `space` or `enter` selects the focused option.
- `tab` when on a sub-field advances to the next sub-field, then to the next
  wizard field (Signing).
- `shift+tab` reverses.

**Validation:**
- `gitdir` path must not be blank if strategy includes `gitdir`.
  Error: `! gitdir path is required` (`ColorWarning`, below the sub-field).
- `hasconfig` URL pattern must match `git@*:*/**` or `https://*/*/**`.
  Error: `! URL pattern must be git@<alias>:<org>/**` (`ColorWarning`).

**Footer while match-strategy selector is focused:**
```
↑↓ select  space choose  tab next field  esc cancel
```

### 1b. Detail Pane — Match Strategy Inline Edit

**Trigger:** `e` to enter inline edit mode on an existing identity; Tab to the
Match Strategy field.

**Existing behavior (5.6 baseline):** the field shows the current strategy type
and value. Pressing `Enter` on this field opens the prove-before-write modal.

**Extension for Phase 5.7:** the expanded selector renders inline in the detail
pane (not a modal), identical in structure to the wizard expansion above, but
displayed in the main pane body below the field label row. The current strategy
is pre-selected. Editing and saving triggers the prove-before-write loop (D-07
from 05.6-CONTEXT.md, unchanged rule).

**Retrofit note (Claude's Discretion per D-13 in 05.7-CONTEXT.md):** if the
existing identity already has `hasconfig` or `both`, `strategyNumFromKind` is
used to pre-fill (existing pattern from Phase 5.5, `tui/detail.go`); no change
to that logic, only the visual selector replaces the prior text-only field.

## New Surface 2: Fragment Adoption Flow (ADOPT-01)

> Source: 05.7-CONTEXT.md D-04, D-05, D-06. Turns the Phase 5.6 read-only
> "unmanaged" sidebar section (05.6-CONTEXT.md D-12/D-13) into an actionable
> Adopt affordance.

### 2a. Sidebar — Unmanaged Entry with Adopt Affordance

**Baseline (unchanged from 5.6):**
```
Unmanaged
  ○ gitconfig_work ~
  ○ orphan_key.pub ~
```

**Phase 5.7 extension — when a fragment-type unmanaged entry is focused:**
```
Unmanaged
  › gitconfig_work  [Adopt]
  ○ orphan_key.pub ~
```

- `[Adopt]` label replaces the `~` glyph on the focused row.
- `[Adopt]` rendered in `ColorAccent` (blue), bold.
- Pressing `Enter` or `A` (Shift+A) on a fragment-type unmanaged entry opens the
  Adopt modal.
- Orphan keys (`.pub` files, raw key files) do NOT show `[Adopt]` — they are
  still read-only with `~` and light affordances only (reveal path, copy pubkey).
  Adoption applies only to plain-style gitconfig fragments (`~/.gitconfig_<name>`
  or `~/.gitconfig.d/<name>` not yet under gitid management).

**Footer when a fragment-type unmanaged entry is focused:**
```
q quit  tab focus  ↑↓ move  A adopt  ? help
```

### 2b. Adopt Modal (modal overlay over dimmed layout)

**Triggered by:** `A` or `Enter` on a focused fragment-type unmanaged entry.

**Modal title:** `Adopt Fragment: <short-name>`

**Modal box:** same style as all 5.6 modals — `StyleModal` (rounded border,
`ColorAccent` blue, `Padding(1, 2)`, `min(terminalWidth-8, 72)` cols max).

**Step 1: Strategy selection**

```
╔═══════════════════════════════════════════════════╗
║  Adopt Fragment: gitconfig_work                   ║
║                                                   ║
║  Fragment detected:  ~/.gitconfig_work            ║
║  Matched identity:   work  (email: w@corp.com)    ║
║                                                   ║
║  Choose adoption method:                          ║
║                                                   ║
║  [x] Migrate  (copy into ~/.gitconfig.d/work,     ║
║                repoint includeIf to new path)     ║
║                                                   ║
║  [ ] Reference in place  (point includeIf at      ║
║                            the existing file)     ║
║                                                   ║
║  Note: original file is preserved. After migrate  ║
║  you will be asked whether to remove the original.║
║                                                   ║
║  tab/space select  enter next  esc cancel         ║
╚═══════════════════════════════════════════════════╝
```

- `[x]` / `[ ]` radio buttons: `StylePass` / `StyleFaint` (same pattern as
  match-strategy selector).
- Default: **Migrate** pre-selected (05.7-CONTEXT.md D-04).
- `Matched identity:` line: shows the identity name + email gitid inferred from
  the fragment filename or content (best-effort, D-06); if ambiguous, shows
  `(unconfirmed — review before writing)` in `StyleFaint`.

**Step 2: Preview/diff and backup notice**

```
╔═══════════════════════════════════════════════════╗
║  Adopt Fragment: gitconfig_work — Preview         ║
║                                                   ║
║  Action: copy ~/.gitconfig_work                   ║
║       →  ~/.gitconfig.d/work                      ║
║                                                   ║
║  Backup: ~/.gitconfig_work.bak.<ts> (before write)║
║          ~/.gitconfig.bak.<ts>     (before write) ║
║                                                   ║
║  includeIf rewrite in ~/.gitconfig:               ║
║    [includeIf "gitdir:~/git/work/"]               ║
║        path = ~/.gitconfig.d/work                 ║
║                                                   ║
║  Write these changes? [Enter to confirm / Esc]    ║
╚═══════════════════════════════════════════════════╝
```

- `Action:` lines: `StyleLabel` for `Action:` (bold, 16-char fixed width);
  `StyleBody` for source path; `StyleFaint` + arrow for destination.
- `Backup:` lines: `StyleFaint`.
- `includeIf rewrite` block: `StyleHeader` label; content `StyleFaint` +
  `PaddingLeft(4)`.
- `Write these changes?` CTA: `StyleBody`.

**Step 3: Write progress and result**

After `Enter`:
```
  [spinner] writing...

  ✓ Backed up ~/.gitconfig_work → ~/.gitconfig_work.bak.20260620-143201
  ✓ Backed up ~/.gitconfig → ~/.gitconfig.bak.20260620-143201
  ✓ Copied fragment to ~/.gitconfig.d/work
  ✓ includeIf block updated in ~/.gitconfig
```
Each line: glyph + `StylePass` text, appended as operations complete.

**Step 4: Offer to remove original (only for Migrate)**

```
  Original file preserved at ~/.gitconfig_work

  It is now unreferenced (gitid points to the new copy).
  Remove the original? [y/N]:  [N]

  Enter to confirm removal  /  n or Esc to keep it
```

- First two lines: `StyleBody`.
- `Remove the original?` prompt: `StyleLabel` + interactive toggle.
- Default: **N** (do not remove). Only on explicit `y` + `Enter` is the file
  removed (05.7-CONTEXT.md D-05 — never auto-delete).
- If user enters `y`:
  - Show: `! This will permanently remove ~/.gitconfig_work. [warning]`
    (`!` glyph + `ColorWarning`)
  - Second `Enter` required to confirm the removal.
  - On removal: `✓ Removed ~/.gitconfig_work` (`StylePass`).
  - On removal failure: `✗ remove failed: <err> [critical]\nFile kept.`

**Step 5: Completion**

Modal closes. The sidebar row for the adopted fragment moves from `Unmanaged`
into the `Identities` managed section. A success toast appears:

```
Fragment "gitconfig_work" adopted as identity "work".
```
(`StylePass`, auto-dismisses after 3 seconds per 5.6 toast rule.)

**"Reference in place" path difference:**
- Step 2 preview shows the `includeIf` pointing at the original path, no copy
  line.
- Steps 3–4: no "remove original" offer. The original file stays in place.
- Toast: `Fragment "gitconfig_work" referenced as identity "work".`

**Error states:**

| Context | Copy |
|---------|------|
| Identity mapping ambiguous | `! Cannot determine identity name. Enter name manually: [        ]` (`ColorWarning`; renders a `textinput.Model` for manual entry) |
| Write failed | `✗ adopt failed: <err> [critical]\nNo changes were written. Press Esc to go back.` |
| Source file not readable | `✗ fragment not readable: <err> [critical]\nCheck file permissions.` |

**CLI parity (D-10):**
- `gitid adopt <path>` mirrors this flow non-interactively.
- `gitid adopt --method migrate|reference --name <name> <path>` flags available.
- Non-TTY: plain text output showing each step; confirmation via `--yes` flag.

**Keymap additions (Adopt modal):**

| Key | Action | Footer hint |
|-----|--------|-------------|
| `A` (Shift+A) | Open Adopt modal from sidebar | `adopt` |
| `tab` / `space` | Cycle/select adoption method (step 1) | `select` |
| `y` | Confirm removal of original (step 4 only) | (inline prompt) |
| `enter` | Advance step / confirm | `confirm` |
| `esc` | Cancel / close modal | `cancel` |

## New Surface 3: Add Repo Interactive Clone Flow (REPO-01)

> Source: 05.7-CONTEXT.md D-07, D-08, D-09.

detail pane press `r` (new binding, lowercase r conflicts with `KeyRefresh` in the
