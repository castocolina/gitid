## CRITICAL

1. **Adoption candidate detection does not exist and is wrongly assumed to be reusable.**
   `internal/doctor/checks/orphans.go` detects three classes — SSH-only managed blocks,
   gitconfig-only managed blocks, and unreferenced key files. It does **not** glob
   `~/.gitconfig_*` or detect plain-style fragments. RESEARCH/CONTEXT/Plan 02 claim adoption
   "reuses doctor's orphan detection" — false; `CheckOrphans` returns **zero** adoption
   candidates. `adopter.ListCandidates` must be built from `filepath.Glob(~/.gitconfig_*)`.
   **Fix:** add an explicit candidate-detection task (not a reuse of `CheckOrphans`); correct
   the RESEARCH/CONTEXT drift in Plan 02's premise.

2. **`unmanagedEntry` has no fragment-type discriminator, so Plan 07's Adopt affordance is unexecutable.**
   `tui/sidebar.go` `unmanagedEntry` (lines 17–33) holds only `shortName`/`keyPath`/`pubLine`;
   `buildUnmanaged` (194–221) populates it **only** from orphan key paths via
   `AllSSHHostIdentityFiles` — never from `~/.gitconfig_*` fragments (line 23 comment: "Adopt
   into gitid is explicitly deferred"). Plan 07 Task 1 dispatches `[Adopt]` on "fragment-type
   rows" that the sidebar never creates. **Fix:** add (earlier wave) an `isFragment`/`kind`
   field on `unmanagedEntry` **and** the logic that populates the sidebar with plain-style
   fragment rows, before Plan 07 consumes them.

3. **Plan 01 Task 3 arity strategy is ambiguous (compiling-RED vs non-compiling).**
   Live `buildTUIDeps()` returns **5** values `(doctor.Deps, identity.Deps, identity.UpdateDeps,
   identity.DeleteDeps, error)`. The acceptance criteria demand `go build ./... exits 0` **and** a
   `TestBuildTUIDepsNilGuard_Phase57` that fails RED against an 8-value return — mutually
   incompatible in Wave 0. **Fix:** lock the exact compiling-RED variant (placeholder that
   compiles vs zero-value-Deps form) with concrete code, so an executor doesn't choose at compile
   time.
