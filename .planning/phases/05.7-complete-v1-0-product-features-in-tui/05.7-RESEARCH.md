## Summary

Phase 5.7 closes the four genuinely-unbuilt v1.0 gaps. All four build directly on the Phase 5.6 integrated TUI and Phase 5.5 core; no new frameworks or third-party packages beyond the optional `teatest/v2` test helper are needed.

**Match-strategy selector** is primarily a wiring gap: `gitconfig.MatchGitdir` and `gitconfig.MatchHasconfig` both exist; the wizard's UI only exposes `gitdir` today. Adding `hasconfig`/`both` means extending the `createWizardModel` form in `tui/wizard.go` with a radio-selector sub-model, deriving the hasconfig pattern from the alias field, and rendering via the existing `gitconfig.RenderIncludeIf`. The detail-pane retrofit follows the same pattern in `tui/detail.go`.

**Fragment adoption (ADOPT-01)** reuses `internal/doctor/checks/orphans.go` for candidate detection and `internal/filewriter` for the copy-then-repoint write path. A new `internal/adopter` package encapsulates the migration/reference logic so it is testable without TUI involvement. The TUI surfaces it through the existing `sidebarModel.unmanaged` slice, adding the Adopt affordance on fragment-type entries.

**Add repo (REPO-01)** orchestrates three external operations: provider detection (URL parsing), `git clone`, and `git pull`. A new `internal/repoclone` package holds the core logic. The TUI modal streams clone/pull output line-by-line via a `tea.Cmd` that reads from a pipe. The no-match path reuses the existing `createWizardModel` as an inline sub-modal. The CLI command `gitid add repo` is a new Cobra sub-command under `cmd/gitid`.

**gh/glab upload (AUTOUP-01)** is a thin wrapper around `os/exec` with detection and auth-status probing. A new `internal/uploader` package handles the `gh`/`glab` detect → auth-check → upload-per-key flow. It is wired into `tui/copy.go` (copy modal upload-assist section) and wizard step 3. The exact commands from the UI-SPEC are confirmed against official gh/glab CLI docs.

**Primary recommendation:** implement all four features as new `internal/` packages with injected-seam `Deps` structs (following the established pattern), wire into TUI `deps.go`, and extend the e2e harness to cover CLI paths before TUI paths (D-13 mandate).

### Pattern 1: Deps-Struct Injected Seam (established — extend for each new package)

Every new `internal/` package exposes a `Deps` struct with function fields, built live in `tui/deps.go` and `cmd/gitid/` entry points. This is the **exact pattern** used by `identity.Deps`, `doctor.Deps`, `identity.DeleteDeps`, and `identity.UpdateDeps`.

```go
// Source: established pattern from internal/identity, internal/doctor
// internal/adopter/adopter.go — Deps struct (new)
type Deps struct {
    ReadFile       func(path string) ([]byte, error)
    WriteFile      func(path string, content []byte, mode os.FileMode) (string, error)
    CopyFile       func(src, dst string) error
    RemoveFile     func(path string) (string, error)  // returns backupPath
    ReplaceBlock   func(content []byte, name, body string) []byte
    ReadFragment   func(path string) (gitconfig.FragmentInfo, error)
    ListGitdotDir  func(dir string) ([]string, error)  // reads ~/.gitconfig_* candidates
    WriteIncludeIf func(gitconfigPath, name, fragPath string, matches []gitconfig.Match) (string, error)
}
```

### Pattern 2: Streaming os/exec Output via tea.Cmd (new — add repo)

The `git clone` and `git pull` commands must stream output line-by-line into the TUI viewport. Use a `tea.Cmd` that reads from `cmd.StdoutPipe()` and sends messages:

```go
// internal/repoclone/repoclone.go
// Deps.Clone runs git clone, streaming output via the lineOut channel.
type CloneResult struct {
    Lines  []string
    Err    error
}

// tui/addrepo.go — streaming via tea.Cmd
func streamClone(deps repoclone.Deps, cloneURL, destPath string) tea.Cmd {
    return func() tea.Msg {
        result := deps.Clone(cloneURL, destPath)
        return cloneResultMsg{result: result}
    }
}
// For line-by-line streaming, use a separate tea.Cmd per read:
func readCloneLine(r io.Reader) tea.Cmd {
    return func() tea.Msg {
        scanner := bufio.NewScanner(r)
        if scanner.Scan() {
            return cloneLineMsg{line: scanner.Text()}
        }
        return cloneDoneMsg{}
    }
}
```

**Alternative streaming approach:** collect all output, emit in one `cloneResultMsg`. This is simpler and avoids per-line command chaining. Acceptable since clone output is bounded in size. Use this unless the UI-SPEC's line-by-line illusion is required.

### Anti-Patterns to Avoid

- **Injected-seam test covering unwired live path (D-13 / recurring blindspot):** writing a TUI test that passes a `fakeAdoptDeps{}` into `adoptModel` while `buildTUIDeps()` wires `adopter.Deps{}` with a nil `WriteFile`. The `TestBuildTUIDepsNilGuard` pattern from `tui/wiring_test.go` must be extended to cover all new Deps structs.
- **Building new clone/upload logic in the TUI model:** the `addRepoModel` must not contain `os/exec` calls. All exec calls go through `internal/repoclone.Deps` and `internal/uploader.Deps`.
- **Calling `shell` expansion in os/exec:** every `exec.Command` is constructed with explicit arg slices (no `exec.Command("sh", "-c", ...)`) per CLAUDE.md / gosec G204.
- **Auto-deleting original fragment files:** the `internal/adopter.Adopt()` function must never call `os.Remove` on the source file without an explicit `removeOriginal bool` parameter set to `true`.
- **Hardcoding `~/git` as an absolute path constant:** expand `~` to `os.UserHomeDir()` at runtime.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic write + backup | Custom copy/rename | `filewriter.Write()` | Already handles backup + temp-then-rename + chmod; every adoption write routes here |
| includeIf block rendering | String concatenation | `gitconfig.RenderIncludeIf()` / `gitconfig.WriteIncludeIf()` | Handles newline-injection guard, trailing slash on gitdir, sentinel delimiters |
| Managed block replace | Regex / sed | `filewriter.ReplaceBlock()` | Idempotent; handles missing block (append) and present block (replace) correctly |
| Orphan fragment detection | Custom file-walk | `doctor/checks.CheckOrphans()` / `doctor.Deps` | Already finds non-included fragments; feeds adoption candidates |
| SSH config write | Direct file write | `sshconfig.Write()` | Handles managed block, global block ordering, idempotency |
| Fragment read/validate | Custom INI parser | `gitconfig.ReadFragment()` | Already implemented; extracts email for name matching in adoption |
| URL rewrite (HTTPS→SSH alias) | String replace | `sshconfig.ParseManagedHosts()` + alias lookup | Managed hosts already indexed by alias; provider → alias lookup is trivial |

**Key insight:** the project has a deep internal library covering all file-mutation concerns. The new packages (`adopter`, `repoclone`, `uploader`) are orchestration layers that compose existing primitives, not new infrastructure builders.

## Common Pitfalls

### Pitfall 1: Fragment Name Matching (adoption)

**What goes wrong:** a file named `~/.gitconfig_work` maps to identity `work`, but the email inside might be `john.doe@corp.com` and the identity in gitid might be named `acmecorp`. Ambiguous mapping causes a silent wrong-identity adoption.

**Why it happens:** gitid identity names don't have a canonical registry — they are inferred at runtime via `identity.Reconstruct()`.

**How to avoid:** implement best-effort matching: (1) check filename suffix against known identity names; (2) check email in fragment against `identity.Account.Email` from `Reconstruct()`. If both fail, surface the ambiguous-identity error state from UI-SPEC and render a `textinput.Model` for manual name entry.

**Warning signs:** test case where fragment email != gitid identity name.

### Pitfall 2: Live Path Not Exercised (D-13 recurring anti-blindspot)

**What goes wrong:** unit tests pass `fakeAdoptDeps` with no-op stubs → all green. `buildTUIDeps()` wires `adopter.Deps{WriteFile: nil}` → panic at runtime.

**Why it happens:** the Phase 4/5/5.6 blindspot: green tests over injected seams hide unwired live paths.

**How to avoid:** extend `TestBuildTUIDepsNilGuard` in `tui/wiring_test.go` to assert non-nil for every field of the new Deps structs. Write this test FIRST (Wave 0, TDD-RED) before implementing `buildTUIDeps()` wiring.

**Warning signs:** a test that only calls `newAdoptModel(fakeAdoptDeps)` without ever calling `buildTUIDeps()`.

### Pitfall 3: gitdir path without trailing slash

**What goes wrong:** `[includeIf "gitdir:~/git/personal"]` (no trailing slash) does not match — git requires a trailing slash for directory matchers.

**Why it happens:** user input or auto-derivation omits the slash.

**How to avoid:** `gitconfig.Match.condition()` already normalizes the trailing slash (`Match.condition()` in `internal/gitconfig/renderer.go`). Always construct `Match{Kind: MatchGitdir, Value: path}` and let `condition()` add the slash — never hand-build the condition string.

### Pitfall 4: gh/glab invocation with shell expansion

**What goes wrong:** `exec.Command("sh", "-c", "gh ssh-key add "+path)` — path injection risk; gosec G204/G204 flag.

**Why it happens:** shell expansion feels easier when building multi-flag commands.

**How to avoid:** always use explicit arg slices: `exec.Command("gh", "ssh-key", "add", pubPath, "--title", title, "--type", keyType)`. Every field in the arg slice is a Go string variable, never a shell-interpolated expression.

### Pitfall 5: Add Repo destination already exists

**What goes wrong:** `git clone` into an existing directory fails with `fatal: destination path '...' already exists`. The UI-SPEC defines an overwrite prompt for this case, but the code may panic or silently fail if the directory exists check is absent.

**Why it happens:** user has already cloned the repo manually.

**How to avoid:** `internal/repoclone.Clone()` must check `os.Stat(dest)` before executing `git clone`. Return a typed `ErrDestExists` error; the TUI modal handles this with the overwrite prompt.

### Pitfall 6: Modal stack depth (add repo → inline create)

**What goes wrong:** the add-repo modal invokes the create wizard as a sub-modal. The 5.6 modal rule is "at most 1 modal open at a time." If both modals render simultaneously, the layout breaks.

**Why it happens:** misunderstanding the modal-stack rule from 05.6-UI-SPEC.md.

**How to avoid (per UI-SPEC):** the add-repo modal REPLACES itself with the create wizard while the wizard runs. The addRepoModel saves its state (cloneURL, detectedProvider) and resumes when the wizard emits a success/cancel message. Implement as a sub-state enum inside `addRepoModel`: `stepDetect`, `stepInlineCreate`, `stepClientPicker`, etc.

## Code Examples

### Match-Strategy Wizard Extension (gap fill)

```go
// tui/wizard.go — extend existing createWizardModel form
// Source: internal/gitconfig/renderer.go (RenderIncludeIf) + tui/wizard.go (existing form)

// matchStrategy is the selector state for the wizard match-strategy field.
type matchStrategy int

const (
    strategyGitdir     matchStrategy = iota // default (D-02)
    strategyHasconfig
    strategyBoth
)

// liveIncludeIfPreview renders the current includeIf preview string.
// Called on every keystroke in the gitdir/hasconfig sub-fields.
func liveIncludeIfPreview(strategy matchStrategy, name, alias, gitdirVal, hasconfigVal string) string {
    fragPath := "~/.gitconfig.d/" + name
    var matches []gitconfig.Match
    switch strategy {
    case strategyGitdir:
        matches = []gitconfig.Match{{Kind: gitconfig.MatchGitdir, Value: gitdirVal}}
    case strategyHasconfig:
        matches = []gitconfig.Match{{Kind: gitconfig.MatchHasconfig, Value: "remote.*.url:" + hasconfigVal}}
    case strategyBoth:
        matches = []gitconfig.Match{
            {Kind: gitconfig.MatchGitdir, Value: gitdirVal},
            {Kind: gitconfig.MatchHasconfig, Value: "remote.*.url:" + hasconfigVal},
        }
    }
    return gitconfig.RenderIncludeIf(name, fragPath, matches)
}

// defaultHasconfigPattern derives the canonical hasconfig pattern from the alias.
// Recipe form: git@<alias>:*/** (05.7-CONTEXT.md D-03)
func defaultHasconfigPattern(alias string) string {
    if alias == "" {
        return ""
    }
    return "git@" + alias + ":*/**"
}
```

func TestAddRepo_LocalClone(t *testing.T) {

    cmd := exec.CommandContext(ctx, bin, "add", "repo", cloneURL, "--client", "personal", "--yes")

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Shell injection via clone URL | Tampering | `exec.Command("git", "clone", cloneURL, dest)` — explicit arg slice, never `sh -c`; gosec G204 |
| Shell injection via pubPath in gh/glab upload | Tampering | `exec.Command("gh", "ssh-key", "add", pubPath, ...)` — explicit arg slice |
| Symlink attack on fragment source path | Elevation of Privilege | `os.Lstat()` before read; reject if symlink outside home dir |
| Clone into attacker-controlled destination | Elevation of Privilege | Validate `dest` is under `~/git/` before running git clone; reject absolute paths from URL |
| Fragment adoption overwriting managed identity | Tampering | Check identity name not already in `identity.Reconstruct()` before adopting; error if conflict |
| Backup file world-readable | Information Disclosure | All backups via `filewriter.Write()` which sets mode 0600 unconditionally |

## Open Questions

1. **glab signing key separate upload?**
   - What we know: `glab ssh-key add --usage-type auth` is documented to cover authentication. GitLab allows "Authentication & Signing" as a usage type.
   - What's unclear: whether `--usage-type auth` alone enables commit signing, or if `--usage-type auth_and_signing` is the correct flag.
   - Recommendation: run `glab ssh-key add --help` at Wave 0 on the CI machine; implement with `--usage-type auth_and_signing` if available, else `auth` for now and document the limitation.

   - What we know: the direct `Update(msg)` pattern is sufficient and is the established 5.6 approach. teatest/v2 adds a rendered-frame integration layer.
   - What's unclear: whether the rendered-frame layer catches bugs the `Update()` layer misses for the new modals.
   - Recommendation: planner's call (Claude's Discretion). Start without it; add if model-integration bugs arise in Wave 1.

3. **Fragment name matching ambiguity handler**
   - What we know: ambiguous mapping surfaces a `textinput.Model` for manual name entry (UI-SPEC error state).
