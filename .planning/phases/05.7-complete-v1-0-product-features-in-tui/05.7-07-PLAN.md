<objective>
Surface the three new capabilities inside the integrated TUI (D-10 parity, D-13
real-entry), composing the live Deps wired in Plan 06. The CRITICAL ordering
constraint (REVIEWS.md #2): the sidebar today (tui/sidebar.go) only builds
ORPHAN-KEY unmanagedEntry rows — buildUnmanaged (lines 194-216) never globs
~/.gitconfig_* fragments, and unmanagedEntry (lines 24-32) has no fragment-type
field. Plan 07 Task 1 therefore FIRST adds an isFragment/kind discriminator to
unmanagedEntry AND populates the sidebar with plain-style ~/.gitconfig_* rows from
deps.adopt.ListCandidates (live since Plan 06); ONLY THEN does the [Adopt]
affordance + dispatch (Task 1 continued + the modal in Task 1) have real fragment
rows to act on. The Add Repo modal (Task 2) follows.

Surfaces delivered:
- Adopt modal (unmanaged-fragment → migrate/reference → preview → write →
  offer-remove-original, never destructive per D-05)
- Add Repo modal (URL → detect → client picker → rewrite preview → streamed
  clone+pull → result, with no-match inline-create resume per D-08)
- gh/glab upload-assist on the Copy modal + wizard step 3 (detect+prompt,
  non-blocking, per-key confirm per D-11/D-12)

All surfaces follow the 05.7-UI-SPEC interaction contract and the locked 5.6
modal-stack rule (one modal at a time; inline-create REPLACES the add-repo modal
then resumes).

Purpose: This is the user-facing close of the phase. The modals contain NO os/exec
— every effect goes through the injected adopter.Deps/repoclone.Deps/uploader.Deps
seams from Plan 06 (anti-blindspot). State machines are white-box package tui
Update(msg) tested, with the optional teatest/v2 layer left to Claude's discretion.
Output: fragment-aware sidebar + Adopt modal + Add Repo modal + copy/wizard
upload-assist.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TUI keyboard input (URL, fragment selection, client) to injected Deps | All user input reaches core packages through the modal → Deps seam |
| TUI modal to adopter/repoclone/uploader effects | The modals must not perform os/exec or file writes directly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.7-07-01 | Tampering | os/exec or file write inside a TUI model | mitigate | Models hold only injected Deps; Task grep gates assert zero os/exec in adopt.go/addrepo.go/copy.go (RESEARCH anti-pattern) |
| T-05.7-07-02 | Tampering | config corruption via adopt-from-TUI | mitigate | Adopt writes flow through adopter.Deps → filewriter (backup/atomic/idempotent) wired in Plan 06 |
| T-05.7-07-03 | Elevation of Privilege | clone into unexpected dest from TUI | mitigate | Dest computed by repoclone.DestPath + dest-under-base guard (Plan 03); modal only passes client + URL |
| T-05.7-07-04 | Information Disclosure | private key exposed via copy/upload modal | mitigate | Copy = public key only (5.6 locked rule); upload-assist passes only the .pub path |
| T-05.7-07-05 | DoS | two modals rendered (layout break) | mitigate | Modal-stack invariant: inline-create REPLACES add-repo modal; test asserts not-both-rendered (Pitfall 6) |
| T-05.7-07-SC | Tampering | npm/pip/cargo installs | accept | No production installs. teatest/v2 is OPTIONAL/test-only and [ASSUMED]; if adopted, add behind a blocking-human legitimacy checkpoint verifying github.com/charmbracelet/x ownership before go get |
