# Phase 05.5: Code Review Report

**Reviewed:** 2026-06-14
**Depth:** standard
**Files Reviewed:** 23 (+ supporting reads: gitconfig/reader.go, list.go, rotate.go, marker_roundtrip_test.go)
**Status:** issues_found

## Summary

Phase 05.5 reconciled the create-flow into an auth-gated loop, added a provider marker
with hostname fallback, a match-strategy picker, a shared overlap detector, and
install-path self-reporting. The auth-gate ordering is correct in the core paths
(`runCreateNew` writes the key to `~/.ssh` first, then persists config artifacts only
after PASS or an explicit skip+confirm, then re-checks overlap), and the file-perm /
backup / no-shell-exec invariants hold across the writers I traced.

However, the adversarial pass surfaced two correctness defects that ship broken behavior:
(1) the `--skip` (skip-&-write) persist path bypasses the pre-write connectivity gate
entirely — a write happens against a *stale* PASS/RNU outcome that may no longer reflect
reality, and more importantly the loop persists on the *previously printed* `pre` result
without re-probing, but the larger problem is the **provider value is never validated**
before being written into `~/.ssh/config` as a marker comment and consumed by URL/host
builders; and (2) the TUI dashboard silently drops the new Overlap check, so overlap
findings that the CLI reports are invisible (and reported as "all checks passed") in the
TUI. Six warnings and four info items follow.

## Critical Issues

### CR-01: Provider value is never validated before being written to `~/.ssh/config` and consumed by URL/hostname builders

**File:** `cmd/gitid/add.go:54-360`, `cmd/gitid/update.go:47,141-143`, `internal/sshconfig/renderer.go:31-45`, `internal/sshconfig/reader.go:28-37`

**Issue:**
The identity `name` is rigorously sanitized and matched against `identityNameRe`
(`^[A-Za-z0-9._-]+$`) in every entry path. The `provider` value is not validated
anywhere. It is accepted verbatim from `--provider` (add.go:54, update.go:47) and from
the interactive prompt (add.go:359), then:

1. written into `~/.ssh/config` as a marker comment `# gitid: provider=<p>` via
   `RenderHostBlock` (renderer.go:42-44), which only rejects `\n`/`\r`;
2. interpolated into the default URL pattern (`defaultURLPattern` → `git@<hostname>:...`)
   and hostname (`defaultHostname` returns `provider + ".com"` for unknown providers,
   add.go:620);
3. round-tripped back via `extractProviderMarker` (reader.go:28-37), which trims the
   *line* but not the *value*.

Consequences:
- A provider containing whitespace (e.g. `--provider "git hub"`) is written as
  `# gitid: provider=git hub` and read back as `git hub` — but `defaultHostname` already
  produced `git hub.com`, an invalid hostname silently baked into the Host block.
- A provider with leading/trailing spaces (`--provider " github "`) does NOT round-trip:
  `RenderHostBlock` writes `# gitid: provider= github ` and `extractProviderMarker`
  returns `github ` (no value-trim), so reconstruction yields a different provider than
  was requested — breaking the D-11/D-12 marker contract this phase introduced.
- The marker comment is the one piece of attacker/operator-influenced free text that
  lands in a security-sensitive file by design; the `\n`/`\r` guard is a panic, not a
  validated rejection, so a newline aborts the whole command with a stack trace instead
  of a clean error.

**Fix:**
Validate provider with a strict charset before use, symmetric with `identityNameRe`, and
trim the value on read:
```go
// rotate.go (next to identityNameRe)
var providerRe = regexp.MustCompile(`^[A-Za-z0-9._-]+$`)

// add.go gatherCreateInput / update.go runIdentityUpdate, after resolving provider:
provider = strings.TrimSpace(provider)
if provider != "" && !providerRe.MatchString(provider) {
    return identity.CreateInput{}, fmt.Errorf(
        "identity add: invalid provider %q (allowed: letters, digits, '.', '_', '-')", provider)
}
```
```go
// reader.go extractProviderMarker — trim the extracted value:
return strings.TrimSpace(strings.TrimPrefix(line, prefix))
```
Replace the `panic` in `RenderHostBlock` (renderer.go:32-34) with a validated caller-side
rejection so malformed input never reaches the renderer.

### CR-02: TUI dashboard silently omits the new Overlap check — overlaps are invisible and falsely reported as passing

**File:** `tui/dashboard.go:92-130, 215-239`

**Issue:**
`doctor.Families()` was extended this phase to include `FamilyOverlap` (doctor.go:295-306),
and `init()` schedules one `makeFamilyCmd` per family (dashboard.go:80-82). But
`makeFamilyCmd`'s `switch fam` (dashboard.go:94-109) has **no `case doctor.FamilyOverlap`**,
so for the overlap family `fn` stays `nil`. The goroutine then runs the `if fn != nil`
guard (dashboard.go:125-127), produces a `familyResultMsg` with `findings: nil` and no
error, and `update` marks the family `familyLoaded` (dashboard.go:146-149). The `view`
renders an empty loaded family as `"✓ all checks passed"` (dashboard.go:227-230).

Result: the CLI `gitid doctor` correctly warns about overlapping match conditions
(`CheckOverlap` is wired in `buildDoctorDeps`, doctor.go:481), but the TUI dashboard
**always shows the Overlap family as green/passing**, even when real overlaps exist. This
is a correctness defect that hides a safety-relevant warning the phase was built to add.

**Fix:**
Add the missing case so the TUI runs the same check the CLI does:
```go
case doctor.FamilyAgent:
    fn = d.CheckAgent
case doctor.FamilyBaseline:
    fn = d.CheckBaseline
case doctor.FamilyOverlap:
    fn = d.CheckOverlap
}
```
Consider failing closed instead of silently passing: when `fn == nil` for a family that
`Families()` advertises, emit a `familyError` ("check not wired") rather than a green
pass, so a future added family cannot regress into a silent false-OK again.

## Warnings

### WR-01: `--skip` persist path writes config on a stale connectivity result without re-probing

**File:** `cmd/gitid/add.go:418-444`

**Issue:**
`runCreateLoop` calls `deps.PreWrite` once at the top of each loop iteration and prints
it. On `[s] skip` the user confirms and the function returns `(true, true, nil)` — but it
never re-probes. The documented intent (D-05) is "write offline/upload-later", so a stale
result is arguably acceptable; however the warning text printed to the user
(add.go:431) claims the key "is not yet authenticated" unconditionally, even if the last
probe was actually `PASS` (which would have auto-persisted) or `ReachableNotUploaded`. The
real risk: the loop's `pre.Outcome == tester.PASS` check happens *before* the menu, so a
PASS never reaches skip — fine — but a `Failure` outcome (host unreachable / DNS failure)
also reaches the skip branch and will happily persist config pointing at an endpoint that
was just proven unreachable, with the generic "not yet authenticated" message masking the
distinction between "uploaded-but-pending" and "totally broken connectivity". The user
cannot tell which situation they are confirming.

**Fix:**
Differentiate the warning by `pre.Outcome` so a `Failure` (unreachable) skip is called out
distinctly from a `ReachableNotUploaded` (normal upload-later) skip:
```go
case "s", "skip":
    if pre.Outcome == tester.Failure {
        fp(out, "Warning: the host is currently UNREACHABLE (not merely unauthenticated). "+
            "Writing config now will reference an endpoint that failed to connect.\n")
    } else {
        fp(out, "Warning: key is not yet authenticated. Upload the key and verify before use.\n")
    }
```

### WR-02: `loadExistingAccounts` swallows reconstruction errors, weakening the overlap guard

**File:** `cmd/gitid/add.go:482-491`

**Issue:**
`loadExistingAccounts` discards both the home-dir error (returns `nil`) and the
`identity.Reconstruct` error (`accounts, _ := ...`). The result feeds
`warnOverlapAndConfirm` (add.go:182). If `~/.ssh/config` or `~/.gitconfig` is malformed
enough that `Reconstruct` errors, the function returns an empty slice and the overlap
detector compares the new identity against *nothing* — silently skipping the D-16 overlap
safety check the phase added. A user with a partially-corrupt config gets no overlap
warning and writes a colliding identity.

**Fix:**
Surface the error to the caller (or at minimum print a warning) so a reconstruction
failure does not silently disable the overlap guard:
```go
func loadExistingAccounts(out io.Writer) []identity.Account {
    home, err := os.UserHomeDir()
    if err != nil {
        fp(out, fmt.Sprintf("Warning: cannot resolve home dir; skipping overlap check: %v\n", err))
        return nil
    }
    ...
    accounts, rerr := identity.Reconstruct(sshBytes, gcBytes, gitconfig.ReadFragment)
    if rerr != nil {
        fp(out, fmt.Sprintf("Warning: could not reconstruct existing identities; overlap check may be incomplete: %v\n", rerr))
    }
    return accounts
}
```

### WR-03: `CopyPub` failure is swallowed with a no-op assignment, hiding clipboard errors

**File:** `cmd/gitid/add.go:159-161`, `internal/identity/identity.go:309-313`

**Issue:**
`runCreateNew` calls `deps.CopyPub` and on error does `_ = cerr` with a comment that
failure is "non-fatal". Clipboard copy *is* best-effort per CLIP-02, but the user is
explicitly told "(also copied to your clipboard)" (add.go:668) regardless of whether the
copy actually succeeded. When `pbcopy`/`wl-copy`/`xclip` is missing or fails, the user is
misled into thinking the key is on the clipboard and may paste stale clipboard contents
into the provider's key field. The error is observable and actionable (print "clipboard
unavailable; copy the key below manually").

**Fix:**
Track the copy result and adjust the message:
```go
clipOK := deps.CopyPub(staged.PubLine) == nil
...
if clipOK {
    fp(out, "Public key (also copied to your clipboard):\n")
} else {
    fp(out, "Public key (clipboard unavailable — copy manually):\n")
}
```

### WR-04: `promptPort` silently rejects ports > 65535, accepting an invalid port as the default

**File:** `cmd/gitid/add.go:702-708`

**Issue:**
`promptPort` accepts any `n > 0` from `strconv.Atoi`. A value like `99999` parses fine and
`n > 0` is true, so it is returned and written verbatim into the `Port` directive of
`~/.ssh/config`. OpenSSH will then fail at connect time with an opaque error. There is no
upper-bound check.

**Fix:**
```go
if n, err := strconv.Atoi(s); err == nil && n > 0 && n <= 65535 {
    return n
}
return def
```

### WR-05: `AddWiring` ssh-host parser corrupts `keyPath` values containing a colon

**File:** `cmd/gitid/doctor.go:415-436`

**Issue:**
The doctor `AddWiring` fix encodes an SSH host re-add as a colon-delimited string
`"ssh-host:<alias>:<hostname>:<port>:<keyPath>"` and decodes it with
`strings.SplitN(rest, ":", 4)`. `SplitN(...,4)` correctly keeps the trailing keyPath
intact even if it contains colons — *but* the `port` field is parsed as `parts[2]`, and if
any of alias/hostname themselves contained a colon (they should not, but nothing validates
this), the fields shift. More concretely, the encoding is constructed elsewhere from
reconstructed data; a Windows-style or unusual `IdentityFile` is the only field tolerant
of colons here. This is fragile string-protocol coupling between encoder and decoder with
no shared constructor and no validation that the encoded fields are colon-free.

**Fix:**
Introduce a single typed payload (struct + explicit constructor) for the wiring line, or
at minimum validate that alias/hostname/port contain no colon at encode time and document
the invariant. A struct passed through the `Fix` descriptor would eliminate the
stringly-typed channel entirely.

### WR-06: Doctor fix-convergence reads `os.Stdin` directly, ignoring the injected `out`/test seams

**File:** `cmd/gitid/doctor.go:96, 102, 109`

**Issue:**
`runDoctor` takes `out io.Writer` for testability but reaches for the global `os.Stdout`
(line 96), `os.Stdin` (lines 102, 109) directly for terminal detection and the fix reader.
This is the exact "injected-seam wiring blindspot" recorded in project memory as a
RECURRING Phase-4/Phase-5 defect: tests that drive `runDoctor` with a buffer cannot
exercise the interactive fix-convergence path because it binds to the real stdin/stdout.
Any green test over `runDoctor` is not actually testing the converge loop's interactive
branch.

**Fix:**
Thread `in io.Reader` and the terminal-detection booleans as parameters (or via a small
deps struct) so the fix loop is drivable from tests, consistent with the
`MEMORY.md → doctor-injected-seam-wiring-blindspot` rule.

### IN-03: `defaultHostname` fallback `provider + ".com"` produces likely-wrong hostnames

**File:** `cmd/gitid/add.go:613-622`

**Issue:** For any provider other than github/gitlab, the default hostname is
`<provider>.com`. For a self-hosted GitLab or Bitbucket this is almost always wrong and the
user must override it. Combined with the unvalidated provider (CR-01) this can yield
nonsense hostnames. Low impact because it is only a prompt default, but worth a clearer
"unknown — please enter the SSH hostname" empty default.

**Fix:** Return `""` for unknown providers so the prompt forces an explicit hostname rather
than suggesting a guess.

### IN-04: E2E `denied`/`timeout` fake-ssh modes are functionally identical, reducing test signal

**File:** `e2e/harness_test.go:118-130`

**Issue:** The `denied` and `timeout` (and `*` default) branches all `exit 255`; only the
echoed line differs. Since the tester classifies on output text this is intentional, but
the `timeout` mode is never exercised by any of the reviewed e2e tests (`create`/`match`
use only `pass`/`denied`). Dead test capability. The fake-ssh static-literal + `0o700`
WriteFile convention from CLAUDE.md is correctly followed (no `exec.Command("sh","-c",...)`).

**Fix:** Either add a test that asserts `tester.Failure` handling via `timeout` mode or
drop the unused branch.

## Remediation (orchestrator, 2026-06-14)

Both **critical** findings were fixed during phase execution (the gate is advisory,
but these were squarely in-scope defects of Phase 5.5's own features):

- **CR-02** (TUI silently drops the Overlap check) — **FIXED** in commit `8aa5fef`.
  Wired `CheckOverlap` into the TUI doctor deps, added the `FamilyOverlap` dispatch
  case, and hardened `makeFamilyCmd` so an unwired family yields an error instead of
  a false "all checks passed". Regression test added (every `doctor.Families()` entry
  must dispatch + be wired).
- **CR-01** (unvalidated provider value) — **FIXED** in commit `d0d86ab`. Added
  `identity.ValidateProvider` (charset `[A-Za-z0-9._-]`, empty allowed) applied at
  every gather site (create-new, add-account, update `--provider`). Table-test added.

The 6 **warning** and 4 **info** findings remain open and are deferred as advisory
input to Phase 5.6 (notably: `runDoctor` binding to global stdio instead of injected
seams — the recurring injected-seam blindspot; `CopyPub`/reconstruction error
swallowing; `promptPort` upper-bound; the colon-delimited `AddWiring` protocol).
