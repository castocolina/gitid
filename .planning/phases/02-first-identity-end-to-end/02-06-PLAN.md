<objective>
Wire the first user-facing vertical slice: `gitid identity add` (create-new mode)
end-to-end. `internal/identity.Create` orchestrates probe → keygen → clipboard →
pre-write test → unified preview → single confirm → FOUR safe writes (SSH config,
gitconfig includeIf, per-identity fragment, AND `~/.ssh/allowed_signers`) → ssh-add →
resolved test → upload steps, with all dependencies injected so it is testable with
fakes and reusable by the future TUI. A companion `gitid identity test <name>` command
re-runs the resolved phase-2 test on demand. The Cobra commands are the thin, real
(non-throwaway) handlers that Phase 5 builds on. Adds `github.com/spf13/cobra`.

Phase Goal (MVP user story):
**As a** developer juggling multiple Git identities, **I want to** create one identity
and have gitid generate the key, prove it authenticates, write all four coordinated
artifacts safely with backup and confirmation, and show me the upload steps, **so that**
I stop hand-editing four config files and never push with the wrong identity.

Purpose: IDENT-01 create-new end-to-end, SIGN-01 (the fourth artifact `allowed_signers`
is actually written here, not just generated), TEST-01/02/03 two-phase proof, SAFE-03
confirm + dry-run, CLIP-01 copy on generate, UP-01/UP-02 upload steps, D-04 reusable
`identity test` entry point. This is the slice the prior five plans' infrastructure exists
to deliver.
Output: `internal/identity/identity.go`, `cmd/gitid/add.go`, `cmd/gitid/test.go`,
`cmd/gitid/upload.go`, modified `cmd/gitid/main.go` (Cobra root), cobra dep.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user (interactive prompts) → gitid | name/email/alias/provider/passphrase entered by the user |
| gitid → four config files + key | the orchestrator triggers all mutations (ssh config, gitconfig, fragment, allowed_signers) via filewriter/git |
| gitid → ssh-agent / Keychain | new key loaded at runtime via ssh-add |
| gitid → provider network | pre-write and resolved tests connect to the provider |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-23 | Tampering/Elevation | command injection via user-entered name/email/alias passed to ssh/git/ssh-add | mitigate | validate input charset before use; arg-slice exec only for ssh-add and all shell-outs (gosec G204) |
| T-02-24 | Denial of Service | writing before proving reachability corrupts/locks config | mitigate | pre-write test gates the write: Failure aborts with no writes (D-01); writes are atomic+backed-up via filewriter (SAFE-03) |
| T-02-25 | Repudiation/Spoofing | write without explicit user consent | mitigate | single explicit y/N confirmation before any write; `--dry-run` previews only (SAFE-03) |
| T-02-26 | Spoofing | wrong key offered at runtime (stale agent key, same alias) | mitigate | `IdentitiesOnly yes` on the alias block + resolved `ssh -G` assertion; ssh-add loads the correct key (D-08) |
| T-02-34 | Spoofing | signature verifies as untrusted because allowed_signers was never written | mitigate | orchestrator calls WriteAllowedSigners as the fourth coordinated write so the signing principal exists for `git log --show-signature` (SIGN-01) |
| T-02-27 | Information Disclosure | full key paths echoed in prompts/upload text | mitigate | surface alias/identity in user output; avoid printing full secret-file paths beyond what the user needs |
| T-02-SC | Tampering | go get pulls a compromised cobra module | mitigate | cobra pre-pinned + Approved in RESEARCH audit; legitimacy checkpoint above; go.sum checksums |
</threat_model>
