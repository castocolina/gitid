## Summary

Phase 3.1 adds a three-surface baseline layer to gitid: a curated git-config block in `~/.gitconfig.d/00-baseline`, an optional HTTPS→SSH URL-rewrite block in the same file, and a managed global gitignore at `~/.gitignore_global`. All three surfaces reuse the proven `filewriter.ReplaceBlock` / `filewriter.Write` chokepoint from Phases 2–3 — no new low-level machinery is required, only composition.

The phase surfaces four non-trivial implementation concerns that prior research did not address: (1) the `[include]` in-place precedence rule, confirmed by live `git` testing, means the managed `[include]` block MUST be prepended to `~/.gitconfig` on first write (not appended), requiring a `PrependIfNotFound` variant that `ReplaceBlock` alone cannot provide; (2) `git config --file --list` reads ALL keys in a file including those inside sentinel-comment blocks, so the conflict scan (D-11) must use text-scanning via `ListBlocks` to isolate user-owned keys before calling `git config`; (3) both baseline and `url-rewrites` blocks must be rendered as fixed raw text (not via `git config set`) to maintain idempotency and sentinel integrity; (4) `merge.conflictstyle=zdiff3` is confirmed available in git 2.35+ (present on this machine as 2.47.0) but needs a version gate for older environments.

**Primary recommendation:** New package `internal/baseline` (or new file `internal/gitconfig/baseline.go`) with `RenderBaselineBlock`, `RenderURLRewritesBlock`, `WriteBaselineToFile`, `WriteIncludeToGitconfig` (prepend-if-not-found variant), `ScanConflicts`, and `ReadBaselineState`. The Cobra command `gitid baseline setup` / `gitid baseline show` follows the Phase 2/3 thin-command pattern.

## Critical Findings

### C1: `ReplaceBlock` Appends; Floor Model Requires Prepend

**Verified by:** Live test on git 2.47.0 (Cases A and B).

`git config` evaluates `[include]` **in-place**, not at file end. The precedence rule is last-writer-wins per key:

```
Case A: [include] at TOP → included values are FLOORS (user keys below WIN)
Case B: [include] at BOTTOM → included values are CEILINGS (they override user keys above)
```

D-10 requires the `[include]` block to be placed **near the top** of `~/.gitconfig`. `filewriter.ReplaceBlock` appends when the block is not found. Therefore, the first write of the `baseline-include` block must **prepend**, not append.

**Required:** A new function `PrependBlockIfNotFound` in `internal/filewriter`:
- If the block already exists (re-run): delegates to `ReplaceBlock` (in-place splice — position preserved, floor maintained).
- If the block does NOT exist (first run): prepend block before all existing content.

Implementation sketch:
```go
func PrependBlockIfNotFound(existing []byte, name, blockBody string) []byte {
    beginMarker := BeginPrefix + name
    // Check if block already exists
    if bytes.Contains(existing, []byte(beginMarker)) {
        return ReplaceBlock(existing, name, blockBody) // in-place update
    }
    // First write: prepend
    block := buildBlock(name, blockBody) // same canonical form as ReplaceBlock
    if len(existing) == 0 {
        return []byte(block)
    }
    return append([]byte(block), existing...)
}
```

This is a pure function with no I/O — fully unit-testable without filesystem.

**Why identity `includeIf` blocks are unaffected:** Identity blocks use `ReplaceBlock` with append-on-first-write. Since the baseline `[include]` is prepended (placed first), all identity blocks added later via `ReplaceBlock` land AFTER the include. Under git's last-wins rule, identity fragment values (name, email, signingkey) override the baseline's default keys — correct behavior.

**Impact on existing Phase 2/3 code:** Zero. `PrependBlockIfNotFound` is a new additive function; `ReplaceBlock` is unchanged.

### C2: Conflict Scan Cannot Use `git config --file --list` Alone

**Verified by:** Live test: `git config --file ~/.gitconfig --list` returns ALL keys in the file including those physically inside `# BEGIN gitid managed: …` comment blocks.

Example: a file containing both a user's `[core] ignorecase = true` and a gitid managed block with `[core] ignorecase = false` produces **two** `core.ignorecase` lines in `--list` output. `git config` has no concept of sentinel comment boundaries.

**Correct conflict scan algorithm (D-11):**
1. Read `~/.gitconfig` as bytes.
2. `filewriter.ListBlocks(content)` → get all managed block names.
3. For each managed block name, call `filewriter.RemoveBlock` to build the "user-owned portion" (bytes with all managed blocks stripped).
4. Write the user-owned portion to a temp file (or parse it in-memory as key=value pairs using a line scanner).
5. `git config --file <temp> --list` → enumerate user-owned keys.
6. Intersect with the baseline key set → conflicts.
7. For each conflict: report `user=<their value>`, `baseline=<our value>`, `winner=user (floor ordering)`.

Step 4 can avoid a temp file by line-scanning the "user portion" bytes directly in Go (the gitconfig format's key=value lines are parseable with `strings.SplitN(line, "=", 2)` after stripping `[section]` context — same pattern as `ReadFragment`).

### C3: Baseline and URL-Rewrites Blocks Are Raw Text, Not `git config set`

`git config --file path key value` writes keys without awareness of sentinel comment lines. If called on a file containing `# BEGIN gitid managed: baseline` … `# END gitid managed: baseline`, git would insert/update keys **inside the block boundaries** (because comments are not meaningful to git), potentially corrupting the block's structure or writing duplicate keys.

The correct approach (same as `includeIf` blocks): render the entire block body as a Go string and write it via `ReplaceBlock`. This is already proven by `WriteIncludeIf`:
```go
body := renderBlockBody(...)    // pure string render
composed := filewriter.ReplaceBlock(existing, identity, body)
filewriter.Write(gitconfigPath, composed, gitconfigMode)
```

The baseline renderer must follow this exact pattern.

**Idempotency requirement:** The rendered block body must be **byte-stable** across runs (same Go constant key ordering, same tab indentation, same alias quoting). Use a fixed string constant or a deterministic template — no map iteration, no runtime ordering.

Verified idempotent format (tab-indented, consistent with git's own output for `git config --file --list` cross-check):
```
[core]
\tignorecase = false
\texcludesfile = ~/.gitignore_global
\tautocrlf = input
\tpager = less -FRX
[push]
\tautoSetupRemote = true
...
```

### C4: `merge.conflictstyle=zdiff3` Requires git 2.35+

**Verified by:** `git config --file <tmp> merge.conflictstyle zdiff3` succeeds on git 2.47.0 (this machine). The `zdiff3` value was introduced in git 2.35 (released January 2022).

**Risk assessment:** git 2.35 is 4+ years old. On macOS the system git (Apple's) is typically 2.39+; on common Linux distros (Ubuntu 22.04+, Fedora 36+) it is 2.35+. Older Ubuntu LTS (20.04) ships git 2.25.

**Planner decision:** Either (a) gate on `git --version` at setup time and omit `merge.conflictstyle=zdiff3` when git < 2.35 (falling back to `diff3`), or (b) include it unconditionally given the 4-year-old threshold and document the minimum git version. Given the project already requires git for `TestSSH`, and the project targets macOS + Linux (Phase 6), option (b) is reasonable. The planner should document this in the baseline command's help text or README.

### C5: `core.autocrlf=input` Is Safe on macOS/Linux

**Verified:** `core.autocrlf=input` means:
- On commit: normalize CRLF → LF (prevents Windows line endings from being committed).
- On checkout: no conversion (LF stays LF).

On macOS/Linux repos where files already have LF endings, `autocrlf=input` is a no-op-at-checkout with a safety net at commit. It does NOT rewrite existing checked-out files. **Safe as a default.** [VERIFIED: live git test]

### C6: `git config --file` Does NOT Follow `[include]` by Default

**Verified by:** Live test comparing `git config --file <path> --list` vs `git config --file <path> --list --includes`:

- Without `--includes`: returns only keys physically in `<path>`, NOT keys from `[include]`-referenced files.
- With `--includes`: follows includes recursively.

**Implications for conflict scan:** When scanning `~/.gitconfig` for user-owned keys (C2 algorithm), use `git config --file ~/.gitconfig --list` (WITHOUT `--includes`) to avoid picking up baseline keys from the already-wired `~/.gitconfig.d/00-baseline`. Then subtract managed-block keys as described in C2.

**Implications for SC-5 read-back:** To verify a baseline key is effectively active (e.g. `git config --get core.ignorecase` resolves to the expected value), use `git config --global --get <key>` (which DOES follow includes), not `--file`. But SC-5 only requires showing the managed state from disk (what gitid wrote), not the effective resolved value — so `ListBlocks` on the three files is sufficient.

---

### C7: `insteadOf` Blast Radius and Failure Modes

**Verified by:** Live test of `git config --file --get-regexp "url\..*\.insteadof"` pattern.

`insteadOf` performs a **prefix replacement** on ALL git URLs globally — not just for `git clone`. Every git operation that resolves a URL (`fetch`, `push`, `ls-remote`, submodule updates, `go get`, `cargo fetch`, `npm install` for GitHub-hosted packages) is affected.

**Confirmed failure modes:**
1. **GitHub Actions / CI with `GITHUB_TOKEN`:** CI pipelines authenticate via HTTPS using a short-lived token embedded in the URL (e.g. `https://x-access-token:${GITHUB_TOKEN}@github.com/...`). An `insteadOf` rule rewrites this to SSH — auth fails because no SSH key is loaded in the runner.
2. **Go module proxy / `go get`:** Packages hosted on GitHub (`github.com/...`) fetched via `go get` use HTTPS by default. After rewrite, Go's module fetcher attempts SSH, which fails unless SSH agent is running with the key.
3. **Read-only HTTPS public clones:** `git clone https://github.com/public/repo` no longer works as anonymous — SSH requires a registered key.
4. **Longest-match wins:** git picks the `insteadOf` value whose prefix is longest. Two conflicting rules for the same host are not an error — the longer prefix wins silently.

**The preview must explicitly surface these risks before confirm (SC-3 / D-06).** Each mapping should show the host and warn: "All HTTPS operations for this host will require SSH auth."

**`git config --get-regexp` read pattern for read-back:**
```
git config --file ~/.gitconfig.d/00-baseline --get-regexp "url\..*\.insteadof"
```
Output format: `url.git@github.com:.insteadof https://github.com/` (key=value, case-folded key).

### Pattern 1: Prepend-If-Not-Found for Floor Placement

**What:** Write the managed `[include]` block at the TOP of `~/.gitconfig` on first write; update in-place on subsequent writes. Ensures included values are floors (user keys below WIN).

**When to use:** Exclusively for the `baseline-include` block in `~/.gitconfig`. All other blocks (identity `includeIf`, `url-rewrites`, `baseline`, `gitignore`) use standard `ReplaceBlock` (append-on-first).

```go
// Source: verified by live test — git evaluates [include] in-place
func PrependBlockIfNotFound(existing []byte, name, blockBody string) []byte {
    beginMarker := BeginPrefix + name
    if bytes.Contains(existing, []byte(beginMarker)) {
        return ReplaceBlock(existing, name, blockBody)
    }
    block := buildCanonicalBlock(name, blockBody) // same form as ReplaceBlock
    if len(existing) == 0 {
        return []byte(block)
    }
    return append([]byte(block+"\n"), existing...)
}
```

### Pattern 2: Fixed-Order Raw Text Render for Baseline Block

**What:** Render the baseline block body as a deterministic Go string (fixed key ordering, tab indent, no map iteration). This guarantees byte-identical output on every call with the same config, so `ReplaceBlock` is a true no-op on re-runs.

**Example (SC-1 idempotency contract):**
```go
// Source: gitconfig format verified via live git config --file --list cross-check
func RenderBaselineBlock(cfg BaselineConfig) string {
    var b strings.Builder
    fmt.Fprintf(&b, "[core]\n")
    fmt.Fprintf(&b, "\tignorecase = false\n")
    fmt.Fprintf(&b, "\texcludesfile = ~/.gitignore_global\n")
    if cfg.AutoCRLF {
        fmt.Fprintf(&b, "\tautocrlf = input\n")
    }
    if cfg.Pager != "" {
        fmt.Fprintf(&b, "\tpager = %s\n", cfg.Pager)
    }
    // ... fixed section ordering: [core], [push], [pull], [fetch], [color], [diff], [merge], [init], [alias]
    return strings.TrimRight(b.String(), "\n")
}
```

The section ordering must match the gist reference: `[core]`, `[push]`, `[pull]`, `[fetch]`, `[color]`, `[diff]`, `[merge]`, `[init]`, `[alias]`.

### Pattern 3: Conflict Scan via Text + git config

**What:** Isolate user-owned keys in `~/.gitconfig` and compare against the baseline key set.

```go
func ScanConflicts(gitconfigPath string, baselineKeys map[string]string) ([]Conflict, error) {
    content, _ := os.ReadFile(gitconfigPath)
    // Strip ALL gitid managed blocks to get user-owned portion
    userPortion := content
    for _, block := range filewriter.ListBlocks(content) {
        userPortion = filewriter.RemoveBlock(userPortion, block.Name)
    }
    // Write user portion to temp file for git config parsing
    tmp, _ := os.CreateTemp("", "gitid-conflict-*.gitconfig")
    tmp.Write(userPortion); tmp.Close()
    defer os.Remove(tmp.Name())
    // Read user-owned keys
    out, _ := exec.Command("git", "config", "--file", tmp.Name(), "--list").Output()
    userKeys := parseKeyValuePairs(out)
    // Find overlaps
    var conflicts []Conflict
    for k, baseline := range baselineKeys {
        if user, ok := userKeys[k]; ok {
            conflicts = append(conflicts, Conflict{Key: k, UserValue: user, BaselineValue: baseline, Winner: "user"})
        }
    }
    return conflicts, nil
}
```

### Anti-Patterns to Avoid

- **Using `git config --file path key value` to write baseline keys:** Destroys sentinel comment integrity; git inserts keys without knowing about block boundaries.
- **Appending the `[include]` block via `ReplaceBlock`:** Produces CEILING semantics (baseline overrides user), not FLOOR. Must use `PrependBlockIfNotFound`.
- **Non-deterministic block body render (e.g., ranging over a map):** Different key ordering on each run makes `ReplaceBlock` non-idempotent (re-writes a "different" block even when the config is unchanged).
- **Using `git config --file --list --includes` for conflict scan:** Follows includes, so baseline keys from `00-baseline` appear in the output — incorrectly flagged as user-set conflicts.
- **Checking `merge.conflictstyle=zdiff3` without a git version gate:** Silent failure on git < 2.35 (git ignores the unrecognized value, falls back to default `merge` style without error).

## Common Pitfalls

### Pitfall A: Append-at-Bottom Creates Ceiling, Not Floor

**What goes wrong:** If `WriteBaselineInclude` uses `ReplaceBlock` on a non-empty `~/.gitconfig`, the `[include]` block lands at the end. git evaluates it last → included values override any user key set earlier in the file. This silently imposes the baseline's `core.ignorecase=false` even on users who explicitly set `core.ignorecase=true`.

**Root cause:** `ReplaceBlock` appends when block not found. `git config` uses last-wins per key, in-place evaluation of `[include]`.

**Prevention:** Use `PrependBlockIfNotFound` for the `baseline-include` block only.

**Warning sign:** After setup, `git config --global --get core.ignorecase` returns `false` even though user had `ignorecase = true` in their `~/.gitconfig` above the include.

---

### Pitfall B: `git config --file` Writes Corrupt Sentinel Blocks

**What goes wrong:** Calling `gitConfigSet(baselineFilePath, "core.ignorecase", "false")` (the function in `fragment.go`) on `~/.gitconfig.d/00-baseline` inserts the key inside the sentinel block boundaries — but git doesn't know about the comment markers. On the next `ReplaceBlock` run, the body has extra keys that don't match the expected rendered body → the block is rewritten on every run (non-idempotent).

**Prevention:** ALL writes to `~/.gitconfig.d/00-baseline` must go through `RenderBaselineBlock` + `ReplaceBlock` only. Never call `gitConfigSet` on this file.

---

### Pitfall C: `git config --file --list` Reports Managed-Block Keys as User Keys

**What goes wrong:** The conflict scan reads `git config --file ~/.gitconfig --list` and sees `core.ignorecase=false` (from inside the `baseline-include`'s managed block for the `[include]` reference). It incorrectly reports this as a user-set conflict.

**Prevention:** Strip ALL managed blocks from the file bytes (via `ListBlocks` + `RemoveBlock` for each name) BEFORE passing to `git config --list`. The conflict scan must operate on user-owned content only.

---

### Pitfall D: Non-Deterministic Alias Render Breaks Idempotency

**What goes wrong:** Rendering aliases by ranging over a `map[string]string` produces different orderings on different runs. `ReplaceBlock` finds the block exists but the body has changed → rewrites the file on every run, producing a new timestamp backup each time. SC-1 ("second run produces no diff") is violated.

**Prevention:** Render aliases in a fixed order (slice of pairs or explicit section). The canonical alias order from the gist: `st`, `co`, `br`, `ci`, `df`, `lg`, `unstage`, `last`.

---

### Pitfall E: `core.excludesfile = ~/.gitignore_global` in `00-baseline` Is a FLOOR

**What goes wrong:** A user who already has `core.excludesfile = ~/.gitignore` in their `~/.gitconfig` has their setting ABOVE the `[include]`. Under floor ordering, user's value wins — `core.excludesfile` resolves to their value, not `~/.gitignore_global`. The global gitignore gitid manages is silently ignored by git.

**Prevention:** The conflict scan MUST flag `core.excludesfile` as a conflict if the user already has it set. The preview must show: `current=~/.gitignore`, `baseline=~/.gitignore_global`, `winner=user (floor ordering)`. The user must explicitly choose to remove their override if they want gitid's gitignore to take effect.

---

### Pitfall F: `insteadOf` Breaks Package Manager Fetches via HTTPS

**What goes wrong:** A global `url "git@github.com:" insteadOf = https://github.com/` rewrite affects `go get github.com/...`, `cargo fetch` for crates with GitHub sources, and CI pipelines using token-auth HTTPS. Users who enable the rewrite and then run `go get` in a terminal (without SSH agent loaded) get `git@github.com: Permission denied (publickey)` from Go module fetcher — cryptic error with no git mention.

**Prevention:** The preview for URL rewrites must explicitly warn: "This affects ALL HTTPS operations for `github.com` — including `go get`, `npm install`, `cargo fetch`, and CI pipelines using GITHUB_TOKEN auth. Make sure your SSH agent is running with the relevant key."

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `merge.conflictstyle=diff3` | `merge.conflictstyle=zdiff3` | git 2.35 (Jan 2022) | `zdiff3` shows original base in conflict markers; safer default for complex merges |
| `push.default=simple` | `push.autoSetupRemote=true` | git 2.38 (Oct 2022) | Eliminates `--set-upstream` manual step |
| `core.autocrlf=false` | `core.autocrlf=input` | Long-standing; best practice | Prevents CRLF from being committed on macOS/Linux; safer for cross-platform repos |
| Inline keys in `~/.gitconfig` | `[include]` to a separate baseline file | Established pattern | Keeps `~/.gitconfig` minimal; baseline is independently replaceable |

**Deprecated:**
- `[url] pushInsteadOf` — a different rewrite that applies only on push; `insteadOf` applies on all operations including fetch. `insteadOf` is the current standard for HTTPS→SSH rewrites.
- `merge.conflictstyle=diff3` — superseded by `zdiff3` which adds the original in the conflict marker.

## Open Questions (RESOLVED)

1. **git version gate for `zdiff3`**
   - What we know: `zdiff3` requires git ≥ 2.35; git on this machine is 2.47.0 (fine); Ubuntu 20.04 ships 2.25 (would silently ignore).
   - What's unclear: Does the project target Ubuntu 20.04 LTS? (Phase 6 deferred, but Phase 4/5 run on macOS.)
   - Recommendation: Planner includes a `git --version` check; omit `zdiff3` if git < 2.35, use `diff3` instead. Add this as a wave-0 utility in `internal/deps`.

2. **Tilde expansion in `include.path`**
   - What we know: git expands `~` in `include.path` at runtime (confirmed by live test with absolute path; tilde form is standard in the reference gist).
   - What's unclear: Does gitid write the literal `~` or the expanded absolute path?
   - Recommendation: Write literal `~/.gitconfig.d/00-baseline` (same as identity fragments write `~/.gitconfig.d/<name>`). Confirmed safe.

3. **Where exactly is "near the top" of `~/.gitconfig`?**
   - What we know: `PrependBlockIfNotFound` places it at line 0 on first write; on re-runs, in-place splice preserves position.
   - What's unclear: If the user already has content at the top (e.g., a `[user]` section), the prepend will push it down. This is intentional — the include must be first.
   - Recommendation: The preview should show the resulting `~/.gitconfig` structure including where the include block will be inserted, so the user can see the reordering.
