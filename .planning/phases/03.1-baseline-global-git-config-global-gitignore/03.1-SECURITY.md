## Note on T-03.1-10 rollback restore exception

`cmd/gitid/baseline.go:219-234` `restoreSnapshot` is the only direct `os.WriteFile`
in cmd for a managed surface. Judgment: **acceptable exception, not a finding.**

- It is a **revert**, not a fresh mutation: it writes back the exact bytes captured
  by `snapshotFile` (line 200-213) before any write began, preserving the original
  `snap.mode` (line 227) and re-asserting it via `os.Chmod` (line 230).
- It cannot widen permissions: the restored mode is the file's own pre-existing
  `info.Mode().Perm()`.
- It only runs on the partial-failure rollback path (`WriteBaselineFile` /
  `WriteBaselineInclude` error), restoring a surface that `filewriter.Write` had
  already mutated, so no new backup semantics are needed.
- It does not bypass the validation chokepoint — the content being written is prior
  on-disk bytes, never new user input.

This is the CR-01 fix from `03.1-REVIEW.md`. The prior review's Critical (multi-surface
write with no rollback) and the include-path/WR-01 bug are both corrected on main and
verified present in the shipped code; no mitigation was weakened by the fixes.
