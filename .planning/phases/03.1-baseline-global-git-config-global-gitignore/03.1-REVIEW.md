# Phase 03.1: Code Review Report

**Reviewed:** 2026-06-11T00:00:00Z
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Phase 03.1 implements a global baseline git-config + global gitignore in idempotent,
sentinel-delimited managed blocks, with HTTPS→SSH `insteadOf` rewrites, all routed
through the `filewriter.Write` chokepoint (backup → atomic temp → rename → chmod).

The security-sensitive surfaces hold up well on the dimensions the prompt flagged:
exec calls use the arg-slice form with no shell (`deps.go`, `ScanConflicts`); user-editable
`insteadOf` URLs, pager, conflictstyle, and default-branch all route through
`validateValue` before render; writes go through the chokepoint with explicit modes;
and the renderers are byte-stable (no map iteration in output paths). No command
injection, path traversal, or hardcoded-secret defects were found.

The defects that remain are correctness and robustness issues, the most serious being
a **multi-file partial-write hazard** in `runBaselineSetup`: a confirmed write that
fails midway leaves the three managed surfaces inconsistent with no rollback and no
cleanup, contradicting the CLAUDE.md backup-before-write safety contract for the
combined operation. Several WARNINGs concern logic correctness (conflict over-reporting,
a silently-ignored function parameter that decouples a write path from its input,
a parser that mis-handles values containing ` = `, and `validateValue`/panic coupling
that can crash the CLI on malformed config).

## Critical Issues

### CR-01: Multi-surface write has no rollback — a mid-sequence failure leaves the baseline inconsistent

**File:** `cmd/gitid/baseline.go:119-137`
**Issue:** After the single confirm gate, `runBaselineSetup` writes three independent
files in sequence — `WriteGlobalGitignore`, then `WriteBaselineFile`, then
`WriteBaselineInclude`. Each is its own backup-and-atomic-write, but there is **no
transaction boundary across the three**. If `WriteBaselineFile` or `WriteBaselineInclude`
returns an error (disk full, permission change, `~/.gitconfig.d` made read-only, signal),
the handler returns the error and aborts — leaving `~/.gitignore_global` already mutated
(and backed up) while `~/.gitconfig` was never given its `[include]` block. The user is
left in exactly the `Incomplete` state that `ReadBaselineState` is designed to detect,
but with no automatic recovery and no message telling them which surface succeeded.

Worse, the dangerous ordering is the *opposite* of safe: the include block (the line
that actually activates the baseline) is written **last**. The more typical hazard is
the reverse — but here, if the baseline file write succeeds and the include write fails,
git silently ignores a fully-written baseline. The user believes "Baseline written." was
never printed, but partial side effects (a new `~/.gitignore_global`, a backup file)
persist. This violates the CLAUDE.md contract that mutations to the user's files happen
as an all-or-nothing, backed-up operation.

**Fix:** Make the three writes recoverable as a unit. Minimum viable fix: on any write
error after the first surface has been mutated, attempt to restore from the backup paths
already returned, and surface a clear message naming what was rolled back. At minimum,
report partial success explicitly so the user knows to re-run:
```go
gitignoreBackup, err := gitconfig.WriteGlobalGitignore(absGitignore, patterns)
if err != nil {
	return fmt.Errorf("baseline setup: writing %s: %w", absGitignore, err)
}
baselineBackup, err := gitconfig.WriteBaselineFile(absBaseline, cfg, rewrites)
if err != nil {
	// gitignore already mutated — restore it before bailing.
	if rerr := restoreBackup(absGitignore, gitignoreBackup); rerr != nil {
		return fmt.Errorf("baseline setup: writing %s failed (%v) AND rollback of %s failed: %w",
			absBaseline, err, absGitignore, rerr)
	}
	return fmt.Errorf("baseline setup: writing %s (rolled back gitignore): %w", absBaseline, err)
}
// ...same pattern for WriteBaselineInclude, restoring both prior surfaces.
```
If full rollback is out of scope for this phase, at least write the include block FIRST
or print a recovery hint ("partial write — run `gitid baseline setup` again to repair")
on the error path.

## Warnings

### WR-01: `WriteBaselineInclude` ignores its `baselineFilePath` argument and hardcodes the path

**File:** `internal/gitconfig/baseline.go:589-594`
**Issue:** The function signature accepts `baselineFilePath string` but the body
discards it (`_ = baselineFilePath`) and writes a hardcoded literal
`"[include]\n\tpath = ~/.gitconfig.d/00-baseline"`. This is a silent contract trap:
a caller that passes a *different* baseline path (the tests in
`baseline_test.go:717` pass `"~/.gitconfig.d/00-baseline"`, but other call sites or
future refactors could pass an absolute path or a custom location) will get a config
that points somewhere it did not ask for, with no error. The parameter exists purely as
documentation, which is misleading API design and a real bug waiting for the first caller
that deviates from the assumed home layout.
**Fix:** Either remove the parameter entirely (and document the path as a fixed constant),
or actually honor it by deriving the include body from the argument after validating it
through `validateValue`. Removing it is the safer choice given the floor-model assumes a
fixed location:
```go
func WriteBaselineInclude(gitconfigPath string) (string, error) {
	const includeBody = "[include]\n\tpath = ~/.gitconfig.d/00-baseline"
	// ...
}
```

### WR-02: `ScanConflicts` reports a conflict even when the user's value equals the baseline value

**File:** `internal/gitconfig/baseline.go:125-134`
**Issue:** The conflict loop emits a `Conflict` whenever a baseline key is *present* in
the user-owned portion, with no comparison of the values. A user who legitimately set
`pull.rebase = true` or `color.ui = auto` (identical to the baseline) is shown a spurious
"conflict ... winner=yours" warning in the preview. This is misleading noise that
undermines the signal of the conflict section — the whole point of which is to flag
*divergent* overrides. It also inflates the `excludesfile` special-case warning path.
**Fix:** Skip the conflict when values match:
```go
for baseKey, baseVal := range baselineKeys {
	if userVal, ok := userKeys[baseKey]; ok && userVal != baseVal {
		conflicts = append(conflicts, Conflict{
			Key: baseKey, UserValue: userVal, BaselineValue: baseVal, Winner: "user",
		})
	}
}
```

### WR-03: `RenderBaselineBlock` / `RenderURLRewritesBlock` panic on invalid input — a setup run can crash the CLI

**File:** `internal/gitconfig/baseline.go:392-406, 489-494`
**Issue:** When `validateValue` rejects a Pager / MergeConflictStyle / InitDefaultBranch /
URL value, the renderers `panic(...)` instead of returning an error. These renderers are
called from `printBaselinePreview` (`cmd/gitid/baseline.go:168,172`) on the *uncontrolled*
preview path before any confirm. The default values are safe, so this never fires today —
but the moment any of these fields becomes user-editable (the prompts at
`baseline.go:96-104` already let the user reshape `rewrites`, and the struct fields are
public), a crafted or pasted value with a newline crashes the whole process via panic
rather than a clean validation error. Panicking across a package boundary for *expected*
invalid input is the wrong control-flow mechanism for a tool that mutates user files.
**Fix:** Change the renderers to return `(string, error)` and propagate validation
failures to the command handler, which already has an error path:
```go
func RenderBaselineBlock(cfg BaselineConfig) (string, error) {
	if cfg.Pager != "" {
		if err := validateValue("core.pager", cfg.Pager); err != nil {
			return "", err
		}
	}
	// ...
}
```

### WR-04: `parseGitconfigBlockBody` corrupts any value that contains ` = `

**File:** `internal/gitconfig/baseline.go:255-259`
**Issue:** The body parser splits each line on the first ` = ` (`strings.SplitN(trimmed, " = ", 2)`).
Git alias/value strings legitimately contain ` = ` — e.g. a user (or a future baseline
addition) with `alias.foo = '!f() { x = y; }; f'` or a pager/command containing ` = `
would be parsed with a truncated value, or for the section-header detection, any indented
line that happens to start with `[` and end with `]` is treated as a section. The current
default `lg` alias narrowly avoids ` = ` so the test passes, but this is a fragile parser
that silently mis-reads `gitid baseline show` output the moment a value contains the
delimiter. Round-trip fidelity is a stated goal of this codebase.
**Fix:** Split on the first `=` and trim, rather than requiring the ` = ` spacing, and
guard the section-header check so it only matches at indent level zero (no leading tab):
```go
if !strings.HasPrefix(line, "\t") && strings.HasPrefix(trimmed, "[") && strings.HasSuffix(trimmed, "]") {
	section = strings.ToLower(trimmed[1 : len(trimmed)-1])
	continue
}
if eq := strings.Index(trimmed, "="); eq != -1 && section != "" {
	key := strings.ToLower(section + "." + strings.TrimSpace(trimmed[:eq]))
	result[key] = strings.TrimSpace(trimmed[eq+1:])
}
```

### WR-05: `ScanConflicts` surfaces every git parse error as a hard failure that blocks setup

**File:** `internal/gitconfig/baseline.go:106-111`
**Issue:** After writing the user-owned portion to a temp file, a non-zero exit from
`git config --file ... --list` is returned as an error, which propagates up through
`runBaselineSetup` (`cmd/gitid/baseline.go:73-76`) and aborts the entire setup. But the
user-owned portion is *the user's existing config minus managed blocks* — if the user has
a pre-existing malformed `~/.gitconfig` (a stray line, a section git dislikes), the tool
refuses to run `baseline setup` at all, even though setup never needed to parse that part
to do its job. A pre-existing user error should not brick an unrelated feature.
**Fix:** Distinguish "git couldn't parse" from "git failed to execute". Treat a parse
failure as "no detectable conflicts" (degrade gracefully, optionally warn) rather than a
fatal error, or scope the error so the preview can still proceed:
```go
out, err := cmd.Output()
if err != nil {
	// User's pre-existing config is malformed — we can't compute conflicts,
	// but that must not block setup of the managed blocks.
	return nil, nil // or return a sentinel the caller renders as "conflict scan skipped"
}
```

### WR-06: Prompt/confirm read errors are silently discarded

**File:** `cmd/gitid/baseline.go:337` (and `confirm` at `add.go:514`)
**Issue:** `promptYN` does `line, _ := r.ReadString('\n')`, discarding the error. On a
read error mid-stream (not just clean EOF), `promptYN` falls through to its default-Y
return, meaning a *failed read* is interpreted as the user *accepting* Tier-2 defaults and
each URL rewrite. For an opt-out flow that mutates `~/.gitconfig` and rewrites all HTTPS
traffic for three hosts, defaulting to "yes" on an I/O error is the unsafe direction.
**Fix:** Treat a non-EOF read error as a decline / abort, or at minimum check for it:
```go
line, err := r.ReadString('\n')
if err != nil && err != io.EOF {
	return false // fail safe: do not silently accept an opt-out default on a read error
}
```

### WR-07: First-write include branch in `PrependBlockIfNotFound` is dead under the only real caller

**File:** `internal/filewriter/block.go:153-155`
**Issue:** The `if len(existing) == 0 { return []byte(block) }` branch is functionally
redundant with the prepend branch below it (`append([]byte(block), existing...)` with an
empty `existing` yields the same bytes). More importantly, the only production caller,
`WriteBaselineInclude`, reads via `os.ReadFile` which returns `nil` for a missing file but
the file is then created — the branch is covered only by a unit test, not a real path. Not
a correctness bug today, but it is untested-in-context special-casing that invites drift if
the canonical block form ever changes (the two branches would have to be kept in sync).
**Fix:** Remove the redundant empty-input branch and rely on the single prepend path, or
add a comment proving the two forms are byte-identical and keep one. Consolidating reduces
the idempotency surface:
```go
// First write (with or without existing content): prepend the canonical block.
return append([]byte(block), existing...)
```

## Info

### IN-01: `printBaselineKeys` maintains two parallel hardcoded key lists that can silently drift

**File:** `cmd/gitid/baseline.go:271-321`
**Issue:** `orderedKeys` (22 entries) and `displayKeys` (22 entries) are hand-maintained
in lockstep, and both must stay in sync with `RenderBaselineBlock`'s section order. Adding
a baseline key requires editing three places; a miss silently drops the key from
`baseline show`. This is a maintainability hazard, not a current bug.
**Fix:** Derive display names from the ordered slice via a single source of truth (e.g. a
slice of `{key, display}` structs), or generate the order from the renderer.

### IN-02: `tildePath` calls `os.UserHomeDir()` on every invocation

**File:** `cmd/gitid/baseline.go:352-361`
**Issue:** `tildePath` is called many times per `show`/summary render, each re-resolving
the home dir. Correctness is fine (the result is stable), but it is repeated work and a
second silent error-swallow point (`if err != nil { return path }`). Resolve home once in
the caller and pass it down.

### IN-03: `_ = baselineFilePath` documentation-only parameter is a code smell

**File:** `internal/gitconfig/baseline.go:593`
**Issue:** Assigning a parameter to the blank identifier "as documentation" is a
linter-suppression smell that signals the unused-but-present design problem flagged in
WR-01. Listed separately because even after WR-01 is resolved, reviewers should treat
`_ = param` lines as a prompt to fix the signature, not silence the linter.

### IN-04: `RemoveURLRewritesBlock` and `BackupAndRemove` are unused by the phase 03.1 paths

**File:** `internal/gitconfig/baseline.go:142-157`
**Issue:** `RemoveURLRewritesBlock` is exported and tested but is not called by
`runBaselineSetup` or `runBaselineShow` — the setup path removes the url-rewrites block
inline via `WriteBaselineFile`'s `RemoveBlock` branch (`baseline.go:539`). It may be
intended for a future `baseline edit`/`unset` command. Confirm it is reachable from a
command before merge, or mark it clearly as a building block, to avoid dead exported API.
