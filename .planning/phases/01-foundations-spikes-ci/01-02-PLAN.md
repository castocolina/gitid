<objective>
Refactor `internal/keygen` from the hard-coded `if p.Algo != "ed25519"` dispatch into a name-keyed algorithm registry that generates real ed25519 (default) and rsa-4096 keys and registers ecdsa-p256/ed25519-sk/ecdsa-sk as explicit not-yet-implemented stubs (KEY-02, D-05/D-06). Add the top-5 algorithm catalog (KEY-01) with security + per-OS availability/variant metadata and a `ResolveAvailability` function that cross-references the catalog against platform-probe results passed in as plain data (no `internal/platform` import — avoids coupling and keeps this plan parallel with 01-01). Re-verify correct key-file permissions across the registry refactor (KEY-04). Per Codex review: assert the generation path can NEVER select a registered-but-stubbed algorithm as if it were generatable (registry presence must not imply generation support).

Purpose: The create flow (Phase 3), rotate/new-key (Phase 5), and the debug/list command (01-06) all generate keys by algorithm name and read the catalog. Final catalog ordering/copy is deferred to Phase 2 design.
Output: `internal/keygen/registry.go`, `catalog.go`, a registry-dispatching `keygen.go` with `generateRSA4096`, and extended tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| keygen (in-memory key material) → caller/filesystem | private key bytes (PrivPEM) are generated in memory and must not leak to logs; only the caller persists them via the filewriter chokepoint |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Information Disclosure | rsa-4096 Material.PrivPEM | mitigate | PrivPEM is in-memory only; the existing "must never be logged or printed" doc contract is preserved; no debug printing added here (the debug command in 01-06 prints catalog metadata only) |
| T-01-05 | Information Disclosure | generated rsa-4096 key file perms | mitigate | Generation routes through the existing `filewriter.Write(privPath, ..., 0600)` path; KEY-04 permission test re-asserts 600/644 after the registry refactor so no algorithm-specific perm regression slips in |
| T-01-06 | Tampering | crypto library usage (RSA/ed25519 serialization) | mitigate | Use only `golang.org/x/crypto/ssh` `MarshalPrivateKey`/`NewPublicKey` (never hand-rolled OpenSSH serialization); pin at v0.53.0 per CLAUDE.md |
| T-01-21 | Spoofing | stubbed algorithm presented as generatable | mitigate | Generatable requires Implemented AND Available; stub generators return a zero Material + named error; a test asserts every Implemented=false entry's generation path errors, so registry presence never implies generation support |
</threat_model>
