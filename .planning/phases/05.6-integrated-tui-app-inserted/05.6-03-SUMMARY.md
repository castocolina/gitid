# Phase 05.6 Plan 03: Health View + Fix Confirm Modal Summary

Vertical slice 2 — the Health view capability end-to-end. Ports the Phase 5.5 async per-family streaming dashboard into the persistent two-pane main pane as `healthViewModel`, fixes the severity glyph regression (warnings render `!` not `✗`), adds per-identity sidebar badges from the live doctor run, and implements in-app auto-fix with a modal confirm gate.

## What Was Built

### Tasks 1+2: Health view + confirm modal + wiring (one logical commit, f584d4d)

**internal/doctor/doctor.go (modified):**

- Added `IdentityName string` to `Finding` struct — the scoping field that enables per-identity badge derivation (D-08). Empty string = global finding (not attributed to any specific identity). Zero value is backward-compatible with all existing struct literals.

**internal/doctor/checks/coherence.go (modified):**

- All per-account findings in `coherenceForAccount` now set `IdentityName: acct.Name`. This covers: Incomplete, missing KeyPath, missing FragmentPath, IdentitiesOnly-missing, gpg.format-mismatch, allowed_signers-missing/mismatch. These are the findings most likely to drive per-identity badges.

**tui/health.go (new):**

- `healthViewModel` struct: `families []familyState`, `findings map[doctor.Family][]doctor.Finding`, `spinners []spinner.Model`, `width/height`, `runID`, `doctorDeps`, `deps`
- `newHealthModel(tuiDeps)`: dynamically sized from `len(doctor.Families())` — no fixed-array drift
- `familyIndex(fam)`: 0-based index lookup into `doctor.Families()`
- `init()`: seeds one family cmd + one spinner.Tick per family = 16-entry Batch (Pattern C mandatory Tick seed)
- `makeFamilyCmd(runID, fam, d)`: VERBATIM port from Phase 5.5 — `recover()` wrap converts panics to `familyResultMsg{err}` (T-05.6-08); nil-guard surfaces unwired checks as errors (never silent pass, T-05.6-09/D-16)
- `update(msg)`: handles `familyResultMsg` with stale-guard (runID check, Pitfall 4), `spinner.TickMsg` propagation, `tea.WindowSizeMsg`
- `refresh()`: increments runID, resets all families to loading, calls `init()` — stale-guard ensures old in-flight results are dropped
- `view(w, h)`: renders 8 family panels (spinner → "checking...", `✗ check failed`, or per-finding list); glyph contract D-10 locked via `SeverityGlyph()`
- `renderFinding(f)`: glyph from `SeverityGlyph(severity, asciiMode())` — warning `!` (yellow), error `✗` (red), info `~` (cyan); `[fix]` inline when `f.Fix != nil`; `Fix.Summary` on its own line BEFORE `[fix]` marker
- `badgesFromFindings(findings)`: derives worst severity per identity from `IdentityName` field; skips global findings (empty IdentityName); returns `map[string]doctor.Severity` for the sidebar

**tui/confirm.go (new):**

- `confirmKind` enum: `fixConfirm` (Plan 03), `deleteConfirm`/`rotateConfirm` declared but not driven until Plan 06
- `confirmModel` struct: `kind`, `finding`, `title`, `body`, `running bool`, `result string`, `sp spinner.Model`, `deps tuiDeps`
- `newConfirmModel(kind, finding, deps)`: pre-renders consequence + Fix.Summary into `body` so view is stateless
- `buildConfirmContent(kind, finding)`: renders "Fix: {summary}" BEFORE "[Enter to apply / Esc to cancel]" (D-09, Accessibility Contract item 5)
- `update(msg)`: Enter → sets `running=true`, dispatches `dispatchCmd()`; Enter when `running=true` → no-op (double-dispatch guard); Esc → emits `clearModalCmd()` (safe default, never writes)
- `fixResultMsg` handler: on success emits `tea.Batch(clearModalCmd(), makeFamilyCmd(...))` for live re-run; on failure sets `result` error string, keeps modal open until Esc
- `view(w)`: renders `StyleModal` box with title, consequence body (or spinner when running, or result after completion)
- `runFixCmd(finding, deps)`: goroutine with `recover()` wrap (T-05.6-08); calls `finding.Fix.Fn()` through the core (T-05.6-07 — no direct os.Chmod); emits `fixResultMsg{family, err}`

**tui/model.go (modified):**

- `modalKind` enum: added `fixConfirmModal` (Plan 03); `deleteConfirmModal`/`rotateConfirmModal` remain as reserved comments
- `rootModel` struct: added `health healthViewModel`, `healthReady bool`, `confirm confirmModel`
- `newRootModel()`: initializes `health: newHealthModel(d)` alongside existing fields
- `Update()`: new cases for `familyResultMsg` (→ health sub-model + badge update), `fixResultMsg` (→ confirm sub-model), `spinner.TickMsg` (→ health sub-model)
- `handleKey()`: `"2"` now seeds `health.init()` on first press (Pattern 5 — seed spinner ticks); `"r"` triggers `health.refresh()` in health view; `"x"` in health view calls `focusedFixableFinding()` and opens `fixConfirmModal`; `fixConfirmModal` case routes keys to `m.confirm.update()`
- `renderContent()`: handles `fixConfirmModal` → `m.confirm.view(m.width)`
- `renderMainPane()`: health view calls `m.health.view(w, h)` (replaces `renderHealthPlaceholder` stub)
- `focusedFixableFinding()`: iterates `health.findings` to find the first fixable finding

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical Functionality] Added Finding.IdentityName to doctor.Finding**
- **Found during:** Task 1 (badgesFromFindings implementation)
- **Issue:** The plan specifies "Determine the scoping field by reading the doctor Finding struct." The Finding struct had no structured identity-scoping field — only free-text Title/Explanation. Deriving identity names from text strings is fragile and incorrect.
- **Fix:** Added `IdentityName string` to `doctor.Finding` (additive, backward-compatible; zero value = global finding). Populated in `coherenceForAccount` for all per-account findings. This is a correctness requirement for D-08, not a feature.
- **Files modified:** internal/doctor/doctor.go, internal/doctor/checks/coherence.go
- **Commit:** f584d4d

**2. [Rule 1 - Bug] Removed unused `cursor` field from healthViewModel**
- **Found during:** `make lint` (golangci-lint unused linter)
- **Issue:** `cursor int` was declared in the struct but never written or read. The plan mentions a cursor for 'x' fix action, but `focusedFixableFinding()` iterates all findings instead of using an index. The field would cause a hard lint failure blocking the commit.
- **Fix:** Removed `cursor int` from the struct. The finding-selection logic uses `focusedFixableFinding()` which scans `health.findings` directly.
- **Files modified:** tui/health.go
- **Commit:** f584d4d

**3. [Rule 3 - Blocking] Tasks 1+2 collapsed into one logical commit per CLAUDE.md**
- **Found during:** Implementation
- **Issue:** `rootModel` in model.go references `healthViewModel` and `confirmModel` before health.go and confirm.go exist. Without both, model.go cannot compile and no commit can pass the pre-commit hooks.
- **Fix:** Implemented both tasks together as one logical commit (f584d4d). CLAUDE.md: "One coherent change — its implementation, its tests, and its docs — is a single commit."
- **Commit:** f584d4d
