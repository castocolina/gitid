# Phase 5.6 — Integrated TUI App: Expert UX Review

**Reviewer:** Senior UI/UX (research-backed critique)
**Date:** 2026-06-16
**Method:** e2e walkthrough of the real render code (`tui/*.go`) + two live screenshots (edit view, add wizard) on a ≥100-column terminal, cross-checked against the locked design contract (`05.6-CONTEXT.md` D-01..D-16, `05.6-UI-SPEC.md`).

## Verdict

The skeleton is right (header tabs, sidebar, master/detail, footer) but the first-run experience is broken in ways that read as "unfinished," not "minimal." Two of the eight findings are **render bugs that make the product look broken** (phantom bordered boxes on every form field; truncated footer that hides the only "add identity" affordance). The rest are **discoverability and sensible-defaults failures**: the user cannot tell how to switch views, the first identity isn't really selected, `Enter` is a dead key, and the sidebar lies about its scope in Health/Global. None of this is a framework limitation — it is wiring, defaults, and layout. The good news: every finding is a contained delta against an otherwise-coherent contract, and most of the worst ones are *already specified correctly in the UI-SPEC* and simply not implemented (empty-state copy, initial selection, footer truncation priority, unmanaged section). The implementation drifted from its own spec.

## Findings (prioritized)

### P0-1 — Phantom empty bordered boxes on every form field

- **Observed:** In both the inline-edit detail pane and the create wizard, every field renders a *second*, empty, bordered rectangle floating up-and-to-the-right of the actual value box. The real value sits in a small box at the label; a larger empty box hovers offset above it. This is the single most damaging visual — it reads as a rendering crash.
- **UX principle violated:** Nielsen #1 *Visibility of system status* and #8 *Aesthetic and minimalist design*. Visual noise that carries no meaning destroys trust; users infer "this is broken" within the 50 ms credibility window (Lindgaard et al., 2006). A control that draws two boxes has no recognizable affordance (Norman, *signifiers*).
- **Root-cause hypothesis (two compounding bugs):**
  1. `tui/detail.go:view` (lines 157-163) and `tui/wizard.go:viewForm` (lines 814-825) wrap `inp.View()` in `StyleInputActive`/`StyleInputInactive`, which add a `RoundedBorder()`/`NormalBorder()` (`tui/styles.go:36-43`). The `textinput.Model` instances are created with `textinput.New()` and **never given a width** (`tui/wizard.go:414-434`, `tui/detail.go:269-277` — no `SetWidth`/`Width` call). bubbles v2 `textinput.View()` emits its own internally-padded/styled segment; bordering that variable-width segment per-field produces a box whose width does not match the label row, so the border corners land at offset columns.
  2. `tui/overlay.go:overlayLine` (lines 84-121) composites the modal with a **rune-for-rune copy that is not ANSI-aware**: it copies `len(fgRunes)` runes (which includes multi-rune ANSI escape sequences) while only `lipgloss.Width(fgLine)` of them are visible. The escape bytes are written into background cells, and the visible-column math drifts — so the bordered fragment is laid down at the wrong x. The file even self-documents this as "lossy" (lines 76-83). This explains why the empty box is *offset* rather than merely adjacent.
- **Recommendation:**
  1. Give every `textinput.Model` an explicit width and a plain prompt, and stop double-bordering. Set `ti.Prompt = "> "` once, call `ti.SetWidth(mainPaneWidth - labelWidth - 4)` (or a fixed `40` in the modal), and render the focused/blurred distinction with **border color only on a single, fixed-width box** — never wrap the already-styled `inp.View()` output in a second variably-sized bordered style. Concretely: build the box with `StyleInputActive.Width(fieldW).Render(inp.View())` where `inp` has its own width *smaller than* `fieldW`, so exactly one border is drawn at a known width.
  2. Replace `overlayLine`'s rune copy with a proper ANSI-aware cell composit (measure with `lipgloss.Width`, advance background by visible cells, and splice `fgLine` as an opaque styled run rather than per-rune). Or — simpler and far less risky — **stop hand-rolling overlay compositing** and use `lipgloss.Place` over the dimmed layout for centering (the spec already references `lipgloss.Place()`), which sidesteps the ANSI splicing bug entirely.
- **Contract:** Fits within D-02 (modal over dimmed layout). The hand-rolled `placeOverlay` is an implementation choice, not a locked decision; `overlay.go:40` marks `placeOverlayAvailable = false` (a spike artifact). Swapping to `lipgloss.Place` is contract-compliant and removes a documented lossy path.

### P0-2 — Footer truncation hides the only "add identity" affordance

- **Observed:** The user could not find how to add an identity; the footer hint line is cut off at the right edge, dropping `a add  d delete  R rotate`. The fix label only became reachable by accident.
- **UX principle violated:** *Discoverability* and Nielsen #6 *Recognition rather than recall*. The footer is the product's entire affordance map; truncating it from the right removes the highest-value actions (`add`) while keeping the lowest (`quit`, `focus`). Information scent is destroyed — there is no path to the primary task (creating an identity), which is a first-run dead end.
- **Root-cause hypothesis:** `tui/model.go:renderFooter` (lines 853-911) builds hints in a fixed order — `q quit, tab focus, ↑↓ move, e edit, c copy, a add, d delete, R rotate, ctrl+p palette, ? help` — then blindly truncates the *joined string* from the right at `width-2` (lines 901-909). On a ~100-col terminal with bold+faint ANSI per token, the visible budget is exceeded and `a add` onward is dropped. The UI-SPEC explicitly requires the opposite priority ("prefer dropping rightmost hints" was intended to drop *low-value* hints, and the spec assumed `bubbles/v2 help.Model` compact mode would manage this — but the code does not use `help.Model`; it concatenates and string-truncates).
- **Recommendation:**
  1. Reorder hints by priority so the survivors are the load-bearing ones: `a add` and `? help` must never be the first dropped. Put `a add  e edit  c copy  ? help` ahead of `tab focus`, `↑↓ move`, `d delete`, `R rotate`.
  2. When the line overflows, **collapse to a short set** rather than mid-string truncating: e.g. `a add  e edit  c copy  ? help  ·  more in ?`. Append a literal `?` discoverability pointer so the full set is one keypress away (the help overlay already exists, `tui/help.go`).
  3. Better: render two lines for the footer when width permits, or adopt the spec'd `help.Model` which lays out and elides bindings as units (not characters), avoiding ANSI-width miscounts.
- **Contract:** Fits within D-04 and the UI-SPEC Footer contract (closes G-02/G-04). No locked decision blocks reordering.

### P0-3 — Cryptic single-char field values in the create wizard (`> e`, `> l`, `> 1`, `> y`)

- **Observed:** The Add form shows `Identity Name: > e`, `Git Name: > e`, `Git Email: > e`, `SSH Alias: > l`, `Match Strategy: > 1`, `Signing: > y`. The user cannot tell whether these are placeholders, stray input, or real values. (`Provider: > github.com`, `Port: > 22` look like real defaults.)
- **UX principle violated:** Nielsen #2 *Match between system and the real world*; placeholder-as-data anti-pattern (NN Group, "Placeholders in Form Fields Are Harmful", 2014 — placeholders disappear, are mistaken for values, and fail recognition). Showing the *first letter of the placeholder* as if it were content is worse than no placeholder at all.
- **Root-cause hypothesis:** Two things combine:
  1. Defaults that ARE set are fine (`github.com`, `22`, `y`, match `1`) — those are real values from `newCreateWizardModel` (`tui/wizard.go:425-434`). But `Match Strategy` default `"1"` and `Signing` default `"y"` are **opaque encodings** surfaced raw to the user. `> 1` for a match *strategy* communicates nothing; `> y` for *signing* communicates nothing.
  2. The blank fields (`Identity Name`, `Git Name`, `Git Email`, `SSH Alias`) showing `> e` / `> l` strongly indicate the **placeholder string is being rendered truncated to its first visible character** because the textinput has no width (same root as P0-1): `"e.g. personal"` → the bordered/zero-width input shows only `> e`; `"leave blank to use provider"` → `> l`. So `> e`, `> l` are placeholders crushed to one glyph by the missing-width bug.
- **Recommendation:**
  1. Fix the width bug (P0-1) so placeholders render in full and in a visibly de-emphasized style (`StyleFaint`/muted), distinct from typed values. bubbles v2 renders placeholder text in the placeholder style automatically once the input has room.
  2. Replace opaque encoded defaults with human-readable controls:
     - **Signing:** render as a labeled toggle, not a free-text `y`/`n` field — `Signing:  [x] enabled  ( ) disabled` (Space toggles). `> y` is recall-not-recognition.
     - **Match Strategy:** the UI-SPEC already specifies a three-option selector with a live `includeIf` preview (D-06, UI-SPEC § Match Strategy field expansion). `> 1` is the un-expanded encoding; implement the spec'd control (`[1] gitdir (folder) / [ ] hasconfig (repo URL) / [ ] both`). This is a *locked* decision being shipped in a degraded form.
- **Contract:** Fits within (and is mandated by) D-06. The match-strategy rich control is LOCKED and currently not implemented — flag this as spec-noncompliance, not a new idea.

### P1-4 — View switching is undiscoverable (no visible 1/2/3 or arrow hint)

- **Observed:** "It's not clear how to switch between Identities / Health / Global Options until I press Ctrl+P." The tabs render (`[Identities]  Health  Global Options`) but nothing signals they are keyboard-switchable, and the footer never shows `1/2/3`.
- **UX principle violated:** *Discoverability* + Nielsen #6. Tabs look like labels, not controls; the activation keys are invisible. The command palette became the only discovered path, which inverts the intended hierarchy (palette is the power-user fallback, not the primary).
- **Root-cause hypothesis:** `tui/model.go:renderTab` (lines 844-849) renders the active tab as `[Identities]` (bracket + bold+underline) and inactive tabs as faint labels — no key affordance. `renderFooter` (lines 853-899) never includes the `1/2/3` bindings even though they are handled in `handleKey` (lines 465-486). The keys work; they are just never advertised.
- **Recommendation:**
  1. Prefix each tab label with its number: `1 Identities   2 Health   3 Global Options`, active one bold+underlined. This is the single highest-leverage discoverability fix — it turns dead labels into a visible keymap (recognition over recall).
  2. Add a compact `1·2·3 views` token to the footer (high priority slot, see P0-2).
- **Contract:** Fits within D-04 (1/2/3 + Ctrl+P locked). Numbering the tab labels and footer-advertising the keys is additive.

### P1-5 — No real initial selection; detail pane empty on launch

- **Observed:** "The first identity is selected but I still need to press arrow to see something." On launch the detail pane shows nothing until the user presses ↓; and there is no per-identity health badge at rest.
- **UX principle violated:** *Sensible defaults* (Nielsen #7 *Flexibility and efficiency*; Krug's "don't make me think"). A master/detail view with a populated list should show the first item's detail immediately. Requiring a keystroke to reveal the obvious default is friction at the worst moment (first run).
- **Root-cause hypothesis:** `sidebarModel.selected` initializes to `-1` (`tui/sidebar.go:57-63`). The *render* path auto-selects index 0 (`view`, lines 269-271) **on a value copy of the model**, so the highlight appears but the real `m.selected` in the root model stays `-1`. `selectedAccount()` (lines 67-72) therefore returns `nil`, and `renderIdentitiesMainPane` (`tui/model.go:928-934`) feeds `nil` to the detail pane → empty state. The "selection" is cosmetic only. Separately, `updateKey` (lines 229-249) only advances from a real index, and badges (`m.sidebar.badges`) are empty until a Health run populates them (`familyResultMsg` handler, `tui/model.go:181`), so no badge shows at rest.
- **Recommendation:**
  1. Set `selected = 0` for real when accounts arrive. In the `refreshSidebarMsg` handler (`tui/model.go:158-161`), after assigning `m.sidebar.accounts`, if `m.sidebar.selected < 0 && len(accounts) > 0 { m.sidebar.selected = 0 }`. This makes the launch detail pane populate without a keystroke and removes the cosmetic-vs-real divergence.
  2. Seed badges at launch: kick off the doctor run (or at least the per-identity coherence/perms families) from `Init()` alongside `sidebar.refresh`, so rows show `✓/!/✗` at rest (D-08 says badges are "fed by one doctor run"). Today badges only appear after the user visits Health.
- **Contract:** Fits within D-01/D-08. D-08 explicitly wants badges seeded from a doctor run; seeding it at launch is contract-aligned (and matches D-16's "drive the real path").

### P1-6 — `Enter` is a dead key in the main view

- **Observed:** "If I press Enter over the option, the UI is not populated, displayed with nothing." Enter on a sidebar identity (focus = sidebar) does nothing.
- **UX principle violated:** Nielsen #4 *Consistency and standards* — `Enter` universally means "open/activate the selected item." A no-op Enter on a selected list row breaks the strongest list convention there is (Jakob's Law: users expect what every other list does).
- **Root-cause hypothesis:** `tui/model.go:handleKey` global switch (lines 452-683) has **no `enter` case** when no modal is open. Enter is only handled inside inline-edit (lines 424-449) and inside modals. With focus on the sidebar, pressing Enter falls through to `return m, nil`. The keymap declares `Select`/`Submit` on `enter` (`tui/keymap.go:76,89`) but the root model never wires it for the sidebar.
- **Recommendation:** Wire `enter` (no modal, identities view):
  - If focus is `sidebar` and an identity is selected → move focus to the detail pane (`m.focused = "main"`) so Enter "drills into" the item. This gives Enter a meaningful, conventional result and pairs with Tab.
  - Optionally, Enter on the sidebar could open inline edit directly (equivalent to `e`) — but focus-to-detail is the less surprising default. At minimum, Enter must not be silent. The spec's IA diagram lists `enter → select identity` (UI-SPEC § View 1 footer and Help overlay: "enter select identity"), so this is spec'd and unwired.
- **Contract:** Fits within D-01. UI-SPEC Help overlay already documents `enter  select identity` — implement the documented behavior.

### P1-7 — Sidebar stays the identity list in Health and Global Options (scope lie)

- **Observed:** "When I navigate to Health and Global Options the Identities left panel keeps active." The sidebar shows identities even when the main pane is about system-wide doctor families or global git config.
- **UX principle violated:** Nielsen #2 *Match between system and real world* + *Visibility of system status*. A persistent control that is irrelevant to the current view creates a false affordance: the user reasonably expects sidebar selection to drive the Health/Global content, but it does not. This is the "spatial scope mismatch" problem — the navigation rail implies a relationship that does not exist.
- **Root-cause hypothesis:** This is **D-01 as designed** — "the sidebar never disappears" — and the implementation faithfully renders the same `sidebarModel` in all three views (`tui/model.go:renderPersistentLayout` always calls `m.sidebar.view`). So this is a *locked decision* producing a real usability complaint, not a bug. The UI-SPEC even states for Health/Global: "Sidebar content: unchanged."
- **Recommendation (requires revisiting LOCKED D-01):** Make the sidebar **contextual per view**, keeping it always-present (which is the real intent of D-01: "always visible," for layout stability) but changing its *content and meaning*:
  - **Identities view:** managed identities + unmanaged (today's behavior).
  - **Health view:** a family list (Dependencies, Permissions, Coherence, Orphans, Signing, Agent, Baseline, Overlap) acting as a jump-list / filter into the streaming panel — selecting a family scrolls/filters the main pane to it. This makes sidebar selection meaningful again.
  - **Global Options view:** a section list (Core, Push/Pull/Fetch, Color, Aliases, Gitignore, URL Rewrites) as an anchor nav into the main pane.
  - See the dedicated recommendation section below for the full rationale and a lower-risk middle option. The literal "identity list in all views" reading of D-01 should be **revisited**; the *spirit* of D-01 (persistent left rail, Tab-cycle focus) is preserved by the contextual approach.
- **Contract:** **Revisits LOCKED D-01** (and the UI-SPEC "Sidebar content: unchanged" lines for Health/Global). Flag explicitly for sign-off.

### P2-8 — `R rotate` label is jargon; and the create-wizard match/signing encodings (overlap with P0-3)

- **Observed:** "What does rotate mean?" The footer/keymap label `R rotate` assumes the user knows SSH key-rotation terminology.
- **UX principle violated:** Nielsen #2 *Match between system and the real world* — speak the user's language, not the system's. "Rotate" is ops jargon; a first-time identity manager has no model for it.
- **Root-cause hypothesis:** Label is hard-coded as `rotate` in `tui/model.go:renderFooter` (line 881) and `tui/keymap.go:85` (`R`, "rotate (CLI)"). No expansion, no tooltip, no help text inline.
- **Recommendation:**
  1. Relabel to an outcome, not a mechanism: `R new key` or `R replace key`. Reserve "rotate" for the confirm modal where there's room to explain: the rotate confirm copy (UI-SPEC § Rotate Confirm) already says "generate a new ed25519 key … and re-test" — good; the *entry label* should match that plain language.
  2. The help overlay (`?`) should gloss it: `R  rotate — generate a fresh key for this identity, replacing the old one`.
- **Contract:** Fits within all decisions (copywriting is Claude's discretion). Pure copy change.

### P2-9 — `←/→` not wired for any navigation (user expectation gap)

- **Observed:** "Why not navigate the sections with arrows ←/→ also?"
- **UX principle violated:** *Flexibility and efficiency of use* + Jakob's Law. Horizontal arrows are a near-universal expectation for switching tabs/sections; leaving them inert wastes a discovered affordance.
- **Root-cause hypothesis:** The keymap *declares* `Left`/`Right` on `h`/`l` (`tui/keymap.go:74-75`) but `handleKey` (`tui/model.go`) has no `left`/`right`/`h`/`l` case at all — they are dead keys. The UI-SPEC reserved `←/→` for "horizontal scroll within viewport" and explicitly removed them from pane switching (UI-SPEC Navigation note, ~line 1003) to avoid the Phase 5 "dead keybindings" issue — yet they ended up dead anyway because no scroll consumer was wired.
- **Recommendation:** Bind `←/→` to **view switching** (cycle Identities → Health → Global Options and back). This directly answers the user, reinforces the `1/2/3` model with a second discoverable path, and is the most intuitive use of horizontal arrows in a tabbed shell. Update the UI-SPEC navigation note accordingly (it currently forbids this, but the no-op is worse than the reuse, and Tab already owns pane focus so there is no conflict). Add `←/→ views` to the footer.
- **Contract:** Minor revisit of a UI-SPEC note (not a CONTEXT D-xx lock). D-04 says view switching is `1/2/3` + Ctrl+P "locked design" — adding `←/→` as a *third* path is additive and does not remove the locked paths, but should be confirmed since D-04 enumerated the switch methods. Flag for sign-off.

## Top 5 Fixes, in order

1. **Fix the phantom-box render bug (P0-1).** Give every `textinput` an explicit width, draw exactly one border at a fixed width, and replace the hand-rolled `overlayLine` ANSI splice with `lipgloss.Place` for modal centering. This removes the single biggest "it's broken" signal and simultaneously fixes the crushed placeholders in P0-3. Highest leverage, two files (`detail.go`, `wizard.go`) + one (`overlay.go`).
2. **Fix the footer so `a add` (and `? help`) always survive (P0-2).** Reorder hints by priority and collapse-with-`more in ?` instead of mid-string truncation. Restores the path to the primary first-run task. Low effort, one function (`renderFooter`).
3. **Make the first identity actually selected + seed badges at launch (P1-5).** Set `selected = 0` in the `refreshSidebarMsg` handler and kick a doctor run from `Init()`. The detail pane and per-row health then populate on launch with zero keystrokes. Low effort, high payoff.
4. **Number the tabs and advertise view keys (P1-4) + wire Enter (P1-6).** Render `1 Identities  2 Health  3 Global Options`, add `1·2·3 views` and `enter select` to the footer, and make Enter focus the detail pane. Turns the shell's navigation from "hidden" to "obvious." Medium-low effort, mostly `renderTab`/`renderFooter`/`handleKey`.
5. **Implement the spec'd match-strategy + signing controls (P0-3 remainder) and relabel `rotate` (P2-8).** Replace `> 1` / `> y` with the locked D-06 three-option match selector + live `includeIf` preview and a signing toggle; rename the rotate label to `new key`. Closes the "cryptic form" complaint and ships a locked decision that is currently degraded. Medium effort, `wizard.go` + `detail.go`.

## Should the sidebar be contextual per view?

**Recommendation: yes — make the sidebar contextual, while keeping it always present. This requires revisiting LOCKED D-01.**

**Rationale.** D-01's true purpose is *layout stability and a guaranteed Tab target* — "the sidebar never disappears" so the two-pane geometry and focus model stay constant. The implementation took it literally ("always the identity list"), which produces the P1-7 complaint: in Health and Global Options the rail is a dead, misleading control. A persistent navigation element that does not relate to the current content violates *visibility of system status* and creates a false affordance — the user expects selecting a row to change what they see, and it does not. The identity list earns its place only in the Identities view.

A contextual sidebar fixes this without sacrificing anything D-01 actually protects:
- **Identities view:** managed identities + unmanaged section (unchanged).
- **Health view:** the 8 doctor families as a jump/filter list — selection scrolls or filters the streaming panel to that family, and the per-family spinner/badge lives in the rail. This is exactly the `tools-installer` model the CONTEXT cites as the ergonomics target (catalog/doctor families in the rail, detail on the right).
- **Global Options view:** the config sections (Core, Push/Pull/Fetch, Color, Aliases, Gitignore, URL Rewrites) as anchor navigation.

In all three, the rail stays 18 cols wide, always visible, and Tab still cycles focus between rail and main — so D-01's geometry and focus contract are fully preserved. Only the *content and meaning* of the rail changes with the view, which is what users already expect from a tabbed master/detail shell.

**Lower-risk middle option (if reopening D-01 is undesirable now):** keep the identity list in all views but **visually de-activate it** outside the Identities view — render it faint, drop the `›` selection indicator, and move keyboard focus to the main pane by default in Health/Global so Tab/↑↓ act on the relevant content. This removes the false affordance (the rail clearly reads as inactive context, not a live control) with a much smaller change and no contract reopen. It is strictly worse than the contextual sidebar for Health/Global usefulness, but it resolves the "scope lie" complaint and is shippable inside D-01 as written.

**My call:** adopt the contextual sidebar (reopen D-01 for sign-off). The middle option is the safe fallback if the milestone cannot absorb a locked-decision change this late. Either way, the current "live identity list in every view" must not ship — it actively misleads.
